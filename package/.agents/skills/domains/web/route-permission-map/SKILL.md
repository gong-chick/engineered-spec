# 路由权限映射与审计

## 使用时机

当你需要：

- 核对路由、菜单和权限是否一致
- 审查未保护路由、死链或错误的菜单显隐
- 梳理某个前端应用的权限边界

## 核心原则

- 先识别框架，再按对应分支扫描
- 只根据仓库实际代码与配置建立映射，不臆测隐藏规则
- 输出以映射表、问题项和修复建议为主
- 若在当前协议流程中，优先把结论沉淀到 `checklist.md`、`verification-review-notes` 或用户指定文档

## 步骤 1：识别前端框架

先看 `package.json`：

- `next` → Next.js
- `vue` / `vue-router` → Vue
- `react` + `react-router` / `react-router-dom` → React
- 多应用仓库时，按当前目标应用所在子包判断

## 步骤 2：定位三类来源

必须同时找出：

1. 路由定义
2. 权限守卫或路由级权限元数据
3. 菜单 / 侧栏配置

优先提取这些字段：

- `path`
- `component / element / page`
- `meta.roles / meta.permission / requiresAuth`
- 菜单 `title / icon / hidden / children`
- 是否懒加载

## 步骤 3：建立映射表

至少输出：

| path | 路由来源 | 权限/角色 | 菜单标题 | 是否隐藏 | 是否懒加载 |
|------|----------|-----------|----------|----------|------------|

如果菜单来自接口，明确写出“菜单数据来自接口，需和前后端约定对齐”。

## 步骤 4：执行四类检查

### 1. 未保护路由

- 除登录页、404、公开页外，未被任何 guard 或权限元数据覆盖的路由

### 2. 死链

- 菜单 path 在路由清单中不存在
- redirect 目标不存在

### 3. 菜单与权限不一致

- 菜单做了权限控制但路由无守卫
- 路由有权限要求但菜单无显隐控制

### 4. 懒加载合理性

- 首屏关键路由被过度懒加载
- 子页面拆包过细，导致请求过多

## 输出模板

```md
## 路由权限映射报告

### 路由清单
| path | 权限/角色 | 是否懒加载 | 来源 |
|------|-----------|------------|------|

### 菜单映射
| 菜单 path | 菜单标题 | 路由存在 | 权限是否一致 |
|-----------|----------|----------|--------------|

### 发现的问题
- 未保护路由：
- 死链：
- 菜单与权限不一致：
- 懒加载建议：
```

## 注意事项

- 路径统一按正斜杠比较
- 区分公开页与“漏加守卫”的业务页
- 只给出审计与修复建议，不自动改代码
