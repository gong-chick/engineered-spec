# 核心产物写入合同

## 01-项目概述.md

要求：

- 项目定位一句话描述必须来自可确认事实
- 技术栈表只写项目实际安装和实际使用的技术
- TypeScript / JavaScript 分支必须按项目实际依赖区分
- 私有包或公司内部包需单独标注用途
- 不得把未经确认的业务背景写成既定事实

## 03-项目结构.md

要求：

- 目录树必须反映 `src/` 下的实际目录
- 每个目录用途需根据 2-3 个样本文件归纳，而不是猜测
- 如果项目存在路由目录、页面组件目录、Mock 策略，写清对应放置规则
- 所有示例文件扩展名必须跟仓库实际一致，不要写死 `.ts`

## context/PROJECT.md

只保留稳定项目上下文：

- 项目定位
- 技术栈摘要
- 关键规范入口
- 业务目标
- 核心术语
- 架构概览
- 非目标

填写要求：

- `项目定位` 与 `01-项目概述.md` 保持一致
- `技术栈摘要` 只写可确认事实
- `业务目标 / 核心术语 / 非目标` 无法确认时明确写 `待补充`
- 不把某次任务的默认假设、页面落点、登录方式等运行态信息写进来

## openspec/project.md

若项目存在 `openspec/`：

- 若 `openspec/project.md` 已存在，只更新其中 `## 项目概述` 段落，使其与 `01-项目概述.md` 的项目定位一致
- 若 `openspec/project.md` 缺失，可创建最小 `project.md`，至少包含：
  - `# Project`
  - `## 项目概述`
  - 与 `01-项目概述.md` 一致的一句话定位
- 不修改 `project.md` 中其他无关章节

若不存在 `openspec/`，跳过此步骤。

## 用户确认摘要

真正写入前，向用户给出最小确认摘要，至少包含：

1. 检测到的技术栈
2. 检测到的目录结构
3. `PROJECT.md` 中的项目定位和关键规范入口
4. 若存在自定义规则待生成，逐项说明将补哪些规则
5. 若存在 OpenSpec，同步到 `openspec/project.md` 的项目定位描述

确认问题建议：

> 以上信息是否准确？若无异议，我将写入 01/03、PROJECT，以及当前缺失的自定义规则。

## 约束

- 一旦用户确认，必须按 `本轮写入清单` 一次性完成全部写入
- 不要只写 `01/03` 后提前结束
