# 接口与契约规范（Spring Boot）

## RESTful 设计原则

| 操作 | HTTP 方法 | URI 示例 |
|------|-----------|----------|
| 查询列表 | GET | `/api/v1/orders` |
| 查询详情 | GET | `/api/v1/orders/{id}` |
| 创建 | POST | `/api/v1/orders` |
| 更新（全量） | PUT | `/api/v1/orders/{id}` |
| 更新（部分） | PATCH | `/api/v1/orders/{id}` |
| 删除 | DELETE | `/api/v1/orders/{id}` |

## 统一响应体（NON-NEGOTIABLE）

所有接口必须返回统一的响应结构：

```java
public class Result<T> {
    private int code;       // 状态码，成功为 0 或 200
    private String message; // 描述信息
    private T data;         // 业务数据
}
```

- 不允许直接返回裸数据，必须包装成统一响应体
- 异常响应同样遵循此格式

## API 版本控制

- URI 路径版本：`/api/v1/`、`/api/v2/`
- 新版本接口不应破坏已有版本的兼容性

## 参数校验

- 使用 JSR-303/380（`@Valid`、`@Validated`）进行请求参数校验
- Controller 层统一触发校验，不在 Service 层重复校验入参格式
- 自定义校验注解放在 `common` 模块的 `validation` 包

## DTO 命名约定

| 场景 | 命名 | 示例 |
|------|------|------|
| 请求参数 | `XxxRequest` / `XxxReqDTO` | `CreateOrderRequest` |
| 响应数据 | `XxxResponse` / `XxxRespDTO` | `OrderDetailResponse` |
| 查询条件 | `XxxQuery` | `OrderListQuery` |
