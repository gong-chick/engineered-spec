# 通用约束

## 语言规范（NON-NEGOTIABLE）

- 所有文档和代码注释必须使用中文
- 代码变量/函数命名仍使用英文（遵循编码规范）

## 可观测性与兜底

### 通用规范

- 错误处理必须有统一机制（错误边界 / 全局异常捕获），禁止吞掉异常不处理
- 生产环境禁止残留 `console.log`、`console.debug`；仅允许 `console.warn` 和 `console.error` 用于真实告警
- 需要调试日志时，使用项目统一的 logger 工具，支持按环境区分日志级别

### AI 业务场景

以下约束适用于包含 AI 能力的业务模块：

- AI 任务需要完整链路：任务 ID、素材/模型引用、日志追踪
- 失败时提供兜底提示与重试机制

## 依赖管理

- 新增第三方依赖前需评估其对产物体积或运行时开销的影响，优先使用项目已有工具库
- 使用项目约定的包管理器，禁止在同一项目中混用多种包管理器
- 禁止引入超过 12 个月无更新且无明确维护者的依赖
- 依赖版本通过 lock 文件锁定，禁止使用 `*` 或 `latest` 版本号

## 安全约束

- 禁止在代码中硬编码密钥、token、密码等敏感信息
- 敏感配置通过 `.env` 文件管理，`.env` 文件必须加入 `.gitignore`
- 用户输入在拼接 HTML 或 URL 前必须转义，防止注入攻击
- 第三方脚本（统计、广告等）需评估安全风险后再引入

## 调试代码约束

- 提交代码前必须清除 `debugger` 语句
- 禁止提交被注释掉的大段代码块（超过 5 行），废弃代码应直接删除
- 临时调试用的 `// @ts-ignore`、`// eslint-disable` 必须附带原因注释，合并前清除

## Git 提交规范

- Commit message 遵循 [Conventional Commits](https://www.conventionalcommits.org/) 格式：`<type>(<scope>): <description>`
- 常用 type：`feat`、`fix`、`refactor`、`style`、`docs`、`test`、`chore`
- 单次提交聚焦一个变更目的，禁止在一个 commit 中混合不相关改动

## 通用约束汇总

以下约束已在各规范文件中详述，此处汇总便于快速查阅：

| 约束 | 关键要点 | 来源 |
|------|----------|------|
| 类型与语言 | TypeScript 优先；纯 JS 项目遵循 02 的 JavaScript 小节；TS 禁止 `any` | 02-编码规范 |
| 命名规范一致性 | 变量 camelCase、常量 UPPER_CASE | 02-编码规范 |
| 项目结构规范 | 禁止新建非标准目录 | 03-项目结构（profile 规范） |
| API 封装规范 | 接口统一通过封装层调用，不在业务层直接发请求 | 各 profile 的 API 规范 |
