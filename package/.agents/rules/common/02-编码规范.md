# 编码规范

## 语言策略（TS 优先 + JS 兼容）

- **新代码与可迁移模块**：优先使用 **TypeScript**（`.ts`、`.tsx`）。
- **存量或未引入 TypeScript 的项目**：允许使用 **JavaScript**（`.js`、`.jsx`），须遵循下文 **JavaScript** 小节。
- **同一目录/同一业务模块内**：避免同一职责在 TS 与 JS 之间随意混用；新增文件优先与周边已有扩展名一致，或逐步统一到 TS。

## TypeScript 规范

- 所有 TypeScript 代码须包含完整类型定义。
- 禁止使用 `any`；类型不确定时优先使用 `unknown` 再收窄。
- 定义对象结构优先使用 `interface`；联合类型、交叉类型、映射类型使用 `type`。
- 接口、类型定义使用 PascalCase。
- 公共函数与模块导出须明确类型签名。

## JavaScript（存量 / 无 TS 项目）

- **类型意图**：对导出函数、公共 API、复杂参数与返回值使用 **JSDoc**（`@param`、`@returns`、`@typedef` 等）表达契约；团队若启用 `checkJs` / `// @ts-check`，须与之保持一致。
- **运行时校验**：对外部数据（接口响应、URL 参数等）须在边界处校验或规范化，避免静默假设结构正确。

## 命名规范

| 类型 | 命名规则 | 示例 |
|------|----------|------|
| 目录/模块 | kebab-case | `user-profile`、`order-service` |
| 变量/函数 | camelCase | `userList`、`fetchData` |
| 布尔值 | is/has/can/should 前缀 | `isLoading`、`hasPermission`、`canEdit` |
| 常量 | UPPER_CASE | `API_TIMEOUT`、`MAX_RETRY` |
| 枚举名 | PascalCase | `ApiMethod`、`UserStatus` |
| 枚举值 | PascalCase | `ApiMethod.Get`、`UserStatus.Active` |
| 接口/类 | PascalCase | `UserInfo`、`BaseResponse` |
| 组合式函数/Hooks | use 前缀 + camelCase | `useUserStore`、`useAuth` |

> 各技术栈的组件、类、文件命名约定由对应 profile 规范约定，此处仅约束代码中的通用命名原则。

## 业务函数命名

- **回调属性（Props / Emits / 事件处理器）**：`onXxx`（如 `onSubmit`、`onChange`）
- **本地处理函数**：`handleXxx`（如 `handleSubmit`、`handleDelete`）
