# ai-spec-auto  整合 Everything Claude Code 与专家协同实施方案

## 1. 新定位

结合你补充的专家分类、临时思路和领导反馈，`ai-spec-auto ` 的目标不应再只是“前端 Skill 规范库”，而应升级为：

> 一个可按需安装、可观测、可验证、可跨 IDE 复用的 AI 驱动前端自动化流水线平台。

对外表达上，不再强调“做了多少个 skill”，而强调：

- 规划了多少条前端自动化流程
- 沉淀了多少个可协同的专家
- 底层复用了多少类技能与规则
- 有多少能力被验证、被使用、被推广

这和你领导的判断是一致的：

- 先把未来的最大边界规划清楚
- 当前阶段仍以技术能力基建为主
- 但汇报口径必须从“技能堆叠”切换到“专家协同与流程平台”

## 2. 核心原则

### 2.1 流程优先，专家显性，技能隐性

建议将能力体系统一抽象为三级：

- 一级：流程
- 二级：专家角色
- 三级：技能实现

也就是：

```text
流程（业务可理解）
  -> 专家（团队可感知）
    -> Skill / Rule / Hook / Eval（技术实现）
```

这样汇报时讲的是“需求设计流、质量治理流、发布交付流”，而不是“我们做了 50 个 skill”。

### 2.2 专家协同优于单点技能

Skill 是底层积木，不是对外主角。

对外应该呈现：

- `需求解析专家`
- `设计协作专家`
- `规范守护者`
- `性能调优专家`
- `错误追踪专家`

而不是直接暴露：

- `figma-import`
- `eslint-config`
- `storybook-doc`
- `lighthouse-audit`

### 2.3 按需注入，不做全量灌入

你补充的“按需注入”判断是对的。全量把 rules、skills、roles 都注入目标项目，会带来 3 个问题：

- 上下文过重，AI 响应变慢
- 规则过多，容易冲突或产生幻觉
- 团队感知不到“装了什么、没装什么”

因此安装策略必须从“全量复制”升级为“按选择编译注入”。

### 2.4 源资产解耦，目标资产编译

源仓库仍然保持单一维护源，但目标项目中不一定要原样落地。

建议：

- 源端统一维护：`rules + skills + roles + flows + hooks + evals + telemetry`
- 安装时按 IDE、流程包、专家包、规则依赖关系进行编译输出

### 2.5 平台必须可观测

你提到的“脚本安装时可配置、要有审计规划埋点、分析用户选择路径和使用效率”非常关键。

`ai-spec-auto ` 如果想推广，不应该只管安装成功，还要知道：

- 用户安装了什么
- 用户常选哪些流程和专家
- 哪些能力被高频触发
- 哪些流程过重、过慢、很少使用
- 哪些规则或专家真的提升了代码采纳率

## 3. 能力总模型：流程 -> 专家 -> Skill

建议把你图片里的分类吸收为一级流程模型，再补上安全和可访问性两个横切面。

### 3.1 一级流程层

建议定义 8 条主流程 + 2 条横切治理流程：

1. `需求设计流`
2. `研发规范流`
3. `工程化流`
4. `测试验证流`
5. `发布交付流`
6. `文档知识流`
7. `性能治理流`
8. `可观测治理流`
9. `安全治理流`
10. `无障碍治理流`

### 3.2 二级专家层

建议按流程组织专家，而不是平铺罗列。

#### 需求设计流

- `需求解析专家`
- `设计协作专家`

#### 工程化流

- `ESLint 规则专家`
- `格式化专家`

- `构建专家`

#### 测试验证流

- `单元测试专家`
- `E2E 测试专家`

- `验证专家`

#### 发布交付流

- `流水线专家`

- `部署专家`

#### 文档知识流

- `组件文档专家`
- `API 文档专家`

#### 性能治理流

- `性能审计专家`

### 3.3 三级 Skill 层

Skill 继续保留，但角色从“主语”降为“实现细节”。

例如：

- `需求解析专家` 复用 `prd-reader`、`create-proposal`
- `设计协作专家` 复用 `design-analysis`、`ui-verification`
- `Vue 规范专家` 复用 `vue-best-practices`
- `规范守护者` 复用 `eslint` / `prettier` / `stylelint` / `commitlint`
- `性能调优专家` 复用 `lighthouse-audit`、`core-vitals`、`bundle-analyze`

最终对外讲“专家”，对内维护“skill”。

## 4. 专家协同模式

这是本次补充里最重要的部分。建议把“专家协同模式”设计成 `ai-spec-auto ` 的核心卖点。

### 4.1 协同模式定义

所谓专家协同模式，不是简单堆专家，而是定义一条自动化流程：

```text
任务进入
  -> 流程识别
  -> 激活对应专家
  -> 专家间交接
  -> 质量闸门检查
  -> 输出交付物
  -> 记录埋点与评估结果
```

### 4.2 推荐的三种协同模式

#### 模式 A：串行交付模式

适合新需求开发。

```text
需求解析专家
  -> 设计协作专家
  -> API 契约专家
  -> 前端实现专家
  -> 规范守护者
  -> 单元测试专家
  -> E2E 测试专家
  -> 性能调优专家
  -> 文档专家
```

#### 模式 B：闸门守护模式

适合任何编码任务后的质量补位。

```text
实现完成
  -> 规范守护者
  -> 安全审查专家
  -> 验证专家
  -> 性能调优专家
```

这是你第二张图里“生成代码后自动触发规范守护者，再由性能调优专家给建议”的正式化版本。

#### 模式 C：按场景分支模式

适合安装时按需注入和运行时按需激活。

例如：

- 如果用户只选“页面开发”，则不注入 CI/CD、容器、部署专家
- 如果用户只选“质量治理”，则只注入规范、测试、性能、安全专家
- 如果用户只选“推广演示版”，则注入演示链路、指标采样和专家目录说明

### 4.3 专家之间必须有交接协议

每位专家不能只是写一段描述，必须有清晰的输入输出契约：

- 输入是什么
- 输出是什么
- 是否会触发下一个专家
- 对哪些规则和 skill 有依赖
- 何时中断流程

建议每个专家定义统一字段：

```text
name
goal
trigger
inputs
outputs
required_rules
required_skills
handoff_to
stop_conditions
telemetry_events
```

这样后续才真的能做“专家协同”，而不是一堆零散 prompt。

## 5. 建议的源目录结构

结合你“在 `.agents/roles/` 为每位专家创建一个专属文件”的想法，建议源端结构调整为：

```text
.agents/
├── rules/                     # 约束层
├── skills/                    # 实现层
├── roles/                     # 专家源定义（新增）
│   ├── common/
│   └── profiles/
├── flows/                     # 专家协同流程定义（新增）
│   ├── common/
│   └── profiles/
├── hooks/                     # hook 模板与脚本（新增）
│   └── claude/
├── evals/                     # 验证基线（新增）
├── telemetry/                 # 埋点 schema 与报表口径（新增）
└── presets/                   # IDE 原生模板（新增）
    ├── claude/
    ├── cursor/
    ├── trae/
    └── opencode/
```

说明：

- 源端建议叫 `roles/`，因为它是跨 IDE 的抽象概念
- 目标端可按 IDE 生成原生形态
  - Claude Code：`.claude/agents/`
  - Cursor：对应规则容器、commands、skills
  - Trae / OpenCode：对应各自规则与技能容器

也就是说：

- `roles/` 是源定义
- `agents/` 是某些 IDE 的目标产物

## 6. 安装模式与按需注入方案

### 6.1 保留三级安装，但重新定义含义

建议保留你当前的 `L1 / L2 / L3`，但升级为下面这套解释：

#### L1：能力基建层

- 注入核心 `rules + skills`
- 面向个人试用、最小可运行
- 不默认开启协同流程和可观测能力

#### L2：专家协同层

- 注入所选 `roles + flows + hooks`
- 面向团队标准接入
- 支持“专家协同模式”

#### L3：平台治理层

- 在 L2 基础上增加 `eval + telemetry + memory + OpenSpec`
- 面向企业治理和效果复盘

### 6.2 安装时的可选维度

安装不要只有 `profile`，还应支持：

- `--level`
- `--profile`
- `--flows`
- `--roles`
- `--packages`
- `--telemetry`
- `--with-openspec`

建议示例：

```bash
ai-spec-auto init --profile vue --level L3 --flows demand,quality --roles requirement-analyst,code-guardian,perf-optimizer
```

### 6.3 推荐的功能包

为了方便推广，建议不要让用户一次选几十个角色，而是优先按“功能包”安装：

- `frontend-core`
  - 需求设计流 + 研发规范流 + 页面交付能力

- `quality-guard`
  - 规范守护者 + 测试验证流 + 性能治理流 + 安全治理流

- `release-ops`
  - 工程化流 + 发布交付流 + 可观测治理流

- `docs-knowledge`
  - 文档知识流 + API 契约流

- `enterprise-governance`
  - eval + telemetry + memory + OpenSpec

### 6.4 安装编译原则

安装脚本不应再简单复制全部内容，而应执行依赖收敛：

1. 用户选择流程包或专家
2. CLI 解析所需规则、skills、hooks、evals
3. 仅把依赖闭包内的内容注入目标项目
4. 未选中的流程节点和专家不进入目标项目

这样可以满足你提出的两个目标：

- 避免 AI 上下文过重
- 让目标项目只保留真正需要的能力

## 7. 可观测平台设计

你的第二张图里提到“审计规划埋点、执行效率评判、用户选择路径分析、看使用率和推广率”，这部分建议单独立项，不要附属处理。

### 7.1 可观测对象

建议分三层埋点：

#### 安装层

- 安装开始 / 成功 / 失败
- 选择的 level / profile / flows / roles / packages
- 哪些能力被跳过

#### 执行层

- 哪条流程被触发
- 哪位专家被激活
- 流程耗时
- 验证是否通过
- 哪个环节失败或被跳过

#### 效果层

- 代码采纳率
- Review 规范问题数
- 测试通过率
- 性能问题发现率
- 活跃项目数
- 常用功能包分布

### 7.2 推荐埋点事件

建议至少定义这些事件：

- `install_started`
- `install_completed`
- `package_selected`
- `flow_selected`
- `role_selected`
- `flow_triggered`
- `role_invoked`
- `role_handoff`
- `verification_passed`
- `verification_failed`
- `expert_skipped`
- `usage_summary_generated`

### 7.3 数据边界

企业场景下建议默认只采集结构化元数据，不默认上传代码内容。

建议默认策略：

- 默认本地落盘
- 团队版可选择上报聚合指标
- 不上传源码正文
- 如需样本分析，必须额外显式开启

## 8. 与 Everything Claude Code 的整合策略

`everything-claude-code` 值得吸收的重点，不是“多”而是“系统性”。

### 8.1 建议优先吸收

- `search-first`
  - 变成所有专家默认工作习惯

- `verification-loop`
  - 作为 `验证专家` 的标准闭环

- `security-review`
  - 作为 `安全审查专家` 的核心能力来源

- `eval-harness`
  - 作为 L3 平台治理层的验证基线

### 8.2 建议改造吸收

- `continuous-learning-v2`
  - 改造成 `pattern-memory`
  - 数据来源优先接 `openspec/changes/archive`、review 缺陷库、典型最佳实践

- `planner`
  - 融入 `需求解析专家` 和 `solution-architect`

- `code-reviewer`
  - 融入 `规范守护者` 与 `code-review-expert`

### 8.3 暂不建议首批引入

- 过多语言专用 skill
- 只适合个人黑客工作流的实验性能力
- 会显著增加维护成本但难以形成业务亮点的命令包装

## 9. 分阶段实施路线

### Phase 1：专家协同 MVP（1 到 2 周）

目标：先把“专家协同模式”做出来，而不是追求专家数量。

交付：

- 新增 `.agents/roles/`、`.agents/flows/`、`.agents/hooks/`、`.agents/telemetry/`
- 落地 5 个首批专家：
  - `需求解析专家`
  - `设计协作专家`
  - `前端实现专家`
  - `规范守护者`
  - `性能调优专家`
- 落地 2 条流程：
  - `需求设计 -> 页面交付流`
  - `实现完成 -> 质量闸门流`
- 安装脚本支持按需注入
- 增加最小埋点：
  - 安装选择
  - 流程触发
  - 专家激活
  - 验证结果

### Phase 2：能力包与治理闭环（2 到 4 周）

目标：让它从“专家 demo”升级为“可推广的产品包”。

交付：

- 补齐功能包：
  - `frontend-core`
  - `quality-guard`
  - `release-ops`
  - `docs-knowledge`
- 落地更多专家：
  - `单元测试专家`
  - `E2E 测试专家`
  - `安全审查专家`
  - `错误追踪专家`
  - `流水线专家`
- 建立 10 到 15 个 eval 基线任务
- 输出安装漏斗、角色使用热度、流程成功率报表

### Phase 3：企业级平台化（4 到 8 周）

目标：形成真正可汇报、可推广、可持续演化的平台故事。

交付：

- `pattern-memory`
- OpenSpec 深度联动
- 团队级可观测报表
- 专家目录与流程目录
- 对比案例与推广演示资产

## 10. 汇报口径建议

以后不建议汇报：

- “我们新增了 50 个 skill”

建议改为：

- “我们规划了 8 条前端 AI 自动化流程”
- “我们沉淀了 12 个可协同的专家角色”
- “这些专家由底层 40+ 个技能能力支撑”
- “已经支持按需注入和分层安装”
- “平台已具备安装、执行、效果三层可观测能力”

一句适合对外汇报的话：

> `ai-spec-auto ` 不是一个 prompt 或 skill 仓库，而是一套面向企业团队的 AI 前端专家协同平台。它把需求、设计、开发、测试、发布、监控拆成可安装的自动化流程，并通过专家角色进行协同执行与治理。

## 11. 下一步最小可执行清单

建议下一步只先做这 6 件事：

1. 把源目录从纯 `rules + skills` 扩展到 `rules + skills + roles + flows + telemetry`
2. 在 `.agents/roles/common/` 新建 5 个专家定义文件
3. 在 `.agents/flows/common/` 新建 2 条专家协同流程
4. 改造安装脚本，支持 `--flows` 和 `--roles`
5. 增加最小 telemetry schema 与本地落盘能力
6. 做一个“需求设计 -> 页面交付 -> 质量闸门”的完整演示链路

做到这一步，`ai-spec-auto ` 就已经不再只是前端规范库，而是一个有明确北极星、能逐步落地的 AI 前端自动化流水线平台。
