# 运行状态落盘规范

## 1. 文档目的

本文档定义 `run（运行编排）` 的运行状态如何落盘。

这份规范解决的问题是：

- 一次 `run（运行编排）` 不能只停留在对话里
- `approve（审批） / resume（恢复） / status（状态）` 需要有统一状态来源
- OpenClaw（远程入口）、IDE（开发工具）和后续插件页面需要读取同一份运行态记录

一句话：

> `lock（锁定清单） / sources（来源清单）` 负责安装态，运行状态需要单独落盘。

## 2. 为什么不能写进 `lock（锁定清单） / sources（来源清单）`

因为它们解决的问题不同：

- `manifest（安装清单）`
  - 用户想装什么
- `lock（锁定清单）`
  - 系统实际装了什么
- `sources（来源清单）`
  - 资产从哪里来
- `run-state（运行状态）`
  - 当前这次任务跑到哪里了

如果把运行状态写进 `lock（锁定清单）`，会造成：

- 安装态和运行态混淆
- 一个项目多次运行时彼此覆盖
- 无法稳定支持 `approve（审批） / resume（恢复） / status（状态）`

## 3. 推荐目录结构

当前阶段推荐采用下面这套最小结构：

```text
.ai-spec/
├── manifest.json
├── lock.json
├── sources.json
├── current-run.json
└── runs/
    ├── run_20260330_001.json
    └── run_20260330_002.json
```

### 3.1 为什么推荐这套

- `current-run.json`
  - 便于快速读取当前活跃运行
- `runs/<run-id>.json`
  - 便于保留历史和做审计

这是比只放一个 `current-run.json` 更稳的做法，也比一上来搞复杂 `history/` 目录更轻。

### 3.2 当前最小实现状态

当前仓库里已经补了两种最小写盘入口：

```bash
ai-spec-auto runtime-state init --run-plan ./run-plan.json --task-anchor ./task-anchor.json
```

```bash
ai-spec-auto task-orchestrator-adapter apply --payload ./.ai-spec/tmp/task-orchestrator-first-response.json
```

```bash
ai-spec-auto task-orchestrator-adapter apply --payload ./.ai-spec/tmp/runtime-action.json
```

```bash
ai-spec-auto runtime-state approve --gate before-implementation --to-role frontend-implementer
```

```bash
ai-spec-auto runtime-state resume --to-role frontend-implementer
```

```bash
ai-spec-auto runtime-state status
```

```bash
ai-spec-auto runtime-state gate-blocked --gate before-implementation --status waiting-approval
```

```bash
ai-spec-auto runtime-state complete
```

```bash
ai-spec-auto runtime-state fail --error "组件规范检查未通过"
```

```bash
ai-spec-auto runtime-state cancel --message "用户主动取消当前任务"
```

它当前负责：

- 读取 `run-plan（运行计划）`
- 可选读取 `task-anchor（任务锚点）`
- 或直接读取 `task-orchestrator（任务主代理）` 首轮桥接载荷
- 或在专家交接时更新当前运行态
- 写入 `.ai-spec/current-run.json`
- 写入 `.ai-spec/runs/<run-id>.json`

但要注意：

> 现在已经补了主代理首轮桥接入口、专家交接更新命令，以及最小的阻断、审批、恢复、状态查询和结束态命令；但还没有继续覆盖完整的审批、恢复和结束状态机。

## 6. 状态值建议

建议允许值：

- `planned`
- `running`
- `waiting-approval`
- `blocked`
- `success`
- `failed`
- `cancelled`

## 7. 落盘时机

当前建议按下面时机落盘：

### 7.1 创建运行计划后

当 `task-orchestrator（任务主代理）` 完成首次 `run-plan（运行计划）` 时：

- 生成首轮 `task-anchor（任务锚点）`
- 组装主代理首轮桥接载荷
- 优先调用 `ai-spec-auto task-orchestrator-adapter apply`
- 写入 `.ai-spec/current-run.json`
- 写入 `.ai-spec/runs/<run-id>.json`

### 7.2 状态变化时

当发生以下事件时，更新同一个 `run-state（运行状态）` 文件：

- 进入审批等待
- 审批通过
- 恢复执行
- 当前执行专家变化
- 运行结束
- 运行失败

当前最小实现推荐使用：

```bash
ai-spec-auto task-orchestrator-adapter apply --payload ./.ai-spec/tmp/runtime-action.json
```

终态最小实现推荐使用：

```bash
ai-spec-auto runtime-state complete
```

```bash
ai-spec-auto runtime-state fail --error "组件规范检查未通过"
```

### 7.3 运行完成后

当运行完成：

- `current-run.json` 可清空或指向最新一次完成记录
- `runs/<run-id>.json` 必须保留

## 9. 当前阶段建议

当前阶段不要把运行状态做成数据库或复杂事件系统。

最稳的做法是：

1. 先用 JSON（结构化数据）文件落盘
2. 先保证一次运行至少有一个 `run-id（运行实例 ID）`
3. 先保证 `current-run.json` 和 `runs/<run-id>.json` 两处可读
4. 先服务 `run（运行编排） / approve（审批） / resume（恢复） / status（状态）`
5. 主代理首轮和后续运行态更新优先通过 `ai-spec-auto task-orchestrator-adapter apply` 自动完成；适配层内部再统一调用 `bootstrap（首轮桥接） / handoff（交接） / gate-blocked（阻断） / approve（审批） / resume（恢复） / status（状态） / complete（完成） / fail（失败） / cancel（取消）`；`ai-spec-auto runtime-state init` 继续保留给底层调试

## 10. 一句话收束

> 如果 `run（运行编排）` 要作为 skill（技能）入口存在，就必须把运行状态从对话里拿出来，单独落到 `.ai-spec/current-run.json` 和 `.ai-spec/runs/<run-id>.json`。这样后面的审批、恢复、状态查询和远程协作才有稳定基础。
