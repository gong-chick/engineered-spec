# 专家定义指令与技能组织建议

## 1. 目的

这份文档用于解决两个实际问题：

1. `roles` 文件到底应该怎么写，才能既真实可用，又适合后续插件页面读取
2. `skills` 到底应该放在 `common/`、`profiles/` 还是按专家单独建目录

目标不是堆很多专家名，而是建立一套稳定、可扩展、可落地的定义方式。

## 2. 参考来源和提炼出的原则

这套建议主要吸收了三类做法：

- Claude Code 官方的 sub-agents / skills 文档
- GitHub 上 `everything-claude-code` 这类以角色和流程为中心的实践
- 你当前项目对 `rules + skills + openspec` 的既有结构

提炼后，最有价值的原则只有四条：

### 2.1 专家要小而专，不要大而空

专家文件应该围绕一个明确职责来写，例如：

- 需求解析
- 前端实现
- 规范守护

而不是写成“全能前端专家”。

### 2.2 可复用的方法下沉到 skill，不要重复写进每个专家

例如：

- 设计稿分析
- proposal 生成
- 组件创建
- API 创建
- UI 验收

这些都是可复用方法，应该留在 `skills`，而不是在每个专家里重写一遍步骤。

### 2.3 流程负责顺序，专家负责职责，规则负责边界

不要让单个专家同时承担：

- 读规则
- 定流程
- 做实现
- 做验收

否则角色会过重，后续无法组合。

### 2.4 专家定义必须写清输入、输出、交接和停止条件

如果没有这四项，专家文件就只是一段人设描述，不能稳定协同。

## 3. 当前项目推荐的角色定义结构

每个专家文件建议使用：

- 英文文件名作为稳定 ID
- 中文标题作为展示名称
- frontmatter 承载插件页面和脚本后续可读的元信息
- 正文承载职责和执行约束

推荐结构如下：

```md
---
id: requirement-analyst
name: 需求解析专家
status: active
domains:
  - demand-design
description: 负责把 PRD、设计稿或自然语言需求收敛为 proposal、tasks 和关键假设。
triggers:
  - prd-input
  - design-input
preferred_skills:
  - create-proposal
  - design-analysis
reads:
  - context/PROJECT.md
  - .agents/rules/
writes:
  - openspec/changes/<change-id>/proposal.md
  - openspec/changes/<change-id>/tasks.md
handoff_to:
  - frontend-implementer
---

# 需求解析专家

## 角色定位
...

## 工作原则
...

## 必做步骤
...

## 输出标准
...

## 禁止事项
...

## 交接
...
```

## 4. frontmatter 字段建议

| 字段 | 必需 | 作用 |
|------|------|------|
| `id` | 是 | 稳定 ID，供脚本、插件、流程引用 |
| `name` | 是 | 中文展示名 |
| `status` | 是 | `active` / `planned` |
| `domains` | 是 | 能力域归属，可多值 |
| `description` | 是 | 一句话职责说明 |
| `triggers` | 否 | 何时激活该专家 |
| `preferred_skills` | 否 | 推荐调用的 skill 列表 |
| `reads` | 否 | 主要输入来源 |
| `writes` | 否 | 主要产物 |
| `handoff_to` | 否 | 默认交接对象 |

建议先控制在这几项，不要一开始把元数据做得很重。

## 5. 专家正文怎么写更真实

建议正文至少包含下面 6 段：

### 5.1 角色定位

说明这个专家负责什么，不负责什么。

### 5.2 工作原则

说明它做事的基本边界，例如：

- 先读规则再行动
- 不越权扩 scope
- 不跳过人工确认

### 5.3 必做步骤

用动作句写清执行顺序，例如：

1. 读取输入
2. 判断任务类型
3. 选择 skill
4. 生成产物
5. 交给下一位专家

### 5.4 输出标准

明确该专家最低要产出什么。

### 5.5 禁止事项

明确不能做什么，防止角色“看起来什么都能干”。

### 5.6 交接

明确交给谁，什么情况下退回主代理或要求人工确认。

## 6. 当前项目的落地建议

### 6.1 当前启用角色

当前 MVP 只需要先把这 4 个角色做实：

- `task-orchestrator`
- `requirement-analyst`
- `frontend-implementer`
- `code-guardian`

这 4 个角色已经足够跑通：

```text
任务输入
  -> task-orchestrator
  -> requirement-analyst
  -> frontend-implementer
  -> code-guardian
```

### 6.2 规划中的能力域

其余能力域先做目录占位和专家清单，不急着全部写成完整 prompt。

这样做的好处是：

- 目录结构已经稳定
- 插件页面后续可直接读取能力域
- 当前维护成本不会失控

## 7. skill 应该怎么组织

结论先说：

> 不建议按专家名组织 skill，建议按复用范围和技术栈组织。

### 7.1 推荐结构

```text
.agents/skills/
├── common/
├── profiles/react/
├── profiles/vue/
└── domains/        # 预留，按需启用
```

### 7.2 什么时候放 `common/`

与技术栈无关、多个专家都能用的技能，放 `common/`。

例如：

- `create-proposal`
- `design-analysis`
- `execute-task`
- `ui-verification`

### 7.3 什么时候放 `profiles/<stack>/`

与 React 或 Vue 强绑定的技能，放到对应 profile 目录。

例如：

- `create-component`
- `create-route`
- `create-view`
- `create-api`
- `create-store`
- `theme-variables`

### 7.4 什么时候放 `domains/`

当一个 skill 同时满足下面两个条件时，再放 `domains/`：

- 明显属于某个能力域，例如性能、安全、可观测性
- 会被多个专家复用，而不是只服务一个专家

例如未来可以出现：

- `domains/performance/lighthouse-audit`
- `domains/observability/sentry-triage`
- `domains/security-a11y/security-review`

### 7.5 为什么不建议 `skills/<expert-name>/`

因为 expert 和 skill 是多对多关系：

- 一个专家会调用多个 skill
- 一个 skill 会被多个专家复用

如果按专家名组织，后续会出现大量重复和迁移成本。

### 7.6 如果当前阶段仍想用 `skills/<expert-id>/`

如果你当前更想让目录先贴近“专家协同”的表达，也不是完全不能做。

但要注意：

- 这只能作为**过渡组织方式**
- 不建议把目录名直接当成 `domains（能力域）`
- 必须额外补一层元数据映射，否则后续 `sync（同步）` 很难稳定聚合 `domains（能力域）`

最小可行做法是：

1. `skill（技能）` 文件声明 `owner_role（归属专家）`
2. `role（专家角色）` 文件声明 `domains（能力域）`
3. `sync（同步）` 先从已选 `roles（专家角色）` 聚合 `domains（能力域）`
4. 如需更细粒度，再从 `skill（技能）` 元数据补充

也就是说，真正可靠的来源是**元数据**，不是目录结构本身。

## 8. 当前项目的最稳方案

针对你现在这个项目，我建议直接采用下面这套：

- 当前启用的专家，继续放 `.agents/roles/common/`
- 规划中的能力域，占位放 `.agents/roles/domains/`
- skill 继续采用 `.agents/skills/common/ + profiles/` 结构
- `domains/` 先预留，不急着一开始填满

一句话收束：

> 先把角色边界、交接和技能组织方式定稳，再逐步补专家；不要反过来先堆很多专家文件，最后发现结构不可维护。
