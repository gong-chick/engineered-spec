# IDE（开发工具）里触发 run（运行编排） 的最小交互协议

## 1. 文档目的

本文档定义当前阶段在 `IDE（开发工具） AI（智能体）` 中触发 `run（运行编排）` 的最小交互协议。

它要解决 4 个问题：

1. 什么话算触发
2. 触发后 `task-orchestrator（任务主代理）` 先输出什么
3. 缺信息时怎么补问
4. 什么时候交给 `frontend-implementer（前端实现专家）` 或其他实现专家

这份协议的定位是：

> 先把 `run（运行编排）` 做成稳定的运行时入口，而不是先做一个很重的 CLI（命令行工具）子命令。

## 2. 当前阶段的触发原则

当前建议采用：

- **显式触发优先**
- **弱自然语言触发作为补充**
- **普通问答不默认进入运行编排**

也就是说，当前不建议让系统看到任何开发话题都自动进入 `run（运行编排）`。

更稳的做法是：

- 用户明确表达“开始做事”
- `task-orchestrator（任务主代理）` 再接管任务收敛和编排

当前仓库已经补了更贴近真实使用的项目级命令入口：

- `/spec-start`
- `/spec-continue`

兼容入口：

- `/spec-orchestrate`

## 3. 什么话算触发

### 3.1 强触发

以下表达建议直接进入 `run（运行编排）`：

- `@task-orchestrator（任务主代理） 创建一个商品组件`
- `开始规范驱动开发：创建一个商品组件`
- `请按当前项目规范执行：新增商品详情页`
- `进入专家协同模式：实现商品列表页`
- `启动任务：根据设计稿开发商品卡片`

这些话的特点是：

- 带有明确动作
- 带有明确交付意图
- 明确要求按规范或按当前项目方式执行

### 3.2 弱触发

以下表达可以视为“可能要进入 `run（运行编排）`”，但建议由 `task-orchestrator（任务主代理）` 先做一次确认：

- `创建一个商品组件`
- `新增商品详情页`
- `根据设计稿做这个页面`
- `补一下这个接口对应的页面`

这些话的特点是：

- 有明确开发意图
- 但没有明确说“现在就开始按规范执行”

当前建议的处理方式是：

> 先识别为候选任务，再由 `task-orchestrator（任务主代理）` 输出简短确认或直接进入最小计划生成。

### 3.3 不触发

以下表达不应直接进入 `run（运行编排）`：

- `这个项目怎么接 Hub（平台）？`
- `为什么要做专家协同？`
- `帮我解释一下 flow（流程模板）`
- `你觉得这个架构怎么样？`
- `有哪些方案可以选？`

这些属于：

- 解释
- 讨论
- 方案比较
- 认知对齐

应按普通对话处理，不默认启动任务。

## 4. 触发后的第一条输出应该是什么

当前阶段，`task-orchestrator（任务主代理）` 触发后**不建议直接开始写代码**。

第一条输出应先给出一个最小 `run-plan（运行计划）` 摘要，至少包含：

1. 任务识别结果
2. 推荐的 `flow（流程模板）`
3. 计划激活的 `role（专家角色）`
4. 当前缺失输入
5. 下一步动作

并在准备交给第一位专家前，补一份对应的 `task-anchor（任务锚点）`。

### 4.1 推荐输出模板

```md
## 任务识别
- 类型：组件开发
- 当前输入：创建一个商品组件
- 推荐流程模板：prd-to-delivery（需求到交付）

## 推荐专家
- 必选：task-orchestrator（任务主代理）、frontend-implementer（前端实现专家）、code-guardian（规范守护者）
- 可选：requirement-analyst（需求解析专家）

## 缺失输入
- 组件放在哪个页面或目录
- 是否有设计稿
- 是否需要接口数据

## 下一步
- 先补齐关键信息，再交给 frontend-implementer（前端实现专家）
```

统一模板来源见：

- [task-orchestrator-run-plan-template.md](../../.agents/orchestration/task-orchestrator-run-plan-template.md)

### 4.2 为什么先输出这个

因为这样可以保证：

- 用户知道系统把任务识别成什么
- 用户知道系统准备调用哪些专家
- 用户知道缺什么信息
- 系统不会在信息不足时直接误写代码

同时建议在首次生成 `run-plan（运行计划）` 后立刻组装首轮桥接载荷，并自动落盘当前运行状态：

- `.ai-spec/current-run.json`
- `.ai-spec/runs/<run-id>.json`

对应规范见：

- [运行状态落盘规范-03-30-14-58.md](运行状态落盘规范-03-30-14-58.md)
- [task-anchor-spec.md](../../.agents/orchestration/task-anchor-spec.md)
- [task-orchestrator-bootstrap-payload.md](../../.agents/orchestration/task-orchestrator-bootstrap-payload.md)

推荐调用：

```bash
ai-spec-auto task-orchestrator-extractor apply --payload ./.ai-spec/tmp/task-orchestrator-reply.md
```

推荐优先通过安装后的项目级命令触发：

- `Cursor（开发工具）`：`/spec-start` / `/spec-continue`
- `Claude（开发工具）`：`/spec-start` / `/spec-continue`

兼容入口：

- `/spec-orchestrate`

## 5. 缺信息时怎么问

当前建议使用：

- **少量问题**
- **只问关键问题**
- **优先问会影响实现路径的问题**

### 5.1 补问顺序

优先级建议如下：

1. 交付对象是否明确
2. 位置和范围是否明确
3. 是否有设计稿 / PRD（产品需求文档）
4. 是否需要接口或状态管理
5. 是否有明确验收标准

### 5.2 补问原则

- 一轮最多问 2 到 4 个关键问题
- 不要一次把所有可能问题都抛给用户
- 已能从 `context（上下文）` 或已有 `openspec（变更产物）` 获取的信息，不重复问
- 能先用项目约定推断的，不先打断用户

### 5.3 针对“创建一个商品组件”的补问示例

推荐优先补问：

1. 这是 `Vue（前端框架）` 组件还是 `React（前端框架）` 组件？
2. 组件准备放在哪个页面或目录？
3. 有没有设计稿或参考样式？
4. 这个组件是纯展示，还是需要接口数据和交互？

### 5.4 信息不足时的推荐话术

```md
我已将该需求识别为组件开发任务。继续前需要补齐几个关键输入：

1. 组件使用的是 Vue（前端框架）还是 React（前端框架）？
2. 组件准备放在哪个页面或目录？
3. 是否有设计稿、截图或现有参考组件？
4. 该组件是否需要接口数据或状态交互？
```

## 6. 什么时候交给实现专家

当前建议只有在下面条件基本满足时，才交给 `frontend-implementer（前端实现专家）`：

### 6.1 可以交接的条件

- 已识别出任务类型
- 已明确交付对象
- 已知项目技术栈
- 目录位置或目标页面基本明确
- 关键阻断信息已补齐，或已显式记录为假设
- 如果存在高风险假设，已标记审批点

### 6.2 暂时不要交接的情况

- 任务目标本身不清晰
- 页面 / 组件边界不清晰
- 设计稿和规则冲突但未确认
- 是否需要接口、状态、埋点完全未知
- 当前更适合先由 `requirement-analyst（需求解析专家）` 收敛

### 6.3 推荐交接路径

#### 路径 A：信息足够

```text
task-orchestrator（任务主代理）
  -> frontend-implementer（前端实现专家）
  -> code-guardian（规范守护者）
```

#### 路径 B：信息不足

```text
task-orchestrator（任务主代理）
  -> requirement-analyst（需求解析专家）
  -> frontend-implementer（前端实现专家）
  -> code-guardian（规范守护者）
```

## 7. 一个完整示例

### 7.1 用户输入

```text
@task-orchestrator（任务主代理） 创建一个商品组件
```

### 7.2 主代理第一轮输出

```md
## 任务识别
- 类型：组件开发
- 推荐流程模板：prd-to-delivery（需求到交付）

## 推荐专家
- 必选：frontend-implementer（前端实现专家）、code-guardian（规范守护者）
- 可选：requirement-analyst（需求解析专家）

## 缺失输入
- 所属技术栈
- 组件目录位置
- 是否有设计稿
- 是否需要接口数据

## 下一步
- 请先补齐上述信息，再进入实现
```

### 7.3 用户补充

```text
Vue（前端框架）项目，放在商品详情页，有设计稿，是纯展示组件
```

### 7.4 主代理第二轮输出

```md
## 运行计划
- flow（流程模板）：prd-to-delivery（需求到交付）
- first_handoff（第一跳交接）：frontend-implementer（前端实现专家）
- approval_gates（审批点）：无

## 执行说明
- 当前输入已足够进入实现
- 将按 Vue（前端框架）项目规范创建组件，并在交付前交给 code-guardian（规范守护者）检查
```

## 8. 当前阶段建议

当前阶段最稳的做法是：

1. **显式触发优先**
2. **先输出最小计划，不直接写代码**
3. **缺信息先补问**
4. **信息足够再交给实现专家**

这样做的好处是：

- 可控
- 不容易误触发
- 更符合“规范驱动 + 专家协同”的定位
- 后续更容易接入 OpenClaw（远程入口）和插件页面

## 9. 一句话收束

> 在 `IDE（开发工具）` 中触发 `run（运行编排）`，当前最合理的方式不是让系统自由猜测，而是通过显式触发 `task-orchestrator（任务主代理）` 进入任务编排；主代理先输出最小计划、补齐关键信息，确认可以落地后，再交给实现专家。
