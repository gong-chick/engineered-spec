# 流程模板矩阵与分阶段建设建议

本文档用于回答两个问题：

1. 当前平台后续到底需要哪些流程模板
2. 除了当前最小模板之外，还有哪些值得保留的模板方向

结论先说：

- 当前不应该只靠 `prd-to-delivery` 一条模板覆盖所有任务
- 也不应该一开始就堆十几条模板
- 最稳的做法是：**先落 4 条核心模板，再为 3 条扩展模板预留规划位**

## 1. 为什么要做“流程模板矩阵”

流程模板不是专家集合的替代物，也不是把所有步骤写死的脚本。

它的作用是：

- 给主代理一个可复用的基础骨架
- 让不同入口都能稳定选择任务类型
- 让 OpenClaw、CLI、插件和 Hub 平台共享同一套执行分类
- 让产物和审批点可预测、可审计

一句话：

> 模板解决“这类任务通常怎么协作”，主代理解决“这次任务具体激活哪些专家”。

## 2. 推荐的模板分层

建议分成两层：

### 2.1 第一梯队：核心模板

这几条模板应优先真正落地，并逐步接入 `ai-spec-auto run`。

### 2.2 第二梯队：扩展模板

这几条模板有价值，但不建议在当前阶段就全部实现，只保留规划位即可。

## 3. 第一梯队：4 条核心模板

### 3.1 `prd-to-delivery`

定位：

- 需求或设计输入到交付结果的基础模板

适用场景：

- 新功能开发
- 设计稿还原
- 有明确范围的增量开发

典型输入：

- PRD
- 设计稿
- 自然语言需求
- 已有 `proposal.md / tasks.md`

典型输出：

- `proposal.md`
- `tasks.md`
- 代码实现
- `checklist.md`
- `iterations.md`

当前状态：

- 已存在，且已升级为流程模板结构

说明：

- 这条模板虽然命名叫 `prd-to-delivery`，但当前实际语义已经更接近“需求/设计驱动的交付模板”
- 后续若要统一命名，可以演进到 `requirement-to-delivery`

### 3.2 `bugfix-to-verification`

定位：

- 问题修复到验证结论的模板

适用场景：

- 线上 bug 修复
- 回归修复
- 功能异常修复

典型输入：

- 缺陷描述
- 报错日志
- 复现路径
- 受影响范围

典型输出：

- 问题分析结论
- 修复实现
- 验证结果
- 风险说明

建议必选专家：

- `task-orchestrator`
- `frontend-implementer`
- `code-guardian`

建议可选专家：

- `verification-reviewer`
- `unit-test-specialist`
- `error-tracker`

优先级：

- 高

说明：

- 这是除交付类模板之外，最值得尽快补的第二条模板

### 3.3 `doc-or-prd-production`

定位：

- 文档生产模板，不以代码落地为主

适用场景：

- PRD 编写
- 技术方案撰写
- API 文档整理
- 组件或项目文档沉淀

典型输入：

- 业务需求说明
- 会议纪要
- 设计草稿
- 现有代码或接口信息

典型输出：

- PRD
- 方案文档
- API 文档
- README / 技术说明

建议必选专家：

- `task-orchestrator`
- `requirement-analyst`

建议可选专家：

- `technical-writing-specialist`
- `api-doc-specialist`
- `component-doc-specialist`

优先级：

- 高

说明：

- 这条模板可以把“产品文档编写”正式纳入平台能力，不再只围绕前端代码

### 3.4 `delivery-or-ops-change`

定位：

- 发布、部署、流水线和环境改造模板

适用场景：

- CI/CD 调整
- Docker / K8s 配置
- 环境变量和部署链路改造
- 交付策略变更

典型输入：

- 发布需求
- 环境变更说明
- 现有流水线配置

典型输出：

- 流水线改动
- 部署脚本或配置改动
- 交付检查结论

建议必选专家：

- `task-orchestrator`
- `code-guardian`

建议可选专家：

- `pipeline-specialist`
- `container-specialist`
- `deployment-specialist`

优先级：

- 中高

说明：

- 这条模板可以把“交付”和“工程化变更”从开发模板里拆出来

## 4. 第二梯队：3 条扩展模板

这 3 条模板我认为是有价值的，但不建议当前阶段优先实现。

### 4.1 `research-to-rfc`

定位：

- 探索、调研到方案收敛模板

适用场景：

- 技术选型
- 大方向评估
- 多方案比较
- 新能力预研

典型输出：

- RFC / 方案评审材料
- 风险和取舍说明

价值：

- 适合领导层和技术负责人决策前使用

当前建议：

- 先保留规划位，不进入 MVP

### 4.2 `refactor-to-regression-check`

定位：

- 重构到回归验证模板

适用场景：

- 目录调整
- 组件重构
- 状态管理重构
- 样式体系迁移

典型输出：

- 重构代码
- 回归验证结论
- 风险和影响面说明

价值：

- 对中大型项目非常有用

当前建议：

- 等 `verification-reviewer`、`unit-test-specialist` 更成熟后再落

### 4.3 `incident-to-postmortem`

定位：

- 故障处理到复盘模板

适用场景：

- 线上事故
- 高优故障
- 紧急恢复和事后复盘

典型输出：

- 事故分析
- 修复动作
- 风险结论
- postmortem 文档

价值：

- 对 OpenClaw 远程办公和群内调度非常有吸引力

当前建议：

- 先作为远期规划位

## 5. 模板矩阵总表

| 模板 ID | 任务类型 | 当前建议 | 优先级 | 当前状态 |
| --- | --- | --- | --- | --- |
| `prd-to-delivery` | 需求 / 设计到交付 | 立即保留并继续打磨 | P0 | 已落地 |
| `bugfix-to-verification` | 问题修复与验证 | 下一条优先补 | P0 | 规划中 |
| `doc-or-prd-production` | 文档生产 | 近期补位 | P1 | 规划中 |
| `delivery-or-ops-change` | 发布与交付改造 | 近期补位 | P1 | 规划中 |
| `research-to-rfc` | 调研与方案收敛 | 保留规划位 | P2 | 规划中 |
| `refactor-to-regression-check` | 重构与回归验证 | 保留规划位 | P2 | 规划中 |
| `incident-to-postmortem` | 故障与复盘 | 保留规划位 | P2 | 规划中 |

## 6. 为什么不是更多

除了上面 7 条，我当然还能继续列更多模板，比如：

- `design-review-to-implementation`
- `api-change-to-consumer-update`
- `data-report-production`
- `compliance-or-security-audit`

但当前阶段不建议继续扩。

原因很直接：

- 模板太多会导致主代理路由边界变模糊
- CLI 和 OpenClaw 很难解释“到底为什么选这条模板”
- 团队分享时会显得空泛，像在堆概念

所以当前更合理的边界是：

- 先稳定 4 条核心模板
- 再为 3 条扩展模板留位
- 其它方向先不进入正式矩阵

## 7. 推荐的实施顺序

### Phase 1

- 稳定 `prd-to-delivery`
- 打通 `task-orchestrator` 的动态选专家
- 接上 `ai-spec-auto run`

### Phase 2

- 补 `bugfix-to-verification`
- 补 `doc-or-prd-production`

### Phase 3

- 补 `delivery-or-ops-change`
- 根据试点情况决定是否推进第二梯队模板

## 8. 当前建议

如果只考虑“最值当的投入”，当前建议就是：

1. 把 `prd-to-delivery` 作为主模板继续打磨
2. 把 `bugfix-to-verification` 作为下一条真实模板
3. 用 `doc-or-prd-production` 把产品文档和技术文档纳入平台
4. 把 `delivery-or-ops-change` 作为工程交付类模板的边界

这样既能覆盖：

- 产品文档
- 开发实现
- 测试验证
- 交付上线

又不会把模板体系做得过重。
