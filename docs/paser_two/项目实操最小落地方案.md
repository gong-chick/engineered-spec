# 项目实操最小落地方案

## 1. 这份方案解决什么问题

针对当前项目 `AI 规范驱动开发 CLI`，当前最重要的不是继续扩展概念，而是先落一套：

- 可演示
- 可扩展
- 不复杂
- 层级不深
- 能和现有 `.agents`、`openspec/` 协同

的最小文件与流程方案。

它主要服务两件事：

1. 支撑团队内部“项目实操”分享
2. 为后续专家协同开发提供一个稳定、浅层的上下文底座

## 2. 当前项目应该怎么理解

当前项目不是两条独立产品线，而是一个产品的两层表达：

```text
AI 规范驱动开发 CLI
  -> 实操层：专家协同开发
  -> 方法层：AI 工程方法论
```

所以当前改造优先级应该是：

- 先把“专家协同开发”的最小闭环做出来
- 同时用最小方法论解释它为什么成立

不建议现在同时铺开完整平台、完整专家矩阵、完整方法论资产体系。

## 3. 最小可扩展文件结构

针对**当前项目自身**，更推荐的最小 MVP 结构不是“纯 `context/`”，而是：

```text
ai-spec-auto/
├── .agents/
│   ├── rules/
│   ├── skills/
│   ├── roles/               # MVP 新增
│   └── flows/               # MVP 新增
├── context/
│   └── PROJECT.md           # 稳定项目背景
└── openspec/
    └── changes/<change-id>/
        ├── proposal.md
        ├── tasks.md
        ├── checklist.md     # MVP 新增
        └── iterations.md    # MVP 新增
```

这个结构有 3 个特点：

- 规则、技能、专家、流程都在 `.agents/` 下，和后续专家协同结构一致
- `context/` 只保留稳定背景，不和变更文档混在一起
- 需求级变化全部挂到 `openspec/changes/<change-id>/`，后续扩展不会乱

## 4. 为什么先用 6 个 md，不上更复杂体系

当前阶段不建议一开始就上 7 个、10 个甚至更多文档文件。

先用下面这些就够了：

- `context/PROJECT.md`
- `openspec/changes/<change-id>/proposal.md`
- `openspec/changes/<change-id>/tasks.md`
- `openspec/changes/<change-id>/iterations.md`
- `openspec/changes/<change-id>/checklist.md`
- `.agents/roles/` 下的 3 个专家定义
- `.agents/flows/` 下的 1 条流程定义

原因：

- 够用，能支撑一个完整 demo
- 不会让项目结构变重
- 团队更容易接受
- 后续再加更多专家和流程时，骨架不需要推翻

## 5. 这 6 个文件分别承担什么角色

### `context/PROJECT.md`

放稳定背景，只改少量内容：

- 项目定位
- 业务目标
- 核心术语
- 角色模型
- 架构概览
- 非目标

### `.agents/rules/`

放硬约束。  
对当前项目来说，这部分已经存在，不需要再额外做一份 `RULES.md` 复制一遍。

你真正需要的是：

- 把 `.agents/rules/README.md` 当成规则入口索引
- 在专家和流程里引用对应规则

### `openspec/changes/<change-id>/proposal.md`

放当前需求设计，是这次变更的核心说明。

建议包含：

- 问题定义
- 目标与非目标
- 页面 / 功能设计
- 接口与数据流
- 风险

### `openspec/changes/<change-id>/tasks.md`

放执行拆解，尽量短而明确：

- 任务列表
- 执行顺序
- 依赖
- 验收点

### `openspec/changes/<change-id>/iterations.md`

放关键过程信息，不记流水账，只记有价值的：

- 试过什么
- 为什么失败
- 为什么改方案
- 当前遗留风险

### `openspec/changes/<change-id>/checklist.md`

放验证清单：

- lint
- typecheck
- test
- build
- review
- 性能
- 安全

## 6. 它和 spec 的关系

针对当前项目，`context/` 应该主要承载**稳定信息**，而不是把每次需求都往里面堆。

建议这样理解：

- `context/PROJECT.md` 是稳定项目上下文
- `.agents/rules/` 是稳定规则上下文
- `openspec/changes/<change-id>/proposal.md` / `tasks.md` 是当前变更上下文
- `openspec/changes/<change-id>/iterations.md` / `checklist.md` 是本次变更的反馈和验证上下文

而 `spec` 主要相当于这套结构中的：

- `proposal.md`
- `tasks.md`
- 部分关键决策说明

所以更准确的说法是：

> `context/` 负责稳定背景，`openspec/changes/` 负责需求增量，`.agents/` 负责规则与专家协同。三者各司其职，不重复维护。

## 7. 当前项目最适合演示的黄金路径

建议你分享时只演示一条路径，不要把所有能力一次讲完。

推荐黄金路径：

```text
输入 PRD / 设计
  -> 需求解析专家产出 proposal.md
  -> 拆成 tasks.md
  -> 前端实现专家开发
  -> 规范守护者检查
  -> checklist.md 验证
  -> iterations.md 记录结果
```

也就是说，当前项目不建议再做一份：

```text
context/DESIGN.md
context/TASKS.md
```

否则就会形成重复源。

## 8. 当前阶段建议只做 3 个专家

为了保证分享聚焦，建议先只演示 3 个专家：

1. `需求解析专家`
   - 输入 PRD / 设计
   - 输出 `proposal.md`

2. `前端实现专家`
   - 输入 `proposal.md` + `tasks.md`
   - 输出代码实现

3. `规范守护者`
   - 输入实现结果
   - 输出规则检查和修正建议

这个组合最容易解释，也最符合当前项目定位。

## 9. 当前阶段不要先做什么

先不要优先做：

- 完整专家矩阵
- 多层复杂目录
- 完整可观测平台
- 完整角色权限系统
- 过多流程分支

这些方向都对，但不适合当前作为第一轮项目实操分享的重点。

## 10. 对当前项目的具体改造建议

围绕当前项目，建议先做下面这些最小改造：

1. 在仓库中提供 `templates/context-pack/`
   - 作为通用模板，而不是当前项目的主结构

2. 在文档里明确 `spec`、`context`、`.agents` 的关系
   - 避免团队误以为是两套体系

3. 在 `.agents/` 下新增最小 `roles/` 和 `flows/` 骨架
   - 直接对齐后续专家协同结构

4. 分享只围绕一个 demo 场景
   - `PRD/设计 -> proposal -> tasks -> 开发 -> 校验 -> 记录`

5. 先用 3 个专家，别一开始铺太满

6. `context/` 只保留稳定背景
   - 不要把每次需求的 DESIGN/TASKS 一直往里面堆

## 11. 最后的建议

如果以“项目实操分享”为目标，当前最优策略不是继续扩概念，而是：

> 用一套稳定上下文 + 一条 OpenSpec 变更链路 + 三个核心专家，把 `AI 规范驱动开发 CLI` 讲清楚、跑通、讲透。

这比一开始就做成完整平台更稳，也更容易让团队理解和接受。
