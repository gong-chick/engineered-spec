# IDE真实触发入口

## 1. 目标

当前单动作链已经接入项目级 `IDE（开发工具）` 命令入口。

安装后推荐优先在 `IDE（开发工具）` 中使用：

- `Cursor（开发工具）`：`/spec-start` / `/spec-continue`
- `Claude（开发工具）`：`/spec-start` / `/spec-continue`

兼容入口：

- `/spec-orchestrate`

## 2. 这条命令做什么

这些命令都不直接承担业务实现，它们负责：

1. 触发 `task-orchestrator（任务主代理）`
2. 生成首轮或单动作 payload（载荷）
3. 写入 `./.ai-spec/tmp/task-orchestrator-reply.md`
4. 调用：
   - 生产环境优先：由宿主层调用内部 `task-orchestrator-runner.advanceRunner({ target })`
   - 底层回退：`ai-spec-auto task-orchestrator-extractor apply --payload ./.ai-spec/tmp/task-orchestrator-reply.md --target .`
5. 自动更新：
   - `.ai-spec/current-run.json`
   - `.ai-spec/runs/<run-id>.json`
6. 如需继续交给当前专家执行：
   - 由 `task-orchestrator（任务主代理）` 继续产出 `./.ai-spec/tmp/current-dispatch.json`
   - 由当前专家产出 `./.ai-spec/tmp/current-execution.json`
   - 由 `task-orchestrator（任务主代理）` 再产出 `./.ai-spec/tmp/current-runtime-action.json` 或新的 `task-orchestrator-reply.md`
   - 每一步都继续由宿主层调用：`task-orchestrator-runner.advanceRunner({ target })`

## 3. 适用场景

### 3.1 新任务启动

```text
/spec-start 创建一个商品组件
```

### 3.2 已有运行态继续推进

```text
/spec-continue 审批通过，继续
/spec-continue 当前任务取消
/spec-continue 查看当前状态
```

## 4. 当前价值

这一步完成后，系统已经具备：

> 从 `IDE（开发工具）` 命令触发，到主代理输出，到抽取器，到适配层，到运行态写盘的真实闭环。

在当前推荐模式下，这条闭环应理解为：

- `AI（智能体）` 负责产出结构化内容
- `Runner（运行器）` 负责消费 `.ai-spec/tmp/` 当前输入并推进状态
- 本地命令不替 `AI（智能体）` 做角色和动作推理

因此现在不只是：

- 有协议
- 有底层命令

而是已经有：

- 可分发的项目级命令入口
- 可安装的 `Cursor（开发工具） / Claude（开发工具）` 触发模板
- 可落盘的最小运行闭环
