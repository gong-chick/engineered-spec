# Phase 1（第一阶段）执行清单与优先级

## 1. 目的

本文档用于收敛当前 `Phase 1（第一阶段）` 的 4 项工作，明确：

- 哪些已经有规范基础
- 哪些现在就能落
- 哪些还缺运行时实现
- 当前应先做哪一项

## 2. 总结结论

当前阶段 4 项里：

1. `task-orchestrator（任务主代理）` 首轮 `run-plan（运行计划）`
   - **优先级最高**
   - **现在就能做**
2. 每轮执行强制注入 `task-anchor（任务锚点）`
   - 现在能做
   - 已补统一注入规范，后续可直接接入交接协议
3. `run-state（运行状态）` 落盘
   - 规范已齐
   - 已补最小写盘工具，下一步是接入主代理运行链
4. 专家之间优先传 `OpenSpec（规范产物） + run-state（运行状态）`
   - 协议可先定
   - 完整执行约束需要后续实现

因此，当前最合理的顺序是：

```text
1. 先稳定首轮 run-plan（运行计划）
2. 再固化 task-anchor（任务锚点） 注入
3. 再接 run-state（运行状态） 写盘
4. 最后收紧专家间快照传递
```

## 4. 为什么先做 `run-plan（运行计划）`

因为它是后面 3 项的基础：

- 没有统一 `run-plan（运行计划）`
  - 就不知道首轮该怎么落 `run-state（运行状态）`
- 没有统一 `run-plan（运行计划）`
  - `task-anchor（任务锚点）` 很难标准化
- 没有统一 `run-plan（运行计划）`
  - 专家交接也没有明确起点

所以当前最稳的做法不是先写运行器，而是先把“主代理第一轮到底怎么说、说成什么结构”定稳。

## 5. 第一优先级的交付目标

本轮先完成下面 3 件事：

1. 补一份 `task-orchestrator（任务主代理）` 的标准 `run-plan（运行计划）` 模板
2. 在主代理角色定义里明确首轮输出必须遵守这份模板
3. 在运行契约和 `IDE（开发工具）` 触发协议里引用同一份模板

完成后，至少可以保证：

- 首轮输出字段统一
- `run-plan（运行计划）` 不再各说各话
- 后续 `task-anchor（任务锚点） / run-state（运行状态）` 有稳定落点

## 6. 本轮不做的事

当前先不做：

- `run-state（运行状态）` 自动更新和多阶段写盘
- 完整 `approve（审批） / resume（恢复） / status（状态） / complete（完成） / fail（失败）` 状态机
- 专家输入自动裁剪器
- 复杂状态机或回滚机制

这些内容保留到后续阶段，不阻塞当前第一优先级落地。

## 7. 一句话结论

> 当前 `Phase 1（第一阶段）` 最该先做的，不是继续扩专家，而是先把 `task-orchestrator（任务主代理）` 的首轮 `run-plan（运行计划）` 输出模板做成全局统一入口。
