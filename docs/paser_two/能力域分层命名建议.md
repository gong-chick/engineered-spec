# 能力域分层命名建议

## 1. 目的

这份文档用于统一插件页面、分享材料和后续专家体系中的一级分类命名。

目标只有三个：

- 更准确
- 更真实
- 更可扩展

## 2. 为什么不建议直接叫“研发阶段”

你当前图片里的一级分类并不全是线性阶段。

例如：

- `需求设计`
- `测试`
- `CI/CD`

这些可以理解为阶段或流程节点。

但下面这些更像治理域或能力域，而不是阶段：

- `代码规范`
- `工程化`
- `性能`
- `可观测性`

所以如果统一叫：

```text
研发阶段 -> 专家 -> 技能
```

概念上会有一点不准确。

## 3. 更推荐的一级命名

如果这套结构主要用于：

- 插件页面展示
- 专家能力地图
- 对外介绍

我更推荐一级统一叫：

> **能力域**

这样你的整体表达就会变成：

```text
能力域 -> 专家 -> 技能
```

这是我认为目前最稳的一版。

## 4. 为什么“能力域”比“阶段”更合适

因为“能力域”能同时包住两类东西：

### 4.1 主流程类

比如：

- 需求设计
- 测试验证
- 发布交付

### 4.2 横切治理类

比如：

- 规范治理
- 工程构建
- 性能体验
- 可观测治理

这样分类后，逻辑会更统一，不会把“阶段”和“治理维度”混在一个词里。

## 6. 如果后面补安全和无障碍，怎么放

如果你后面要把安全和无障碍补进来，我建议不要硬塞进已有域里，而是新增一个独立一级域：

### 9. 安全与可访问性域

包含专家示例：

- 安全审查专家
- A11y 审计专家
- ARIA 专家

这样更完整，也不会破坏现有结构。

## 7. 展示层和执行层要分开理解

这里有一个很关键的点：

### 页面展示层

建议用：

```text
能力域 -> 专家 -> 技能
```

因为这是给用户看、给团队理解的能力地图。

### 系统执行层

建议用：

```text
规则/上下文 -> 任务主代理 -> 流程 -> 专家 -> 技能
```

因为这是系统真实运行时的编排逻辑。

所以：

- 对外展示，用“能力域”
- 对内执行，用“流程”

这两者不冲突，反而是最真实的设计。

## 8. 最终推荐文案

如果你要一版可以直接放进图里或文档里的话，我建议写成：

> 为了同时覆盖前端交付链路与横切治理能力，本项目采用“能力域 -> 专家 -> 技能”的展示分层。在系统执行层，则采用“规则/上下文 -> 任务主代理 -> 流程 -> 专家 -> 技能”的编排模型。前者用于表达能力地图，后者用于驱动真实协作流程。

## 9. 最后的建议

所以我的最终建议是：

- **图片和插件页**：用 `能力域 -> 专家 -> 技能`
- **底层实现和流程编排**：用 `规则/上下文 -> 任务主代理 -> 流程 -> 专家 -> 技能`

这样概念最准确，后续扩展也最稳。
