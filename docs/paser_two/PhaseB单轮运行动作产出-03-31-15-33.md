# Phase B（第二步）单轮运行动作产出

## 1. 目标

Phase B（第二步） 的目标不是自动推进整条链，而是让 `task-orchestrator（任务主代理）` 产出的：

- `current-runtime-action（当前运行动作草案）`

可以被系统稳定记录、回放和消费。

也就是：

> 让“当前专家这一轮结束后建议做什么”继续保持在 `task-orchestrator（任务主代理）` 里决策，但在本地拥有稳定的结构化落盘。

## 2. 当前实现

当前命令：

```bash
ai-spec-auto expert-executor apply-action --payload ./.ai-spec/tmp/current-runtime-action.json
```

或：

```bash
cat ./.ai-spec/tmp/current-runtime-action.json | ai-spec-auto expert-executor apply-action --stdin
```

## 3. 当前落盘

会写入：

- `.ai-spec/current-runtime-action.json`
- `.ai-spec/current-runtime-action.md`
- `.ai-spec/runtime-actions/<run-id>/<action-id>.json`
- `.ai-spec/runtime-actions/<run-id>/<action-id>.md`

## 4. 当前边界

当前这一步只做：

- 校验 `task-orchestrator-runtime-action（主代理运行动作载荷）`
- 落盘

当前这一步不做：

- 本地推理是 `handoff（交接）`、`gate-blocked（阻断）` 还是 `complete（完成）`
- 自动调用 `task-orchestrator-adapter（自动执行适配层）`
- 自动递归推进下一轮
- 自动连跑完整专家链

## 5. 当前价值

这一步的价值是把系统从：

- “只有运行状态，没有标准下一步动作”

推进到：

- “`task-orchestrator（任务主代理）` 能给出标准下一步动作，系统也能稳定记录和消费”

这样后续不管是：

- `IDE（开发工具）`
- OpenClaw（远程入口）
- 插件页面

都可以消费同一份动作草案，而不是让本地脚本替 `task-orchestrator（任务主代理）` 做语义推理。
