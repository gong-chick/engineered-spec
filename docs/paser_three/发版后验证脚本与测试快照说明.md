# 发版后验证脚本与测试快照说明

这份文档对应一个固定目标：

- 每次 `npm publish` 之后，快速验证“已发版包”在测试项目里是否还能正常工作
- 自动生成一份本地测试快照，方便查看本轮验证结果
- 看完后可以直接删除，不影响仓库主线

当前固定脚本是：

- [post-publish-verify.sh](../../scripts/post-publish-verify.sh)

当前快照目录是：

- [test-snapshots](test-snapshots)

## 这份脚本会做什么

脚本会按固定顺序做 3 件事：

1. 重新安装并同步测试项目
2. 跑一次 `demo-runtime-smoke`
3. 跑一次 `auto-fix` 真实 CLI 验证

它验证的是“发版后的安装包”而不是本地源码直调，所以更贴近真实使用。

## 一键运行

默认测试项目目录：

```bash
/Users/lizhenwei/workspace/test/test-ai-spec/ai-spec-cursor-test
```

如果刚发完版，最推荐这样跑：

```bash
AI_SPEC_PACKAGE_SPEC='@engineered/ai-spec-auto@0.0.41' \
bash ./scripts/post-publish-verify.sh
```

如果你要指定别的测试项目目录：

```bash
AI_SPEC_PACKAGE_SPEC='@engineered/ai-spec-auto@0.0.41' \
bash ./scripts/post-publish-verify.sh \
  /Users/lizhenwei/workspace/test/test-ai-spec/ai-spec-cursor-test
```

如果不传 `AI_SPEC_PACKAGE_SPEC`，脚本会默认使用当前仓库 [package.json](../../package.json) 的 `name@version`。

## 具体验证内容

### 1. 安装/同步

脚本会先调用：

- [setup-cursor-spec-archive-test.sh](../../scripts/setup-cursor-spec-archive-test.sh)

它会在测试项目里完成：

- `init`
- `sync`
- `validate-registry`

### 2. smoke 闭环

脚本会调用测试项目里已安装的：

```bash
./node_modules/.bin/ai-spec-auto demo-runtime-smoke --target ./.tmp/runtime-smoke-demo --json
```

这一步主要确认：

- 已发版 CLI 能正常执行
- `expert-delivery` schema 仍能跑通
- 运行态与归档主链没有回退

### 3. auto-fix 回环

脚本会再调用：

- [post-publish-auto-fix-check.js](../../scripts/post-publish-auto-fix-check.js)

这一步会在测试项目里创建一个临时工作区，验证：

- `frontend-implementer` 首次验证失败后，只会自动回自己一次
- 第二次修复通过后，会流转到 `code-guardian`
- `review_contract` 能读到 `latest_auto_fix` 和 `latest_verification`
- `.ai-spec/checkpoints/` 和 `.ai-spec/repo-map.json` 都能落盘

## 测试快照会生成什么

每次执行后，都会在：

- [test-snapshots](test-snapshots)

生成一组本地文件，命名形如：

```text
20260410-103000-0.0.41-post-publish.md
20260410-103000-0.0.41-post-publish.setup.log
20260410-103000-0.0.41-post-publish.smoke.json
20260410-103000-0.0.41-post-publish.auto-fix.json
```

其中：

- `.md`
  这一轮的摘要说明，最适合先看
- `.setup.log`
  安装、同步、校验的完整日志
- `.smoke.json`
  smoke 闭环的 JSON 结果
- `.auto-fix.json`
  auto-fix 回环的 JSON 结果

## 怎么验收

最简验收顺序：

1. 打开本轮生成的 `.md` 快照
2. 看“安装/同步”“smoke 闭环”“auto-fix 回环”三项是否都是 `passed`
3. 如果某一项失败，再顺着快照里的文件路径看 `.log` 或 `.json`

通过标准很简单：

- 安装/同步通过
- smoke 闭环通过
- auto-fix 回环通过

如果这三项都过，说明当前发版至少满足：

- 包可安装
- 主链未回退
- 自动修复回环未回退

## 看完后怎么删

你看完之后，可以直接删本轮快照，不需要保留。

例如：

```bash
rm -f ./docs/paser_three/test-snapshots/20260410-103000-0.0.41-post-publish.*
```

如果还想把测试项目下的临时目录一起删掉：

```bash
rm -rf /Users/lizhenwei/workspace/test/test-ai-spec/ai-spec-cursor-test/.tmp/runtime-smoke-demo
rm -rf /Users/lizhenwei/workspace/test/test-ai-spec/ai-spec-cursor-test/.tmp/auto-fix-cli-demo
```

## 当前边界

这套脚本当前只负责：

- 发版后安装验证
- 主链 smoke 验证
- auto-fix 回环验证

它暂时不负责：

- 真正驱动 Cursor UI 自动点击
- 人工审批文案体验验收
- 长链路业务功能验收
- Hub 对接验收

也就是说，它是“发版后底座自检脚本”，不是完整产品验收脚本。
