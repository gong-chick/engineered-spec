# 团队同学 AI 规范开发最佳实践

> 适用对象：日常使用 AI 参与需求分析、开发实现、联调交付的团队同学  
> 阅读目标：先理解“为什么要做 AI 规范开发”，再掌握“在项目里应该怎么做”

## 1. 为什么团队要做 AI 规范开发

这篇文档不是先教你记命令，也不是先介绍平台实现细节。

它首先想解决一个认知问题：

**为什么团队用了 AI 以后，反而更需要规范开发。**

原因很简单。AI 确实能提升速度，但如果缺少规范，常见问题会同时放大：

- 需求还没说清楚，AI 已经开始写代码
- 代码能跑，但目录、路由、接口、样式都不符合项目约定
- 改动做完了，却没有留下可复盘的设计、任务和检查结果
- 一个同学能用，换一个同学或换一个会话就接不上

所以，团队要的不是“让 AI 写得更快”，而是：

- 让 AI 先按规则理解需求
- 让 AI 按项目约定落到正确位置
- 让过程有产物、有状态、有归档
- 让同一套方法能被整个团队复用

这就是 AI 规范开发的意义。

---

## 2. 先记住这句话

如果你第一次接触这套机制，先记住下面这句话：

> `OpenSpec` 负责管变更产物，`.agents` 负责管执行规则，`task-orchestrator` 负责管流程推进，`.ai-spec` 负责管运行状态。

只要先记住这句话，后面很多概念都会顺。

你可以把整套机制理解成：

- `OpenSpec` 负责把需求和交付结果沉淀成规范产物
- `.agents` 负责告诉 AI 应该遵守什么规则、怎么做事
- `task-orchestrator` 负责安排谁先做、谁后做、什么时候停下来等你确认
- `.ai-spec` 负责记录“当前做到哪里了”

---

## 3. 这套机制和“直接让 AI 写代码”有什么区别

很多同学第一次接触时，最容易误解的一点是：

“这是不是只是给 AI 外面又包了一层工具？”

不是。

真正的差别在于，团队不是让 AI 直接开始改代码，而是先进入一条规范驱动的交付链。

当前项目的最小主链是：

`requirement-analyst -> frontend-implementer -> code-guardian`

也就是：

1. 先把需求说清楚
2. 再进入实现
3. 最后再做检查和归档

这意味着一次任务不再只有“代码结果”，还会留下这些中间产物：

- `proposal.md`
- `specs/<domain>/spec.md`
- `design.md`
- `tasks.md`
- `checklist.md`
- `iterations.md`

这些产物的价值在于：

- 方便自己回看
- 方便同事接手
- 方便后续复盘
- 方便把一次做对的方法沉淀成长期规范

## 4. 第一次接触时，你应该先做什么

如果你是第一次接触项目，不建议一上来就去看所有源码。

推荐按下面顺序进入：

### 第一步：先建立整体认知

先知道这套机制的目标不是“替代研发同学”，而是“让 AI 参与研发时也遵守团队规范”。

### 第二步：先看最关键的 4 类目录

- `.agents/`
  团队规则、技能、角色、流程都在这里
- `openspec/`
  需求变更、规范产物、schema、配置都在这里
- `.ai-spec/`
  当前运行状态和历史事件都在这里
- `bin/` 与 `internal/`
  这是平台执行逻辑入口，知道位置即可，不需要第一次就全部读透

### 第三步：先跑通一个最小例子

对新人来说，最好的上手方式不是背概念，而是跑通一次最小示例，确认你知道：

- 一次需求怎么发起
- 中间产物会落到哪里
- 流程卡住时去哪里看
- 最终成功是什么状态

## 5. 一次真实需求是怎么走完的

你可以把一轮规范开发理解成下面 6 步。

### 5.1 发起需求

例如：

```bash
ai-spec-auto protocol-step --target . --user-input "新增一个商品 mock 页面" --json
```

这一步不是直接编码，而是先创建当前协议轮次。

### 5.2 主代理决定当前怎么做

`task-orchestrator` 会结合：

- 当前仓库事实
- `openspec/config.yaml`
- `.agents/flows/`
- `.agents/registry/`

决定：

- 当前任务由谁先接手
- 要产出哪些文件
- 是否需要门禁确认

### 5.3 需求专家先落结构化产物

在实现开始前，会优先沉淀：

- `proposal`
- `specs`
- `design`
- `tasks`

这一步的意义是先把“要做什么、为什么这么做、怎么验证”讲清楚。

### 5.4 实现专家再进入代码变更

只有到了实现阶段，才开始真正改业务代码、页面、接口、状态或样式。

### 5.5 守护专家做检查和补充

实现完成后，还会补：

- `checklist`
- `iterations`

也就是检查结论、问题记录、修正动作和残留风险。

### 5.6 归档并结束

所有结果确认无误后，再进入归档。

当前项目对归档做了一个很重要的优化：

- 如果你明确表达“归档”，系统会直接走本地 fast-path
- 不再为了一个确定性的归档动作再跑一轮重型 AI 编排

这也是团队做 AI 规范开发的一个典型原则：

> 该让 AI 判断的地方交给 AI；可以确定执行的动作尽量本地化、自动化。

## 6. 团队同学日常最常用的命令

日常使用先记住这 3 个命令就够了。

### 6.1 发起一轮新需求

```bash
ai-spec-auto protocol-step --target . --user-input "新增一个商品 mock 页面" --json
```

### 6.2 推进当前流程

```bash
ai-spec-auto protocol-advance --target . --json
```

### 6.3 补充需求或表达审批意见

```bash
ai-spec-auto protocol-update --target . --user-input "同意归档" --json
```

排障时再补两个命令：

```bash
ai-spec-auto runtime-state status --target .
ai-spec-auto validate-registry
```

## 7. 你在项目里最需要理解的几个文件

### 7.1 `openspec/changes/`

这里放本次需求的进行中产物和归档历史。

你要养成一个习惯：

**不要只看聊天记录，要看这里是否真的落盘。**

### 7.2 `openspec/specs/`

这里放长期有效的规范。

一个需求一旦归档完成，增量规范应当合并到这里，作为后续项目和后续会话的长期上下文。

### 7.3 `.ai-spec/current-run.json`

这是当前运行状态的事实来源。

如果你想知道：

- 当前做到哪一步
- 当前卡在哪个门禁
- 当前轮到哪个专家
- 最终是否成功闭环

先看这个文件。

### 7.4 `.agents/`

这里不是普通配置目录，而是团队协作规则中心。

你要重点理解 4 类内容：

- `rules/`：哪些事不能乱做
- `skills/`：这些事应该怎么做
- `roles/`：谁负责什么
- `flows/`：流程怎么串起来

## 8. `config.yaml`、schema、rules、skills 到底怎么分工

这是团队里最容易混淆的一块，建议固定按下面方式理解。

| 组件 | 解决的问题 | 适合放什么 |
| --- | --- | --- |
| `openspec/schemas/*` | 这套流程长什么样 | 产物结构、依赖关系、模板落点 |
| `openspec/config.yaml` | 这个项目怎么接入流程 | `schema`、项目事实、阶段边界 |
| `.agents/rules/*` | 哪些事不能乱做 | 目录约定、命名约定、禁止项、边界 |
| `.agents/skills/*` | 具体应该怎么做 | 建页面、建接口、写测试、提案增强 |

只记一个原则：

> `config.yaml` 负责定边界，`.agents` 负责给做法。

也就是说：

- 不要把完整执行手册塞进 `config.yaml`
- 不要把所有团队规则重复复制到 `config.yaml`
- `config.yaml` 只负责让 OpenSpec 在生成产物时不要脱离项目事实

## 9. 团队同学最应该遵守的最佳实践

这一章最重要，建议团队统一口径。

### 9.1 发起需求时，先说清范围

尽量把下面几件事说清楚：

- 目标是什么
- 不做什么
- 页面、路由、接口、状态分别涉及哪些范围
- 是否允许 mock
- 是否需要真实接口、真实数据、真实联调

说得越清楚，后面返工越少。

### 9.2 不要跳过中间产物

不要觉得 `proposal`、`design`、`tasks` 是“为了流程而流程”。

它们是为了避免：

- 需求没说清就开始写
- 实现方案和项目结构不对齐
- 验收标准只存在于口头沟通里

### 9.3 不要只盯着代码 diff

一次任务是否做对，不能只看代码改没改出来，还要看：

- `proposal` 是否把目标讲清楚
- `specs` 是否可测试
- `design` 是否和项目结构对齐
- `checklist` 是否真的检查过
- `iterations` 是否记录了本轮问题

### 9.4 归档前先确认是不是值得沉淀

归档不是“机械移动文件”，它代表的是：

- 这次增量规范值得进入长期上下文
- 后续类似需求可以复用这次结果

所以归档前要确认：

- 规范是否足够稳定
- 内容是否与真实实现一致
- 是否存在明显冲突或待补事项

### 9.5 能确定执行的动作，尽量不要继续堆模型

例如归档确认这类动作，本质是：

- 记录你的意图
- 更新运行状态
- 执行归档命令

这种动作就适合本地 fast-path，而不是再跑一轮复杂推理。

### 9.6 先看仓库事实，再谈“最佳实践”

团队的最佳实践不是脱离项目现实的模板，而是：

- 对齐现有目录
- 对齐现有路由
- 对齐现有接口封装
- 对齐现有样式与测试约定

离开仓库事实谈规范，最后一定会失真。

## 10. 常见误区

### 误区一：AI 规范开发就是让 AI 多写几份文档

不是。

真正目标是让需求、实现、检查、归档形成闭环，而不是单纯增加文档数量。

### 误区二：`config.yaml` 写得越多越专业

不是。

`config.yaml` 只需要负责桥接信息和阶段边界，写太多反而会增加噪音。

### 误区三：只要代码跑通，这次需求就算结束

不是。

如果没有规范产物、没有运行状态、没有归档结果，这次需求对团队来说就不可复用。

### 误区四：遇到问题先回忆聊天内容

不推荐。

优先去看：

- `.ai-spec/current-run.json`
- `openspec/changes/<change-id>/`
- `openspec/specs/`

这些才是事实来源。

## 11. 遇到问题时，先按这个顺序排查

### 11.1 看运行状态

先看：

```bash
ai-spec-auto runtime-state status --target .
```

或者直接看：

- `.ai-spec/current-run.json`

### 11.2 看产物是否真的落盘

重点检查：

- `openspec/changes/<change-id>/`
- `openspec/specs/`

### 11.3 看流程卡在哪个门禁

如果任务没推进，通常要先确认：

- 当前是不是卡在 `before-implementation`
- 当前是不是卡在 `before-archive`
- 当前是否已经给过明确的审批或补充输入

### 11.4 看规则和注册表是否一致

如果怀疑流程配置有问题，再执行：

```bash
ai-spec-auto validate-registry
```

## 13. 最后给团队的一句话

团队做 AI 规范开发，不是为了让流程变重，而是为了让 AI 参与研发以后，依然能保持：

- 需求清楚
- 实现可控
- 过程可追溯
- 结果可复用

只要团队形成这个共识，后面的工具、命令、规则和平台化能力，才会真正发挥价值。
