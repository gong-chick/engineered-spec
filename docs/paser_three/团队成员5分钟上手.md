# 团队成员 5 分钟上手

## 1. 先记住这句话

> `OpenSpec` 负责管变更产物，`.agents` 负责管专家执行，`task-orchestrator` 负责管流程，`.ai-spec` 负责管运行状态。

如果先把这句话记住，后面很多概念都会顺。

---

## 2. 这个项目到底在做什么

`ai-spec-auto` 不是让 AI 随便写代码，而是让 AI 按一条固定主链完成交付：

`requirement-analyst -> frontend-implementer -> code-guardian`

也就是：

- 先收敛需求
- 再实现代码
- 最后做交付检查

你可以把它理解成：这是一个“让 AI 按规范协作交付”的底座。

---

## 3. 团队成员最需要知道的 4 个目录

### `.agents/`

这是团队规范中心，优先看这里。

- `rules/` 是硬约束
- `skills/` 是具体做法
- `roles/` 是角色边界
- `flows/` 是专家交接顺序

### `openspec/`

这是需求和交付产物底座。

- `changes/` 放进行中的变更
- `specs/` 放长期有效的能力规范
- `config.yaml` 决定 OpenSpec 如何接入团队规范

### `.ai-spec/`

这是运行态事实来源。

- 看当前 run 做到哪一步
- 看当前 expert 是谁
- 看流程是否卡在门禁或审批

### `bin/` 与 `internal/`

这是平台实现入口，普通团队成员知道它们在这里就够了，不需要第一时间读透源码。

---

## 4. 日常最常用的 3 个命令

### 发起一轮新需求

```bash
ai-spec-auto protocol-step --target . --user-input "新增一个商品 mock 页面" --json
```

### 推进当前流程

```bash
ai-spec-auto protocol-advance --target . --json
```

### 补充需求或审批意见

```bash
ai-spec-auto protocol-update --target . --user-input "同意继续实现" --json
```

再补两个排障常用命令：

```bash
ai-spec-auto runtime-state status --target .
ai-spec-auto validate-registry
```

## 5. 你应该怎么理解 `config.yaml`

只记两个原则：

1. `config.yaml` 负责把 OpenSpec 和团队规范桥接起来
2. `config.yaml` 只定边界，不负责承载整套执行手册

也就是说：

- 阶段约束写进 `config.yaml`
- 具体规则看 `.agents/rules/`
- 具体操作看 `.agents/skills/`

---

## 6. 一次完整流程是怎么走的

最简理解如下：

1. 你发起需求，生成当前协议轮次
2. `task-orchestrator` 决定先给哪位专家
3. 专家在 `openspec/changes/<change-id>/` 或代码目录里落产物
4. `expert-executor` 把结果映射成运行态动作
5. Runner 自动交给下一位专家或结束

如果流程卡住，先看：

- `.ai-spec/current-run.json`
- `openspec/changes/<change-id>/`

## 8. 常见误解

- 误解一：这是一个“AI 自由写代码工具”
  实际上它先走协议轮次，再进入专家执行。
- 误解二：`config.yaml` 应该写得越细越好
  实际上它只应该描述桥接信息和阶段边界。
- 误解三：所有人都要先读实现源码
  实际上团队成员先建立流程认知更重要。
