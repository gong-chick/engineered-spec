# 协议与专家增强记录

## 背景

真实项目里的问题通常不是“会不会开始”，而是：

- 进行到一半补了需求，怎么优雅打断
- 归档前才发现实现和预期不一致，怎么修而不是重跑
- 归档后发现还有一个小问题，怎么补而不污染原归档
- 专家交接到底要不要问一下，问到什么程度才不烦

当前主链已经稳定，但更偏“线性推进”。这一轮增强的目标，是在不推翻现有链路的前提下，把中断、修正、暂停、恢复和轻确认补齐。

## 设计目标

本轮增强固定围绕这些目标展开：

- 自然语言优先，命令兜底
- 小修正在同一 `change_id` 内吸收
- 归档前修正和归档后补丁要有清晰分流
- 门禁要轻重分层，而不是每一跳都重审批
- 三专家骨架不变，但允许按场景插入可选专家
- 所有增强要留下可追溯的设计记录

## 当前实现

### 1. 命令层增强

在保留 `/spec-start`、`/spec-continue` 的基础上，新增：

- `/spec-update`
- `/spec-stop`
- `/spec-status`

当前语义：

- `/spec-update`：补充需求、修正方向、归档前修正说明
- `/spec-stop`：暂停当前 run，保留上下文与产物
- `/spec-status`：只读查看当前阶段、门禁和下一步建议

自然语言仍是一等入口。例如：

- “这个地方改一下，不要重新走全流程”
- “先暂停这个需求”
- “先别归档，这个实现不对，改成……”
- “给上个归档变更补一个修正”

### 2. 变更影响判断

`protocol-update` 已补上变更分类：

- `patch`
- `scope-delta`
- `re-scope`
- `archive-fix`
- `followup-patch`

运行态补充了这些字段：

- `change_impact`
- `reconcile_strategy`
- `artifacts_to_update`
- `reopen_reason`
- `parent_change_id`

这使主代理可以先判断：

- 当前是不是同一 change 内的小修正
- 是否需要回退到需求专家做 delta 收敛
- 是否已超范围，应该建议新建 change
- 是否属于归档前修正
- 是否属于已归档后的补丁修正

### 3. 增量修订规则

对 `patch / scope-delta / archive-fix`，默认都在同一 `change_id` 内增量修订。

当前约束：

- `proposal.md` 只补本轮修订、范围调整、新假设
- `tasks.md` 只增删改受影响任务
- `design.md / specs/` 只更新受影响章节
- `checklist.md / iterations.md` 保留历史，追加本轮记录

明确不做：

- 因一条补充输入整份重写已有产物
- 清空 checklist / iterations
- 无关章节一起重排

### 4. 归档阶段修正

#### 归档前修正

如果当前停在 `before-archive`，且用户表达：

- “先别归档”
- “这个实现不对”
- “改成……”

系统会识别为 `archive-fix`，默认：

- 不归档
- 清除当前归档待决
- 回退到最合适的专家修正
- 修完后重新进入 `code-guardian`
- 再次回到 `before-archive`

#### 已归档后补丁

如果变更已经归档成功，又要再修：

- 默认不改旧 archive
- 新开一个 `followup-patch` change
- 记录 `parent_change_id`
- 只围绕修正内容继续推进

这保证了归档可追溯，也避免把原归档目录改脏。

### 5. 混合门禁

当前门禁不再只有“审批 / 不审批”，而是分三层：

- `silent handoff`
- `confirm gate`
- `approval gate`

当前默认策略：

- `requirement-analyst -> frontend-implementer`：默认 `silent`
- `frontend-implementer -> code-guardian`：默认 `silent`
- `code-guardian -> archive-change`：维持 `approval`

同时，运行态已经补上对 `waiting-confirm` 的支持，用于“轻确认后继续”的场景。

### 6. 轻量主代理

主代理仍然保留，但职责缩成最小判断：

- 识别 `change_impact`
- 决定 `reconcile_strategy`
- 决定回退到哪位专家
- 记录本轮需要增量更新哪些产物
- 在已归档场景下决定是否新开补丁 change

明确不做：

- 每次补充输入都重跑完整规划
- 重建整套产物
- 把小修正升级成全量重排

### 7. 可选专家

在三专家基础骨架不变的前提下，当前协议已支持按场景激活可选专家：

- `design-collaborator`
- `api-contract-specialist`
- `unit-test-specialist`
- `verification-reviewer`
- `performance-auditor`

当前阶段：

- 已支持识别与激活
- 已在 guidance 中暴露
- 已为后续插入执行链预留转移关系

默认仍不全开，只有命中场景才激活。

## 与旧方案的差异

这轮不是回退到旧的“多问一次、多做一轮”，而是做了三件关键事：

1. 从线性推进升级成支持增量回退
2. 从只有审批门禁升级成轻重分层门禁
3. 从归档后只能重开大流程升级成支持 follow-up patch

同时保留这些稳定基线：

- 不改 `protocol-step / protocol-advance / protocol-update` 主职责
- 不推翻 `task-orchestrator -> requirement-analyst -> frontend-implementer -> code-guardian`
- 不削弱 `before-implementation / before-archive`

## 参考模式（归纳后落地）

本轮没有引入新的外部框架依赖，但吸收了几类成熟模式：

- checkpoint / interrupt / resume
- human-in-the-loop 的确认与恢复
- 长链路任务中的最小重做
- 自然语言优先、命令补充兜底

这些模式都经过归纳后落进当前架构，没有照搬外部产品实现。

## 当前风险与未决问题

- `waiting-confirm` 已有运行态支持，但自动触发策略仍可继续细化
- 可选专家已经能激活，但插入时机还可以继续打磨
- patch/followup-patch 当前以自然语言优先，是否独立命令化要看使用频率
- delta 产物虽然已支持局部更新语义，但后续仍可继续增强质量

## 后续增强候选

下一阶段优先考虑：

1. 继续增强三专家的 delta 能力
2. 让 `waiting-confirm` 更稳定地承接方案分歧场景
3. 为 patch/followup-patch 场景补更强的回归验证策略
4. 细化可选专家插入条件和回退路径
5. 如果 patch 成为高频操作，再评估是否补 `/spec-patch`
