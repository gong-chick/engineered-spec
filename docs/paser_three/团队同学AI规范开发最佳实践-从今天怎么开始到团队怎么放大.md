# 团队同学 AI 规范开发最佳实践：从今天怎么开始，到团队怎么放大

> 面向对象：团队同学、项目使用者、试点项目参与人  
> 阅读建议：先看完这篇，再决定要不要把当前项目接入规范开发闭环  
> 这篇文章的目标不是讲清所有技术细节，而是回答一个更现实的问题：**为什么我们现在值得升级做法，以及今天怎么开始。**

## 第一章：为什么我们现在要升级做法

这一章想解决的问题是：**AI 已经很好用了，为什么团队还要专门做“AI 规范开发”？**

这几年 AI Coding 的变化非常快。大家一开始接触它时，常见的使用方式是：

- 打开对话框
- 输一句需求
- 看 AI 给出代码
- 复制、修改、粘贴

这套方式在个人场景里很爽，但一旦进入团队协作，很快就会碰到一批更真实的问题：

- 需求还没说清楚，AI 已经开始改代码
- 代码能跑，但目录、路由、接口、样式都不符合项目约定
- 当前会话里做得不错，换一个同学、换一个会话就接不上
- 一次任务做完后只剩代码，没有 proposal、design、tasks、checklist、archive 这些资产
- Review 依然要花大量时间纠正“规范问题”，而不是讨论真正重要的设计取舍

这也是 [AI Coding时代，我们处在什么位置？](../AI Coding时代，我们处在什么位置？.md) 里最值得团队吸收的判断：

> AI Coding 的竞争，已经不是“谁更会写 prompt”，而是“谁能把 AI 稳定纳入软件工程体系”。

对团队来说，这个变化意味着一件事：

- 过去，拼的是谁更会用 AI
- 现在，拼的是谁能把 AI 变成团队可复用的工程能力

所以我们现在做 `ai-spec-auto`，并不是想再多造一个工具，而是想把下面这些能力沉淀下来：

- 团队规范能不能被 AI 稳定遵守
- 一次需求能不能先形成结构化产物，再进入实现
- 一次交付能不能留下可复盘、可归档、可复用的资产
- 不同 IDE、不同同学、不同会话，能不能尽量共享同一套做法

一句话总结这章：

> AI Coding 下半场，重点不是让 AI 更快写，而是让 AI 按团队方法稳定地做。

## 第二章：外部都怎么回答这个问题，我们吸收什么

这一章想解决的问题是：**现在外部主流路线都在做什么，我们应该吸收什么，而不是盲目照搬什么。**

为了避免把这篇文章写成产品盘点，我们只看 3 条最值得对标的方法路线。

### 1. `spec-kit` 路线：先把需求变成 spec，再进入实现

官方定位可以直接看 [GitHub Spec Kit](https://github.com/github/spec-kit)。它的核心思路很清晰：

- 先建立原则
- 再生成 spec
- 再生成 plan
- 再拆成 tasks
- 最后再 implement

它更偏解决什么问题：

- 把“聊天式需求”变成结构化产物
- 让实现建立在 spec / plan / tasks 之上，而不是直接开始写

它最值得我们吸收的地方是：

- 需求和变更要先沉淀成 artifact
- spec 不是附属材料，而是工程主线的一部分

它为什么不是我们当前团队的完整答案：

- 它强在 spec-driven，但不直接替团队定义 rules、skills、roles、flows 和运行态
- 对团队来说，它更像规范产物底座，而不是完整协同系统

### 2. `superpowers` 路线：把方法论和技能固化成可复用工作流

官方仓库见 [obra/superpowers](https://github.com/obra/superpowers)。它的核心表达非常鲜明：

- 不是让 agent 一上来就写代码
- 而是先做 brainstorming、plan、TDD、review、subagent execution
- 通过 skills 和初始指令把整套工作流固化下来

它更偏解决什么问题：

- 把“会做事的方法”变成 agent 默认执行的流程
- 把高频开发纪律，比如 TDD、review、plan，沉淀成可复用工作流

它最值得我们吸收的地方是：

- 高质量交付不能只靠临场发挥
- 做法本身应该被沉淀成 skills 和 workflow

它为什么不是我们当前团队的完整答案：

- 它很强，但更偏 agent 方法体系
- 对我们来说，还需要和团队项目规范、OpenSpec 产物、Hub 资产、运行态状态一起协同

### 3. `harness engineering` 路线：系统比模型更重要

官方站点见 [Harness Engineering](https://www.harness-engineering.xyz/)。它最核心的判断是：

> Agent = Model + Harness

这里的 harness 指的是：

- context engineering
- architectural constraints
- feedback loops
- tool & memory management

它更偏解决什么问题：

- 为什么同一个模型，在不同系统设计下效果会差很多
- 为什么真正决定可靠性的，不只是模型本身，而是约束、上下文和反馈回路

它最值得我们吸收的地方是：

- 不要迷信“换更强的模型就能解决问题”
- 团队真正该建设的是一整套约束、上下文和验证系统

它为什么不是我们当前团队的完整答案：

- 它更像一套上层工程思想
- 但团队落地还需要具体的规则文件、技能文件、流程模板、运行态和安装链路

### 4. 我们自己的路线：偏实操，不偏空谈

对比下来，我们当前项目的路线可以概括成一句话：

> 用 `OpenSpec` 管产物，用 `.agents` 管方法，用 `.ai-spec` 管状态，把团队做事的方法沉淀成底层能力。

具体来说：

- `OpenSpec` 管 proposal、specs、design、tasks、checklist、iterations
- `.agents` 管 rules、skills、roles、flows
- `.ai-spec` 管当前运行状态和事件事实

这也是我们和外部路线最大的区别：

- 不是只讨论理念
- 不是只讨论 agent 体验
- 而是更偏团队怎么今天就开始落地

### 竞品对比表

| 路线 | 更偏解决什么问题 | 最值得吸收什么 | 为什么不是我们的完整答案 |
| --- | --- | --- | --- |
| `spec-kit` | 让需求先形成 spec / plan / tasks | artifact-first、spec-driven | 缺团队级 rules / skills / runtime 增强层 |
| `superpowers` | 把方法论和技能变成工作流 | skills workflow、执行纪律、review/TDD 约束 | 更偏 agent workflow，不直接等于团队资产平台 |
| `harness engineering` | 让 agent 可靠运行 | context、constraints、feedback loops | 更偏工程思想，需要落到具体项目资产上 |
| 我们当前路线 | 团队今天怎么把 AI 纳入工程体系 | `OpenSpec + .agents + runtime + Hub/manifest` 一体化 | 入口和平台化仍在持续建设中 |

这一章最后只留一句话：

> 外部路线给了我们很多启发，但我们的任务不是复制理论，而是把它们变成团队今天就能用的实操链路。

## 第三章：今天怎么开始，后面怎么放大

这一章想解决的问题是：**如果团队不是想“听懂概念”，而是真想开始用，应该怎么走。**

我们建议把方案拆成两段来看。

### 方案一：人工快速使用

适用场景很简单：

- 团队还没完全平台化
- 但想先开始试
- 不想等“所有工具都准备好”才动

最小启动路径建议固定成下面这 5 步：

1. 先选 1 个试点项目  
   不要上来就全团队铺开，先选一个前端项目做试点。

2. 再选 1 个高频场景  
   比如“新页面开发”“mock 页面”“设计稿还原”“常规 CRUD 页面”。

3. 先补齐最关键的团队约定  
   重点不是一次补齐所有规则，而是先把最常影响交付质量的几类约定说清楚：
   - 目录落点
   - 路由约定
   - 接口封装
   - 样式规则
   - 测试要求

4. 先跑一条最小闭环  
   用当前已有的主链：
   `requirement-analyst -> frontend-implementer -> code-guardian -> before-archive -> archive-change`

5. 再根据试点结果补规则和技能  
   不是一开始就写一大堆规则，而是先看哪里最容易返工、最容易说不清，再针对性沉淀。

这条路径的价值在于：

- 启动快
- 风险可控
- 不需要先把整个平台做完

但它不是终局。  
因为如果长期只靠人工推进，团队还是会回到“靠少数人会讲规则”的状态。

### 方案二：工具加速使用

当团队开始验证这套方法有效后，下一步就不是“再多讲几次”，而是把这套方法做成可复制资产。

当前已经有一个很清晰的放大路径，就是 `skill-q-platform` 这条 Hub 方案链。

结合 [虾球 Hub README](/Users/lizhenwei/workspace/vueworkspace/bairong/skill-q-platform/README.md)、[scenario-manifest.ts](/Users/lizhenwei/workspace/vueworkspace/bairong/skill-q-platform/src/lib/scenario-manifest.ts)、[Scenario 页面](/Users/lizhenwei/workspace/vueworkspace/bairong/skill-q-platform/src/app/scenarios/[slug]/page.tsx) 和 [Manifest安装清单规范](../paser_two/Manifest安装清单规范.md)，这条链路已经可以讲得很明确：

1. **Hub 负责“选资产”**  
   在 Hub 里维护和组织：
   - Skill
   - Rule
   - Scenario Package
   - 推荐角色链

2. **场景方案页负责“组合资产”**  
   一个场景会组合出：
   - 哪些角色
   - 哪些技能
   - 哪些规则
   - 推荐入口角色

3. **manifest 负责“描述安装清单”**  
   `scenario-manifest.ts` 已经把最小结构收敛成：
   - `profile`
   - `ides`
   - `scenario_packages`
   - `roles`
   - `skills`
   - `rules`
   - `entry_role`

4. **`@engineered/ai-spec-auto` 负责“解析并安装”**  
   场景页已经能生成类似这样的命令：

   ```bash
   npx @engineered/ai-spec-auto@latest init . --profile vue --manifest <manifest-url>
   npx @engineered/ai-spec-auto@latest sync . --manifest <manifest-url>
   ```

5. **目标项目负责“承接并运行”**  
   安装后，目标项目承接的是：
   - `.agents`
   - `openspec`
   - `.ai-spec`
   - 对应的规则、技能、角色和流程

也就是说，这条链已经不再是纯概念，而是开始形成：

> Hub 选资产 -> manifest 描述安装清单 -> `@engineered/ai-spec-auto` 安装到目标项目 -> 目标项目按统一规则运行

### 人工方案 vs 工具方案

| 维度 | 人工快速使用 | 工具加速使用 |
| --- | --- | --- |
| 启动速度 | 快 | 中等 |
| 初始门槛 | 低 | 略高 |
| 可复制性 | 低到中 | 高 |
| 团队复用能力 | 一般 | 强 |
| 适合阶段 | 试点期 | 扩展期 / 平台期 |
| 核心价值 | 先开始 | 放大已经验证的方法 |

> **今天就能怎么做**
>
> 先不要等“平台全部做完”。  
> 最好的开始方式，是先选一个试点项目、一个高频场景，先把规则补齐、把闭环跑通。  
> 当你们已经知道“什么方法有效”之后，再用 Hub、manifest 和 `@engineered/ai-spec-auto` 把这套方法放大。

这一章最后只留一句总结：

> 人工方案解决“先开始”，工具方案解决“可复制、可扩展、可持续”。

## 第四章：推广不是装工具，而是看闭环、复用和提效

这一章想解决的问题是：**团队怎么判断，这套方法到底有没有真正产生价值。**

先说一个很容易踩坑的点：

不要只统计：

- 安装次数
- skill 数量
- 规则数量

这些只能说明“有资产”，不能说明“团队是否真的用起来了”。

更合理的统计方式，是看三层指标。

### 1. 接入层：有多少项目开始真的接入

这一层主要看：

- 接入项目数
- L1 / L2 / L3 占比
- Profile 覆盖情况
- 使用入口分布  
  例如：
  - 本地 IDE
  - Hub + manifest
  - 后续 works 插件

这一层解决的问题是：

> 这件事到底有没有开始从“少数人试”变成“团队有人在用”。

### 2. 过程层：有没有形成真实闭环

这一层建议重点看：

- `protocol-step` 发起数
- proposal 到 archive 的闭环完成率
- `before-implementation` / `before-archive` 门禁触发次数
- 活跃流程数
- 活跃角色数
- 活跃场景数

这一层解决的问题是：

> 团队不是只“装上了”，而是真的开始按流程做事了。

### 3. 结果层：有没有减少返工、增加复用

这一层更重要，建议重点看：

- Review 里的规范问题有没有下降
- 人工返工率有没有下降
- 新人第一次跑通完整闭环要多久
- 同一个规则 / 技能 / manifest / 场景包被复用了多少次

这一层解决的问题是：

> 这套方法到底有没有让团队变得更稳、更快、更容易协作。

### 当前项目现状能提供什么背书

基于当前仓库事实，可以先作为“推广基础盘”引用这些数字：

- 当前版本：`0.0.37`
- 规则文档：`21`
- 技能文档：`149`
- 已注册角色：`32`
- active 角色：`5`
- active flow：`1`

这些数字说明：

- 团队不是从零开始
- 已经有一套真实可用的资产骨架
- 当前重点不是继续堆数量，而是提高闭环率和复用率

### 团队同学更容易感知的变化

如果这套方法真的开始有效，团队最先感知到的变化通常不是“平台上线了”，而是：

- “这个文件该放哪”这类问题变少了
- Review 里纯规范问题变少了
- 相似任务越来越容易复用已有做法
- 一次任务结束后，别人更容易接着做
- 归档后的规范真的能在下一次任务里继续被用到

> **团队未来会看到什么变化**
>
> 以前：每次都重新解释项目约定。  
> 现在：规则、技能、流程开始沉淀成团队资产。  
> 以后：同类任务越来越少靠“个人经验”，越来越多靠“团队已沉淀的方法”。

这一章最后只留一句话：

> 推广不是为了多装一个工具，而是为了让团队真正更少返工、更多复用、交付更稳。

## 第五章：这件事最后会走到什么样子

这一章想解决的问题是：**团队现在值得投入这件事，是因为它未来会把我们带到哪里。**

这里不讲空泛愿景，只结合当前蓝图和已有材料，说一个团队能理解的路径。

### 短期：先把最小闭环跑稳

短期最重要的，不是继续补概念，而是把当前主链稳定下来：

- 稳定 `prd-to-delivery`
- 把 `expert-delivery` schema、主链流程、归档闭环继续打磨稳
- 用更多最小示例和试点项目验证可行性
- 把团队文档、测试说明、最佳实践材料补齐

对团队同学来说，短期最直接的变化会是：

- 流程更稳定
- 文档更清晰
- 常见场景更容易复用

### 中期：把入口和能力包做出来

中期重点不是再多写几篇文档，而是降低使用门槛。

结合 [AI规范驱动开发平台-综合蓝图](../paser_two/AI规范驱动开发平台-综合蓝图.md) 和 [流程模板矩阵与分阶段建设建议](../paser_two/流程模板矩阵与分阶段建设建议.md)，中期最值得期待的事情是：

- Hub 场景包和 manifest 安装链更成熟
- works 插件成为更自然的团队入口
- 流程模板不再只有 `prd-to-delivery` 一条
- 能按技术栈、能力域、流程包选择能力

对团队同学来说，这会意味着：

- 不再主要靠命令行理解能力
- 使用入口更直观
- 能按场景更自然地接入需要的能力

### 长期：形成完整的团队协同控制面

长期目标不是“再多一个页面”，而是形成一个完整体系：

- 平台资产层：rules / skills / roles / flows / scenario packages
- 入口层：IDE、Hub、works 插件
- 运行层：`task-orchestrator + OpenSpec + .ai-spec`
- 控制层：远程触发、审批放行、状态回传
- 治理层：CI/CD 校验、度量统计、效果复盘

结合 [OpenClaw团队协同控制面定位与价值说明](../paser_two/OpenClaw团队协同控制面定位与价值说明-03-31-13-03.md)，长期最值得期待的变化是：

- 任务不只从本地 IDE 发起
- 审批和状态回传不只在个人会话里完成
- 团队开始拥有真正的“协同控制面”

换成更通俗的话：

- 今天：少数人会用 AI
- 以后：团队按统一方法使用 AI
- 今天：主要靠本地 IDE
- 以后：IDE / Hub / 远程控制面共享同一套底座

这一章最后只留一句总结：

> 我们最终要做成的，不是一个更会写代码的 AI，而是一套让团队可以长期、稳定、可治理地使用 AI 的工作方式。

---

## 最后想发给团队的一段话

AI 规范开发的核心，不是多装一个工具，也不是多加几层流程。  
它真正解决的是：当团队开始大规模使用 AI 时，怎样还能保持需求清楚、实现可控、过程可追溯、结果可复用。

我们现在做的，不是让 AI 替代研发同学，而是让 AI 真正按团队的方法做事。  
只要这件事建立起来，后面的 Hub、works、OpenClaw、统计和平台化能力，才会真正有意义。
