# 团队同学最佳实践：从背景到落地

> 适用对象：团队同学、项目使用者、试点项目参与人  
> 阅读目标：理解为什么团队要做 AI 规范开发，以及在当前项目里应该如何落地、推广和持续使用

## 1. 背景

### 1.1 为什么团队同学要关心这件事

很多同学对 AI 的第一感受是“它能帮我写代码，效率会更高”。这个判断没错，但在团队协作里，只追求“更快写出来”通常还不够。

真实工作里更常见的问题是：

- 需求还没说清楚，AI 已经开始改代码
- 代码能跑，但目录、路由、接口、样式不符合项目约定
- 一个会话里做得还行，换一个会话、换一个同学就接不上
- 改完后只有代码，没有提案、设计、任务、检查、归档这些中间结果
- Review 阶段还是要花很多时间纠正规范问题

所以，团队做 AI 规范开发，不是为了让流程变重，而是为了让 AI 真正变成团队可复用、可交接、可治理的生产力。

### 1.2 当前项目在做什么

当前项目 `ai-spec-auto` 正在做的，不是单点工具增强，而是一套规范驱动开发底座。它把 AI 研发活动拆成几层：

- `OpenSpec` 负责变更产物
- `.agents` 负责规则、技能、角色、流程
- `task-orchestrator` 负责流程推进
- `.ai-spec` 负责运行状态

当前已经跑通的最小主链是：

`requirement-analyst -> frontend-implementer -> code-guardian -> before-archive -> archive-change`

也就是说，团队现在已经不是“只靠聊天让 AI 写代码”，而是开始让 AI 先产出规范，再进入实现，再做检查和归档。

### 1.3 这对团队同学的意义

对使用者来说，这套机制最直接的价值是：

- 不用每次都从头给 AI 讲项目规范
- 更容易把需求说清楚，而不是边写边猜
- 改动结果更容易落在正确目录和正确边界里
- 一次任务结束后，别人更容易接手和复盘
- 同样的方法可以被更多项目复用，而不是只靠个人经验

## 2. 竞品分析

这一章不是为了做“谁比谁强”的比较，而是帮助团队同学理解：外部常见路线解决了什么，我们自己的路线又解决了什么。

### 2.1 OpenSpec 路线

OpenSpec 的核心思路是：

- 先 proposal
- 再 apply
- 最后 archive

它擅长的是把需求变更做成结构化产物，并把规范长期沉淀到仓库里。

对团队同学来说，可以把它理解成：

> OpenSpec 负责把“这次要做什么”和“做完后留下什么”管起来。

它的优点是规范清晰、适合增量开发；不足是它本身不替团队定义具体规则、技能和专家协同方式。

### 2.2 Cursor / Copilot 这类路线

这类产品路线更强调：

- IDE 内即时协作体验
- 远程 Agent、后台执行
- PR、Review、Issue 工作流配合

它们很强的地方是“让 AI 更方便执行任务”，但不一定直接帮团队建立自己的规范资产。

对团队同学来说，可以把它理解成：

> 它们更像高效入口和执行器，但团队自己的规则、流程、资产还是要自己沉淀。

### 2.3 当前项目的差异点

`ai-spec-auto` 当前更关注的是：

- 如何把团队规则、技能、角色、流程沉淀下来
- 如何让 AI 在不同 IDE 下都尽量遵守同一套规范
- 如何让一次任务从提案、实现、检查到归档都有记录

所以，它的定位不是“替代某一个 AI IDE”，而是：

> 给团队建立一套不依赖单一入口、能长期复用的 AI 规范开发底层能力。

### 2.4 对团队同学的结论

如果你只是日常使用者，可以这样理解外部路线和当前项目的关系：

- `OpenSpec` 解决“规范产物怎么沉淀”
- `Cursor / Copilot` 解决“AI 怎么更顺手地协作”
- `ai-spec-auto` 解决“团队怎么把规范、流程、角色、状态组织成一个长期可用的系统”

## 3. 方案设计

从团队落地角度，可以把方案拆成两种。

### 3.1 方案一：人工方案

人工方案的做法是：

- 团队先写规范文档
- 同学在每次对话里手动告诉 AI 项目约定
- 用人工 Review 保证目录、接口、样式、测试不跑偏
- 用口头方式完成需求、设计和验收协作

优点：

- 上手快
- 成本低
- 适合前期试点和摸规则

不足：

- 很依赖个人经验
- 规则容易漏讲、漏用
- 同一个团队很难稳定复用
- 难以形成可统计、可归档、可观测的闭环

适合场景：

- 规则还在摸索阶段
- 只做很小范围试点
- 团队暂时还没准备好接入完整工具链

### 3.2 方案二：工具快速方案

这也是当前项目主推的方案。

它的做法是：

- 用 `OpenSpec` 管 proposal、specs、design、tasks、checklist、iterations
- 用 `.agents/rules/` 管团队硬约束
- 用 `.agents/skills/` 管高频做法
- 用 `.agents/roles/` 和 `.agents/flows/` 管专家协同链
- 用 `.ai-spec/current-run.json` 管当前运行状态

优点：

- 团队规则会被沉淀成仓库资产
- 需求、设计、实现、检查、归档形成闭环
- 更适合多人协作和跨项目复用
- 后续更容易接入 works 插件、Hub、远程入口和统计能力

不足：

- 需要先整理规则、技能和流程
- 使用习惯要重新建立
- 入口体验还需要进一步产品化

适合场景：

- 团队已经准备进入稳定推广
- 需要跨成员复用
- 希望把 AI 从“个人技巧”升级成“团队能力”

### 3.3 当前推荐方案

当前最适合团队同学的不是完全人工，也不是一上来追求全自动平台，而是：

> 人工先沉淀规则，工具承接执行闭环。

更具体一点：

1. 先用人工方式明确目录、路由、接口、样式、测试等关键约定
2. 再把这些约定沉淀到 `.agents/rules/` 和 `.agents/skills/`
3. 用 `OpenSpec + expert-delivery schema` 承接变更产物
4. 用 `task-orchestrator` 和 `.ai-spec` 把流程和状态跑起来

这条路径的好处是：

- 不会一开始就过度投入
- 能边试点边沉淀
- 一旦沉淀下来，就能更快复制到更多项目

### 3.4 对使用者最重要的理解

对团队同学来说，最需要记住的是：

- AI 规范开发不是多写几份文档
- 也不是把流程变复杂
- 而是把“口头经验”变成“团队资产”

只要这个前提认同了，后面的工具接入、流程推进、归档沉淀，都会更容易接受。

## 4. 推广统计

这一章不是让每个同学去做报表，而是帮助大家理解：如果团队真的在推广这套机制，应该看哪些信号判断它有没有价值。

### 4.1 当前项目已经具备哪些可推广资产

基于当前仓库现状，可以确认这些基础资产已经存在：

- 当前版本：`0.0.37`
- 支持 `Vue`、`React` 双 Profile
- 支持 `Cursor`、`Claude Code`、`OpenCode`、`Trae` 等多入口
- 已有 `21` 条规则文档
- 已有 `149` 份技能文档
- 已注册 `32` 个角色
- 当前有 `5` 个 active 角色
- 当前有 `1` 条 active 主流程
- 已有 `expert-delivery` 自定义 schema

对使用者来说，这意味着：

- 团队不是从零开始
- 已经有一批可以直接拿来试点和复用的资产
- 当前重点不是“先写更多规则”，而是“先让已有能力真正用起来”

### 4.2 推广时最值得关注的指标

如果团队后续要看推广效果，建议不要只看“装了多少次”，更应该看下面这些：

#### 使用指标

- 有多少真实需求通过规范流程发起
- 有多少任务完成了从 proposal 到 archive 的闭环
- 有多少任务卡在门禁，需要补充需求或确认

#### 质量指标

- `checklist` 的通过率怎么样
- `current-run.json` 的成功率怎么样
- 规范类 Review 意见有没有减少
- 人工返工率有没有下降

#### 团队指标

- 新人跑通第一次规范开发闭环要多久
- 同一个规则 / 技能 / 流程被复用了多少次
- 团队里有多少同学已经形成稳定使用习惯

### 4.3 对团队同学更有感知的变化

对日常使用者来说，推广是否有效，往往体现在这些更直接的感受上：

- “这个文件应该放哪”这类问题变少了
- Review 里纯规范问题变少了
- 同一个页面/接口/路由类任务更容易复用已有做法
- 一次任务做完后，别人更容易接着做
- 归档后的规范真的能在下次任务里继续被用到

### 4.4 当前可以参考的目标口径

按照现有项目需求文档里的预期，后续试点可以参考这组目标做前后对照：

- AI 代码接受行比：从约 `30%` 提升到约 `70%+`
- 代码采纳率：从约 `40%` 提升到约 `80%+`
- Code Review 耗时：减少约 `50%+`
- 新人上手时间：缩短约 `70%+`
- 规范沟通频次：减少约 `80%+`

这组数据更适合作为团队试点目标，而不是直接当成当前已验证结果。

### 4.5 对使用者的提醒

作为团队同学，不一定每个人都要管统计，但要知道一件事：

> 推广不是为了多装一个工具，而是为了让团队真的更少返工、更少反复解释、更多复用已有做法。

## 5. 后续规划

这一章的目的，是帮助团队同学理解：当前做到哪里了，下一步会给大家带来什么变化。

### 5.1 短期规划

短期重点是把当前主链继续打磨稳：

- 继续稳定 `requirement-analyst -> frontend-implementer -> code-guardian -> archive-change`
- 继续打磨 `config.yaml + schema + rules + skills` 的协同
- 用更多最小示例和试点项目验证闭环可靠性
- 继续完善团队文档、测试说明和最佳实践材料

对团队同学的直接影响是：

- 流程会更稳定
- 文档会更清晰
- 常见场景会更容易复用

### 5.2 中期规划

中期重点是降低使用门槛、让更多同学更自然地接入：

- works 插件成为更自然的入口
- 支持按技术栈、能力域、流程包选择能力
- 从当前 active 角色逐步扩到更多可启用专家
- 补齐更清晰的引导、推荐阅读、操作入口和统计能力

对团队同学的直接影响是：

- 不再主要依赖命令行理解能力
- 使用入口更直观
- 按需选择能力会更容易

### 5.3 长期规划

长期目标是把当前底座升级为更完整的平台能力：

- Hub 负责能力管理和分发
- works 插件负责入口和交互体验
- OpenClaw 承接远程触发和状态回传
- CI/CD 承接校验、回归和规范合规性
- 统计和治理能力承接推广效果和质量度量

对团队同学来说，长期变化会是：

- AI 规范开发不再只是少数核心同学掌握的能力
- 而会逐步变成团队标准工作方式的一部分

### 5.4 当前最合理的期待

当前阶段不适合期待：

- 一上来就完全自治
- 一上来就有完整平台页面
- 一上来就覆盖所有开发场景

更合理的期待是：

- 先把高频场景跑通
- 先把团队共识建立起来
- 先让一批同学稳定用起来

只要前面这三步做好，后面的平台化和规模化推广就会顺很多。

## 6. 最后给团队同学的一句话

团队做 AI 规范开发，不是为了让大家多学一套工具，而是为了让 AI 真正变成团队可依赖的协作能力。

如果把这篇文档只压缩成一句话，建议记成下面这句：

> 先把团队做事的方法沉淀下来，再让工具把这个方法放大。

这也是当前项目最值得坚持的最佳实践。
