## 验收明细

### P4.1 Event Gateway（事件网关）

- 实现 EventGateway 类：事件接收、校验、脱敏、NDJSON 持久化、查询、统计、导出
- 复用 AuditLog 的 NDJSON 持久化模式（fs.appendFileSync + loadFromFile + 坏行容错）
- 复用 redactSensitive / redactObject 自动红脱
- 新增 visual/index.js 和 run/index.js barrel 导出
- 31 个测试用例全部通过

### P4.2 Run Timeline（运行时间线）

- 实现 RunTimeline 类：事件聚合、阶段排序、分组、详情、摘要、序列化
- 固定 7 阶段排序：pre-task → pre-edit → post-edit → pre-test → post-test → repair → archive
- 阶段状态取最严重级别，失败原因取最后一条失败消息
- 21 个测试用例全部通过

### P4.3 Hook/Test/Repair 可视化

- 实现 HookDashboard 类：Hook 结果分析、Test 通过率、Repair 成功率、失败摘要
- Hook 按 hookId 分组，支持通过/失败/跳过统计和平均耗时
- Test 支持 passRate 计算和失败列表提取
- Repair 按 runId 聚合，支持平均修复次数和成功率
- 20 个测试用例全部通过

### P4.4 Agent 协作可视化

- 实现 AgentVisual 类：Agent 时间线、Handoff 视图、冲突视图、人工门禁视图
- 支持 agent.started/completed/escalated/blocked 状态追踪
- Handoff 提取 fromAgent/toAgent/reason
- 冲突类型：tool_denied、file_scope、max_iterations
- 21 个测试用例全部通过

### P4.5 质量指标与趋势分析

- 实现 MetricsEngine 类：5 项核心指标 + 趋势聚合
- 指标：taskSuccessRate、firstPassRate、repairSuccessRate、hookFailureRate、ruleHitRate
- 趋势：按 daily/weekly 窗口聚合，每个桶独立计算指标
- 23 个测试用例全部通过

### P4.6 风险与审计看板

- 实现 RiskBoard 类：策略拒绝、安全风险、越权尝试、审计操作、风险摘要、多维过滤
- 风险等级：low/medium/high/critical（基于安全事件 severity 和策略拒绝次数）
- 21 个测试用例全部通过

### P4.7 P4 集成回归与阶段验收

- 15 个集成测试用例：全链路验证各模块、模块间联动、接口稳定性、幂等性、隐私安全
- P1+P2+P3 回归验证通过
- barrel 导出验证通过
- NDJSON 事件流作为可视化数据源验证通过
- 19 个测试用例全部通过

### P4.8 可观测可靠性与风险语义加固

- **EventGateway 错误可见性增强**：
  - 新增 `_loadErrors` / `_writeErrors` 内部字段
  - `_loadFromFile()` 坏行不再静默跳过，记录到 `_loadErrors`（含 type/message/timestamp/lineNumber/line）
  - `_appendToFile()` 写入失败不再静默吞掉，记录到 `_writeErrors`
  - 新增 `getLoadErrors()` / `getWriteErrors()` 方法（返回数组副本）
  - 新增 `throwOnWriteError` 选项，为 true 时写入失败抛出异常
- **EventGateway clear() 增强**：
  - `clear()` 默认只清内存，保持向后兼容
  - `clear({ clearFile: true })` 同时清空 NDJSON 文件（写空字符串，不删文件）
  - 清空文件失败记录到 `_writeErrors`，throwOnWriteError=true 时抛出异常
- **RiskBoard 风险语义统一**：
  - 新增 `mapSeverityToRiskLevel(severity)` 函数：info→low, warn→medium, error→high, blocking→critical
  - 新增 `getRiskLevelRank(riskLevel)` 函数：low=0, medium=1, high=2, critical=3
  - `_computeTopRisks` 排序修复：使用 `riskLevel` + `getRiskLevelRank` 替代直接 `severity` + `RISK_LEVELS.indexOf`（解决 `info/warn` 不在 RISK_LEVELS 中导致 indexOf=-1 的排序不稳定问题）
  - topRisks 每项新增 `riskLevel` 字段
  - 新增函数均已导出，供测试和 P5 复用
- **测试新增 23 个用例**（event-gateway 9 + risk-board 9 + integration TC16 的 5 个子用例）
- 全部 179 个 P4 测试用例通过，`npm run verify:p4` 全量通过

## 验收口径说明

1. P4 全部模块位于 `src/visual/` 目录，纯本地数据处理，不依赖外部 API
2. 复用 P3 AuditLog 的 NDJSON 持久化模式和红脱逻辑
3. 禁止上传完整源码、禁止展示 Prompt 原文
4. 不替代 Sentry/DevOps、不做业务 BI
5. npm run verify:p4 已真实执行并通过后，才允许进入 P5

## P4.8 加固说明

- **P4.8 之前**：EventGateway 有基础 NDJSON 能力，但坏行和写入失败静默跳过，错误不可见
- **P4.8 之后**：EventGateway 支持 loadErrors、writeErrors、throwOnWriteError、clearFile，错误完全可追溯
- **P4.8 之前**：RiskBoard 的 severity 与 riskLevel 语义不完全一致，topRisks 排序可能不稳定（info/warn 不在 RISK_LEVELS 中导致 indexOf=-1）
- **P4.8 之后**：统一 severity→riskLevel 映射，topRisks 排序使用 getRiskLevelRank，排序稳定可靠
- 当前 P4 仍不是完整生产级 APM / Sentry / BI 平台
- 当前满足 P5 资产质量评分和反馈闭环的数据可靠性要求
