# AI Coding 时代，我们处在什么位置？

> 本文档用于团队内部分享：厘清 **AI 辅助编程的演进阶段**、**个人与团队应强化的能力**，以及 **ai-spec-auto**（npm：`@engineered/ai-spec-auto`）在其中的定位。  
> 可与 [《项目需求说明》](http://git.100credit.cn/zhenwei.li/ai-spec-auto/-/blob/main/docs/项目需求说明.md)、[《规范驱动开发团队内部培训手册》](http://git.100credit.cn/zhenwei.li/ai-spec-auto/-/blob/main/docs/规范驱动开发团队内部培训手册.md) 配合阅读（仓库 [ai-spec-auto](http://git.100credit.cn/zhenwei.li/ai-spec-auto)，默认分支 `main`）。

## 1. 时代背景：不只是在「下指令」

过去很长一段时间里，大家谈 AI 写代码，默认画面是：**打开对话框 → 输入一段自然语言 → 复制粘贴结果**。这可以称为 **「指令编码」的初级阶段**：产出高度依赖单次提示的质量，上下文散落在会话里，难以复用、难以审计，换人即失效。

今天，行业正在明显 **进化**：

- **工具侧**：从「单点补全」到 **Agent、多文件编辑、终端与仓库级操作**；从「一个 IDE」到 **多工具链并存**（Cursor、Claude Code、OpenCode、Trae 等）。
- **工程侧**：从「个人 prompt 技巧」到 **项目内可版本化的 Rules、Skills、工作流与 spec**——**可复用、可溯源、可升级**。
- **角色侧**：高频机械编码被加速摊平，**判断、拆解、架构、验证与领域理解** 的权重上升。

也就是说：**AI Coding 已不是「会不会写几句漂亮话」的比拼，而是「能否把智能体稳定地纳入软件工程体系」的比拼。**

---

## 2. 与「Vibe Coding」同频：快感、陷阱与价值锚点

Andrej Karpathy 提出的 **Vibe Coding** 描述了一种真实状态：凭感觉描述需求、运行、复制粘贴，「大部分时候能跑」。这与「指令编码」叠加后，前端尤其容易出现 **「从搬砖到审砖」** 的转变——UI 一眼可见，AI 生成 + 目检闭环很快，但这也意味着 **草率合并** 的风险：可访问性、错误边界、性能陷阱、与设计系统冲突的样式等，常被一带而过。

一篇面向前端同行的梳理（**《Vibe Coding 时代，前端工程师的生存指南》**）把其中的 **价值重估** 写得很直白，核心取向与本文一致，推荐阅读原文：  
[Vibe Coding 时代，前端工程师的生存指南](https://mp.weixin.qq.com/s/VxT5DXTdFmd2AV6ifHfGZQ)

文中强调的几点，可映射到我们自己的行动：

| 外部叙述（文章脉络） | 对我们的启示 |
|----------------------|--------------|
| 角色从「演奏者」转向 **「指挥家」** | 你定义边界、拆解问题、验收结果；AI 执行与迭代，但不替你承担工程决策。 |
| **问题拆解** 比「会不会写 prompt」更能拉开差距 | 同一套模型，拆清楚边界、状态、失败路径，产出可差一个量级。 |
| **架构、复杂调试、领域深度** 更难被摊平 | 这些是应持续投资的「内在能力」，而不是被 AI 替代后无处安放的老本行。 |
| 「别止步于 prompt」 | 更要会 **验证输出、引导迭代、判断 AI 边界**——本质是 **技术判断力**，不是话术。 |

**冷静结论（与原文精神一致，不贩卖焦虑）**：前端不会消失，但「只会堆页面、不理解系统」的空间在收窄；**省下来的时间，应明确投向更高层的问题**——否则工具越强，个人越像被动接稿的「粘贴工」。

## 3. AI Coding 的三阶叙事：我们在爬哪一层？

可以用一列简化的「阶梯」描述行业与团队的常见状态（非官方标准，便于对齐语言）：

### 第一阶：指令驱动（Ad-hoc Prompting）

- 特征：依赖即时对话，规范在脑子里或群里；产出风格随人而异。
- 风险：不可继承、难 Code Review、难复盘「当初为什么这样写」。

### 第二阶：约束驱动（Rules / 项目上下文）

- 特征：把 **编码边界** 写进仓库（如各 IDE 的 rules、AGENTS.md 等），AI **默认遵守**。
- 收益：一致性明显提升；仍可能「知道规矩但不知道步骤」。

### 第三阶：规范 + 流程 + 可溯源 spec（Spec-driven / 工程闭环）

- 特征：除 **声明式规范（Rules）** 外，还有 **过程式技能（Skills）**；需求与基线以 **结构化变更与归档（如 OpenSpec）** 留在仓库里，形成 **提案 → 实施 → 归档** 的链路。
- 收益：**个人能力团队化**、**变更有迹可循**、多 IDE **单源** 同步，减少「工具换一家就要从头教」的摩擦。

**我们处在什么位置？**  
- 若仍停留在第一阶：工具再强，团队层面仍是 **个人英雄主义 + 隐性知识**。  
- 若已上第二阶：已迈出关键一步，但容易停在「只约束、不流程」。  
- **团队主推方向** 应对齐第三阶：**让 AI 在「有边界的轨道」上跑「可重复的流程」**，并把关键决策写进 **可 diff 的资产**。

## 4. 提升「内在能力」：与工具进化配套的四件事

工具进化不会自动带来职级进化；以下能力与 **ai-spec-auto 所倡导的规范驱动** 互为表里：

1. **架构与边界判断**  
   目录、状态归属、数据流、与后端/产品的契约——AI 可以生成实现，但 **「为什么这样切」** 需要人定调；这正是 Rules 要固化、OpenSpec 要留痕的部分。

2. **问题拆解与验收标准**  
   把需求拆成可验证子任务（含失败路径、性能与无障碍），对应 Skills 中的执行范式；没有拆解，仅有「帮我做个 xx 页」，仍停留在第一阶。

3. **验证 AI 输出的心智模型**  
   安全、性能、可维护性、与设计系统一致性——与文中举例的「useEffect + 不稳定依赖」类问题相同：**判断力来自对运行机制的理解**，不是更多形容词。

4. **领域纵深（可选但高回报）**  
   可视化、跨端、构建与微前端、体验与无障碍等 **长反馈链** 方向，仍是建立差异度的常见路径。

## 5. ai-spec-auto 在版图中的位置：从「会写指令」到「资产化规范」

**ai-spec-auto** 的定位是：**AI Coding 团队的规范驱动开发 CLI（当前版本面向前端，内置 Vue 3 / React Profile）**；通过 npm 包 **`@engineered/ai-spec-auto`**（`npx @engineered/ai-spec-auto@latest init` 等）安装到业务仓库。目标一句话可概括为：

> **把「提示词能力」改造成「项目内资产」**——从依赖个人记忆与口头约定，转为 **结构化规范文件** 让 AI 自动遵循，并与 **可溯源的需求流程** 一体联动（L3）。

与上文「三阶叙事」的对应关系：

| 能力层次 | 在 ai-spec-auto 中的落点 |
|----------|------------------|
| 声明式约束 | `.agents/rules/`：编码、API、组件/路由/状态/样式、测试等 |
| 过程式流程 | `.agents/skills/`：提案分析、任务执行、测试与设计稿分析等 |
| 单源多 IDE | `.cursor` / `.claude` / `.opencode` / `.trae` 等 **链接同一 `.agents`** |
| 外部上下文（L2+） | MCP 等模板：接口、设计、浏览器等，减少 AI 空猜 |
| 需求闭环与溯源（L3） | `openspec/` + `config.yaml` 与 `.agents` **桥接**：提案 → 实施 → 归档 |

**愿景（摘自需求说明）**：**让每一个使用 AI 编码的团队成员，无论提示词能力高低，都能产出符合团队标准的高质量代码。**  
这不是贬低 prompt，而是承认现实：**团队交付不能押宝在少数「会写词的人」身上**；应把共性抽成 **可安装、可更新、可检查** 的规范与流程（`init` / `update` / `check`）。

## 6. 结论：我们在时代里的「位置」与下一步

- **时代位置**：AI Coding 已从 **初级阶段（纯指令）** 走向 **工程化与 spec 化**；前端尤其暴露在 **Vibe Coding** 的「快」与「糙」之间，**价值锚点上移** 是普遍趋势。  
- **个人位置**：从 **与 AI 比打字速度**，转向 **指挥、拆解、架构、验证与领域深度**——与 [上文所引用的生存指南](https://mp.weixin.qq.com/s/VxT5DXTdFmd2AV6ifHfGZQ) 中的「指挥家」隐喻一致。  
- **团队位置**：通过 **ai-spec-auto** 的 **L1 → L2 → L3 渐进安装**，把 **Rules + Skills +（可选）OpenSpec** 变成 **共享资产**，减少对人、对单次会话的依赖，**对齐第三阶：规范驱动 + 可溯源**。

**最后一问（与参考文章同构）**：当 AI 与 **ai-spec-auto** 所交付的规范、技能与流程帮你扛住大量重复劳动时，你和团队 **把省下的时间投在什么问题上了？**  
这个问题的答案，决定了在 AI Coding 时代 **我们究竟处在什么位置**——是停留在「更快的粘贴」，还是 **更强的工程与产品判断力**。
