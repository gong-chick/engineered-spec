# visual ↔ auto 端到端手动验证手册

> 本手册基于当前仓库实际状态校准（容器名、端口、Prisma 字段、CLI 子命令），可**直接复制执行**。
> 执行前请先完成 §0 的「必改变量」。

---

## 0. 一次性准备

### 0.1 必改变量（执行前请核对/替换）

以下三个变量是整份文档里唯一需要你手动确认的地方：

```bash
# visual 仓库路径（默认值正确，可不改）
export VISUAL_PATH="/Users/lizhenwei/workspace/vueworkspace/bairong/engineered-spec-visual"

# auto 仓库路径（默认值正确，可不改）
export AUTO_PATH="/Users/lizhenwei/workspace/vueworkspace/bairong/br-ai-spec"

# visual 服务地址（默认 18780，可不改）
export VISUAL_URL="http://localhost:18780"

# ⚠️ 旧项目场景 B 必改：你本地真实存在的旧项目（已装过 ai-spec-auto 并跑过需求）
export OLD_PROJECT="/path/to/your/old-project-with-ai-spec"
```

### 0.2 启动 visual 服务（端口 18780）

```bash
cd "$VISUAL_PATH"
./start-with-db.sh        # 同时拉起 MariaDB（13306）与 Next.js（18780）
```

**验证服务起来：**
```bash
curl -s "$VISUAL_URL/api/health" | head -c 200
# 期望：{"status":"ok",...}
```

### 0.3 获取数据库容器名（一次性确认）

```bash
DB_CONTAINER=$(docker ps --format '{{.Names}}' | grep -E 'visual.*db|mariadb' | head -1)
echo "DB_CONTAINER=$DB_CONTAINER"
# 期望输出类似：DB_CONTAINER=engineered-spec-visual-db-1
```

> 真实账号密码（已从 `docker-compose-db-only.yml` 确认）：
> - 账号：`root`
> - 密码：`root_password_123`
> - 数据库：`br_ai_spec_visual`

## 场景 A：全新项目（干净，从零接入）

### A.1 创建空项目并安装 auto

```bash
mkdir -p /tmp/visual-newproj && cd /tmp/visual-newproj
npm init -y >/dev/null
npm install "$AUTO_PATH" >/dev/null 2>&1 \
  || npm install --install-links "$AUTO_PATH"   # 兜底：遇到 EUNSUPPORTEDPROTOCOL 时走 install-links
```

**断言 1（零侵入）—— 安装完没有任何 visual 文件：**

```bash
[ -d .ai-spec ] && echo "❌ .ai-spec 存在（错误）" || echo "✅ .ai-spec 不存在（正确）"
grep -ic visual package.json
# 期望：第 1 条 "✅ .ai-spec 不存在"
# 期望：第 2 条 "0"
```

### A.2 跑一次 protocol-status，确认无 visual 干扰

```bash
npx ai-spec-auto protocol-status
# 期望：状态正常显示（例如 No active run），不出现 visual 相关报错
```

### A.3 显式开通 visual 桥接（opt-in）

```bash
npx ai-spec-auto visual init --yes
# 输出末行会打印 connect_token

# 把 server_url / workspace_id 改成我们要用的值
node -e "
const fs=require('fs');
const f='.ai-spec/visual-bridge.json';
const j=JSON.parse(fs.readFileSync(f,'utf-8'));
j.server_url=process.env.VISUAL_URL;
j.workspace_id='newproj-demo';
fs.writeFileSync(f, JSON.stringify(j,null,2)+'\n');
console.log(j);
"
```

**断言 2（opt-in 边界）—— 此时仅多出 1 个 json 文件 + 1 个空 inbox 目录：**
```bash
find .ai-spec -maxdepth 2 -mindepth 1
# 期望：只看到 .ai-spec/visual-bridge.json 与 .ai-spec/inbox（空）
```

### A.4 在 visual 中建对应的 Workspace

走 SQL 直连最快（字段与 Prisma schema 一致，已核对）：

```bash
docker exec -i "$DB_CONTAINER" \
  mariadb -uroot -proot_password_123 br_ai_spec_visual -e \
  "INSERT IGNORE INTO Workspace (id, slug, name, status, rootPath, createdAt, updatedAt)
   VALUES ('newproj-demo','newproj-demo','New Demo','active','/tmp/visual-newproj', NOW(), NOW());"

# 验证写入成功
docker exec -i "$DB_CONTAINER" \
  mariadb -uroot -proot_password_123 br_ai_spec_visual -e \
  "SELECT id, slug, rootPath FROM Workspace WHERE id='newproj-demo';"
```

> `rootPath=/tmp/visual-newproj` 同时开启了**文件兜底通道**——visual 写 outbox 时会同时落到 `/tmp/visual-newproj/.ai-spec/inbox/control-*.json`。

### A.5 跑一次桥接连通自检

```bash
cd /tmp/visual-newproj
npx ai-spec-auto visual test
# 期望输出三行（顺序可能略有不同）：
# [visual] ping http://localhost:18780 → ok (status 200)
# [visual] pull pending → 0 written (transport=http)
# [visual] push probe receipt → ok
```

打开浏览器访问 `http://localhost:18780/overview`，应能在工作区健康卡里看到 `newproj-demo`。

### A.6 触发一次真实事件上行

```bash
cd /tmp/visual-newproj
npx ai-spec-auto protocol-step --user-input "visual e2e 测试-A6" 2>&1 | tail -20
```

**前往 visual UI 验证：**
- `http://localhost:18780/workspaces/newproj-demo/board` → 应看到一张新 run 卡片出现在泳道
- `http://localhost:18780/workspaces/newproj-demo` → run 详情的 Trace Stream 应有 `run.started` / `run.state_changed` 事件

### A.7 控制面闭环：浏览器审批 → auto 应用 → 回执

**推荐（UI 操作）：**
- 在 Kanban 上把 run 卡片拖到下一个泳道 → 触发 approve
- 或在 RunDetail 右下 Gate Panel 点击「批准」

**立即观察文件兜底通道（rootPath 已配好，visual 会同时落盘）：**

```bash
ls /tmp/visual-newproj/.ai-spec/inbox/
# 期望看到 control-<ts>-<id>.json

cat /tmp/visual-newproj/.ai-spec/inbox/control-*.json 2>/dev/null | head -40
# 期望看到 outbox_id / command / payload / signature 等字段
```

**让 auto 消费这条控制指令：**

```bash
cd /tmp/visual-newproj
npx ai-spec-auto protocol-status       # 任一 protocol-* 命令都会触发 inbox-consumer
ls /tmp/visual-newproj/.ai-spec/inbox/ # control-*.json 应消失
ls /tmp/visual-newproj/.ai-spec/inbox/.applied/ \
   /tmp/visual-newproj/.ai-spec/inbox/.processed/ 2>/dev/null
# 期望：在 .applied 或 .processed 目录里看到原文件（带时间戳前缀）
```

**回到 visual UI：**
- Kanban 上该 run 卡的 outbox 标记应从 `pending → delivered → applied`
- RunDetail 的 Gate Panel 历史区域显示「已应用 ✓」

### A.8 停服降级验证

```bash
# 停 visual（只关 18780 端口，不动数据库）
lsof -ti:18780 | while read pid; do kill -9 "$pid"; done
sleep 2
curl -s "$VISUAL_URL/api/health" >/dev/null 2>&1 \
  && echo "❌ visual 仍在运行" \
  || echo "✅ visual 已停"

# 在 auto 项目里继续推进
cd /tmp/visual-newproj
time npx ai-spec-auto protocol-status 2>&1 | tail -5
# 期望：命令在 1-2 秒内完成；stderr 可能有 push failed 警告，但 exit code = 0
```

### A.9 卸载 visual

```bash
cd /tmp/visual-newproj
npx ai-spec-auto visual disable
# 或彻底清理：
# rm -rf .ai-spec/visual-bridge.json .ai-spec/inbox

ls -la .ai-spec/
# 期望：除了 inbox 历史归档（如有），再无 visual 行为
```

## 场景 B：旧项目（已装 auto 且跑过需求）

> 关键诉求：**接入前后 `.ai-spec` 状态文件完全不变**，只有显式 opt-in 后才出现 visual 文件。

### B.1 选定旧项目并做「接入前快照」

```bash
# ⚠️ 确认 OLD_PROJECT 已在 §0.1 中设置为你的真实路径
[ -d "$OLD_PROJECT/.ai-spec" ] || { echo "❌ OLD_PROJECT 无效或无 .ai-spec"; }

cd "$OLD_PROJECT"
mkdir -p /tmp/old-pre
rm -rf /tmp/old-pre/.ai-spec.before
cp -r .ai-spec /tmp/old-pre/.ai-spec.before
ls /tmp/old-pre/.ai-spec.before
```

### B.2 升级 ai-spec-auto 到当前最新

```bash
cd "$OLD_PROJECT"
npm install "$AUTO_PATH" >/dev/null 2>&1 \
  || npm install --install-links "$AUTO_PATH"

npx ai-spec-auto --version 2>/dev/null || echo "(当前 CLI 不支持 --version，可忽略)"
ls node_modules/@engineered/ai-spec-auto/internal/visual-hooks/ 2>/dev/null \
  || ls node_modules/ai-spec-auto/internal/visual-hooks/
# 期望：能看到 inbox-consumer.js / control-puller.js / receipt-pusher.js / index.js / push-client.js / config-loader.js
```

**断言（无破坏性）—— 升级后 `.ai-spec` 文件完全不变：**

```bash
diff -rq /tmp/old-pre/.ai-spec.before "$OLD_PROJECT/.ai-spec"
# 期望：无任何输出（= 完全相同）
```

### B.3 跑一次 protocol-status，确认旧项目状态可用

```bash
cd "$OLD_PROJECT"
npx ai-spec-auto protocol-status
# 期望：旧 run 状态正常输出
```

### B.4 把历史数据回灌到 visual（Collector）

```bash
cd "$VISUAL_PATH"
WORKSPACE_ID="oldproj-demo"

# 先在 visual 里建好 workspace
docker exec -i "$DB_CONTAINER" \
  mariadb -uroot -proot_password_123 br_ai_spec_visual -e \
  "INSERT IGNORE INTO Workspace (id, slug, name, status, rootPath, createdAt, updatedAt)
   VALUES ('${WORKSPACE_ID}','${WORKSPACE_ID}','Old Demo','active','${OLD_PROJECT}', NOW(), NOW());"

# 回灌
npm run collector -- \
  --workspace-id "$WORKSPACE_ID" \
  --project "$OLD_PROJECT" \
  --server "$VISUAL_URL"
# 期望：打印 workspace_id / agent_id / run_id 以及各 *_files / raw_events 计数
```

打开 `http://localhost:18780/workspaces/${WORKSPACE_ID}`：
- 看到旧 run 列表
- `/board` 上旧 run 出现在对应泳道
- `/runs/<runId>` Evidence Wall 上能看到旧 proposal / design / tasks 痕迹

### B.5 开通 visual 桥接（opt-in）

```bash
cd "$OLD_PROJECT"
npx ai-spec-auto visual init --yes
node -e "
const fs=require('fs');
const f='.ai-spec/visual-bridge.json';
const j=JSON.parse(fs.readFileSync(f,'utf-8'));
j.server_url=process.env.VISUAL_URL;
j.workspace_id=process.env.WORKSPACE_ID || 'oldproj-demo';
fs.writeFileSync(f, JSON.stringify(j,null,2)+'\n');
console.log(j);
"
```

**断言（开通边界）—— 现在比快照只多 1 个 json + 1 个空 inbox：**

```bash
diff -rq /tmp/old-pre/.ai-spec.before "$OLD_PROJECT/.ai-spec" \
  | grep -v "Only in $OLD_PROJECT/.ai-spec: visual-bridge.json" \
  | grep -v "Only in $OLD_PROJECT/.ai-spec: inbox"
# 期望：无任何额外输出
```

### B.6 验证桥接

```bash
cd "$OLD_PROJECT"
npx ai-spec-auto visual test
npx ai-spec-auto visual status
# 期望：ping ok / pull 0 written / push ok；status 显示 enabled=true
```

### B.7 触发增量事件并观察实时上行

```bash
cd "$OLD_PROJECT"
npx ai-spec-auto protocol-step --user-input "旧项目-B7 实时增量验证" 2>&1 | tail -10
```

刷新 `http://localhost:18780/workspaces/${WORKSPACE_ID}` → 新 run 出现在 Active；`/overview` 健康卡数字也会变。

### B.8 控制面闭环

同 A.7：
1. 在 UI 上 approve（拖卡片或点 Gate Panel）
2. `ls $OLD_PROJECT/.ai-spec/inbox/` 应看到 control-*.json
3. `npx ai-spec-auto protocol-status` 触发消费
4. 文件移到 `.applied/` 或 `.processed/`
5. UI outbox 状态变成 `applied`

### B.9 停服降级（关键合规）

```bash
lsof -ti:18780 | while read pid; do kill -9 "$pid"; done
cd "$OLD_PROJECT"

# 旧业务命令 —— 必须不受影响
time npx ai-spec-auto protocol-status 2>&1 | tail -5
# 期望：≤ 2 秒完成，exit code 0
```

### B.10 关闭 visual 集成（回到接入前状态）

```bash
cd "$OLD_PROJECT"
npx ai-spec-auto visual disable
# 或彻底移除：
rm -f .ai-spec/visual-bridge.json
rm -rf .ai-spec/inbox

diff -rq /tmp/old-pre/.ai-spec.before "$OLD_PROJECT/.ai-spec"
# 期望：无任何输出（完全恢复到接入前）
```

## 常见问题 / 排错

- **`DB_CONTAINER` 为空**：先 `docker ps`，确认 mariadb 容器正在运行；或把 §0.3 的 grep 规则换成你实际的容器名关键词。
- **`npm install "$AUTO_PATH"` 报 EUNSUPPORTEDPROTOCOL / workspace 错误**：改用 `npm install --install-links "$AUTO_PATH"`，或 `cd "$AUTO_PATH" && npm pack` 后装 `.tgz`。
- **看不到 control-*.json 落盘**：确认 visual 数据库里 `Workspace.rootPath` 已设置为目标项目绝对路径（A.4 / B.4 的 SQL）。
- **visual UI 上 outbox 一直 pending**：在 auto 项目里跑任一 `protocol-*` 命令触发消费；HTTP-pull 默认 50ms 超时，慢网会回退到 file 通道。
- **`visual test` 报 push failed 但 ping ok**：检查 `connect_token` 与 visual 端 `AI_SPEC_BRIDGE_SECRET` / `REALTIME_CONNECT_SECRET` 是否一致；不一致会被 loose token 放行入库，但 Outbox 验签会拒绝。
- **停服后命令变慢（>2s）**：默认超时 pull=50ms / push=200ms，都是 best-effort；若异常慢请检查 `bin/visual-bridge.js` 是否被改动。
