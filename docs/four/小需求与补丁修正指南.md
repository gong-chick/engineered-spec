# 小需求与补丁修正指南

这份指南只回答一个问题：小需求到底该怎么走，才能既不太重，也不失去留痕。

## 先记一条总原则

- 大需求继续走 `prd-to-delivery`
- 小需求先判断上下文，再决定是复用已有 change、走轻量快修，还是升级回完整主流程

补充一条当前默认语义：

- 完整主流程默认是 `auto（自动） + none（无阻塞审核）`
- `manual（手动）` 只表示手动指定 `flow（流程模板）`，不表示“每一步都审核”

## 决策表

| 场景 | 怎么说 | 系统默认选择 | 留痕位置 |
| --- | --- | --- | --- |
| 新功能、跨模块改动、要评审 | `/spec-start ...` 或自然语言描述 | `full-change` -> `prd-to-delivery`，默认按 `none（无阻塞审核）` 自动推进；需要人工审核时再显式切到 `main-flow-blocking（主流程阻塞审核）` | `openspec/changes/<change-id>/` |
| 当前 run 里的小修正 | `/spec-update ...` 或自然语言补充 | `patch` | 当前 run + 当前 change |
| 当前 open change 里的范围变化 | `再补一个接口字段 / 改验收口径` | `scope-delta` | 当前 change，回到 requirement |
| 归档前发现实现不对 | `先别归档，这里改成...` | `archive-fix` | 当前 change |
| 已归档内容补一个修正 | `给上个归档变更补个修正...` | `followup-patch` | 新 patch change，保留 `parent_change_id` |
| 全新、低风险、小范围小修正 | 自然语言直接描述 | `quick-fix` -> `bugfix-to-verification` | `.ai-spec/history/<run-id>/` |

## 什么时候属于 quick-fix

通常需要同时满足：

- 单点小修正，通常落在现有页面、组件或模块内
- bug 修复、样式微调、文案调整、小交互修正
- 不属于“新建页面 / mock 页面 / 演示版页面搭建”
- 不新增真实 API、路由、全局状态
- 不改变需求范围和验收口径
- 风险等级低

只要命中下面任意一条，就不应继续留在 quick-fix：

- 新增真实 API、路由、store
- 新建页面、mock 页面、演示版页面搭建
- 跨模块联动
- 权限、支付、风控、合规
- 需要长期归档、评审、spec 留痕
- 守护阶段发现其实已经不是“小修正”

## 真实输入示例

### 1. 全新小修正 -> quick-fix

输入：

```text
把列表标题文案改一下，按钮也更简洁一点
```

结果：

- `change_context = no-change`
- `route_decision = quick-fix`
- `selected_flow = bugfix-to-verification`

### 2. 全新 mock 页面 -> full-change

输入：

```text
创建一个商品详情 mock 页面，只做演示版，数据本地 mock
```

结果：

- `change_context = no-change`
- `route_decision = full-change`
- `selected_flow = prd-to-delivery`

### 3. 当前 open change 里的文案/样式调整 -> patch

输入：

```text
这个列表页标题改短一点，按钮颜色再轻一点
```

结果：

- 复用当前 open change
- `route_decision = patch`
- 默认回到 `frontend-implementer`

### 4. 当前 open change 里的范围变化 -> scope-delta

输入：

```text
这个变更再补一个接口字段调整，并改一下验收口径
```

结果：

- 复用当前 open change
- `route_decision = scope-delta`
- 回到 `requirement-analyst`

### 5. 归档前修正 -> archive-fix

输入：

```text
先别归档，这个实现不对，改成卡片布局
```

结果：

- `route_decision = archive-fix`
- 不进入 archive fast-path
- 回退到正确专家修正后再归档

### 6. 已归档内容补修 -> followup-patch

输入：

```text
给上个归档变更补一个修正，补一条关键回归测试
```

结果：

- `change_context = archived-change`
- `route_decision = followup-patch`
- 新开 patch change，并保留 `parent_change_id`

### 7. 多个 open change 并存 -> waiting-confirm

输入：

```text
这个按钮颜色再调一下
```

结果：

- 如果当前有多个 open change，系统不会猜
- 会先要求你明确本次要复用哪个 `change_id`

## quick-fix 会留下什么

默认写到：

- `.ai-spec/history/<run-id>/bugfix.md`
- `.ai-spec/history/<run-id>/implementation-notes.md`
- `.ai-spec/history/<run-id>/checklist.md`
- `.ai-spec/history/<run-id>/iterations.md`

这意味着它不是“没流程”，只是不用每次都进入 `openspec/changes/<change-id>/`。

## 最后怎么判断

如果你还拿不准，就按这句判断：

- “这次改动是否需要长期作为需求规范沉淀？”

如果需要，走 `prd-to-delivery`。如果不需要，而且确实只是低风险小修正，就让系统走 `bugfix-to-verification`。

一旦被升级回完整主流程，就会重新回到 `requirement-analyst（需求解析专家） -> frontend-implementer（前端实现专家） -> code-guardian（规范守护专家）` 这条主链。若你显式启用 `main-flow-blocking（主流程阻塞审核）`，才会在 `before-implementation / before-guardian / before-archive` 三个节点停下等待审核。
