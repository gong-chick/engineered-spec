# 开发最佳实践指南

> 面向开发者的默认入口文档

`ai-spec-auto` 是一套面向前端项目的 AI 规范驱动开发工具。它把项目规则、专家资产、IDE 命令入口、OpenSpec 交付产物和 `.ai-spec` 运行状态组织到同一个项目中，让开发过程从“依赖对话临时发挥”转向“按规则、流程和产物稳定沉淀”。

第一次使用时，不需要先理解全部底层机制。先按“安装、提需求、继续推进、看产物”这条主路径跑通一轮，再继续深入协议命令和内部实现即可。

## 介绍

这份文档只解决六个问题：

- `ai-spec-auto` 是什么
- 这套方法适合解决什么问题
- 去哪里获取并安装
- 第一次使用前需要准备什么
- 如何安装并开始第一个需求
- 做完之后应该去哪里看结果

如果只保留一句产品定义，可以先理解成：

> `ai-spec-auto` 不是单个 AI 工具的替代品，而是一套把需求、实现、检查、归档和复盘串成团队开发链路的项目级交付底座。

更简单地说：`OpenSpec` 管变更产物，`.agents` 管规则和专家，`protocol-step / protocol-update / protocol-advance` 管流程推进，`.ai-spec` 管当前事实状态。


## 什么时候使用

这套方法更适合下面这些场景：

- 现有前端项目中的增量需求开发
- 页面开发、列表页改版、设计稿还原
- 希望把需求说明、设计约束、任务拆解和交付检查沉淀为项目产物的团队场景

第一次试点时，建议优先选择低到中风险、半天到一天内可以完成验收的任务。

下面这些场景不建议作为第一轮试点：

- 高风险的权限、合规、安全相关需求
- 时间极短的线上救火问题
- 基础工程和项目约定尚未稳定的仓库

## 当前默认模式

当前内测阶段，完整需求主流程默认使用：

- `mode（运行模式） = auto（自动）`
- `review_policy（审核策略） = none（无阻塞审核）`

这意味着系统会自动选 `flow（流程模板）` 并默认连续推进。若你需要把主链显式停下来等待人工审核，再切换到 `main-flow-blocking（主流程阻塞审核）`。

## 你会得到什么

完成一轮最小闭环后，开发者通常会得到两类结果：

- 代码层结果：页面、路由、接口、样式或交互实现落到项目中
- 产物层结果：需求、设计、任务和检查结果沉淀到可复用目录中

相比只停留在对话里，这种方式更适合多人接力、后续维护和同类需求复用。

## 如何获取

当前推荐的获取方式是通过内网 npm registry 直接安装 `@engineered/ai-spec-auto`。

包名：

```text
@engineered/ai-spec-auto
```

默认安装命令：

```bash
npx @engineered/ai-spec-auto@latest init .
```

如果你读完文档想直接试用，可以优先记住这条命令。更多安装说明可继续看 [安装指南](../install-guide.md)。

## 环境准备

当前包通过内网 npm registry 分发。在 `~/.npmrc` 中加入：

```ini
@ex:registry=http://nodejs.100credit.cn/
```

如果你已经完成过这一步，可以直接进入安装。

## 安装

在目标项目根目录执行：

```bash
npx @engineered/ai-spec-auto@latest init .
```

如果需要明确技术栈，也可以显式指定：

```bash
npx @engineered/ai-spec-auto@latest init . --profile vue
npx @engineered/ai-spec-auto@latest init . --profile react
```

安装完成后，这个项目会具备统一的规则、技能、IDE 命令入口和交付产物目录。

如果你希望更明确地知道“这个工具装进项目后会落下什么文件”，可以继续看 [5 分钟快速上手](../quick-start.md)。

## 快速开始

第一次使用时，推荐按下面顺序进行：

1. 安装到目标项目
2. 初始化项目规范
3. 发起一个真实但低风险的需求
4. 继续推进流程
5. 查看交付产物

这就是普通开发者的默认主路径。第一次不需要绕到更深的协议命令或状态文件。

## 第一个完整示例

如果你希望不是只看命令，而是看一条完整链路如何推进、如何迭代、如何归档，建议直接阅读：

- [需求示例：从发起到归档](./需求示例-从发起到归档.md)

这篇示例会覆盖：

- 第一次发起需求时调用了什么命令
- 中途补充需求时会更新哪些产物
- `/spec-continue` 推进后会新增哪些文件
- 最后如何确认归档，以及归档后文件去了哪里

## 初始化项目规范

安装完成后，在 AI IDE 中执行：

```text
/project-init
```

这一步会先把项目规范背景补齐，后续需求推进会更稳定。

如果你是第一次接触这套体系，到这里为止只需要理解一件事：项目已经准备好进入规范化开发流程。

## 第一个需求

建议直接从一个低风险的真实需求开始，例如：

```text
/spec-start 创建一个商品详情 mock 页面，只做演示版，数据本地 mock
```

如果你直接走协议命令，当前推荐显式写成：

```bash
./node_modules/.bin/ai-spec-auto protocol-step --target . --user-input "创建一个商品详情 mock 页面，只做演示版，数据本地 mock" --mode auto --review-policy none --json
```

如果中途需要补充或修正需求，可以使用：

```text
/spec-update 把页面顶部筛选改成状态 Tab，先不要高级筛选
```

如果当前轮需要继续推进，使用：

```text
/spec-continue
```

## 如何选择 `mode（运行模式）`

通常只需要记住下面三条：

- `auto（自动）`：默认选项。适合大多数完整需求，主代理会自动选 `flow（流程模板）` 并推进。
- `suggest（建议）`：适合你想先看主代理建议，再决定是否启动第一位专家的场景。它会先停在 `start-review（启动确认门禁）`。
- `manual（手动）`：适合你非常确定要走哪条流程模板时使用。当前只支持“手动指定 `flow（流程模板）`”，必须搭配 `--flow`，不支持手工编辑完整专家链。

最简单的判断方式是：

- 不了解底层流程时，用 `auto（自动）`
- 想先看计划再决定时，用 `suggest（建议）`
- 只在你明确知道要锁定哪条流程模板时，才用 `manual（手动）`

## 什么时候不要选 `manual（手动）`

下面这些情况，不建议一上来就选 `manual（手动）`：

- 你还不确定当前需求应该走 `prd-to-delivery（需求到交付流程）` 还是 `bugfix-to-verification（缺陷修复到校验流程）`
- 你只是想让系统“每一步都停下来审核”
- 你第一次试点，还没有建立对主流程和轻量流程边界的直觉

原因很简单：`manual（手动）` 只负责锁定流程模板，不负责替代审核策略。若你只是想保留人工审核，应该显式切换到 `review_policy = main-flow-blocking（主流程阻塞审核）`。

## 需要人工审核时怎么开

当你显式启用 `review_policy = main-flow-blocking（主流程阻塞审核）` 时，它只覆盖主流程三个核心专家：

- `requirement-analyst（需求解析专家）`
- `frontend-implementer（前端实现专家）`
- `code-guardian（规范守护专家）`

它的设计目标不是拖慢主代理，而是把最容易放大返工成本的三个节点显式停下来：

- 需求刚收敛时，确认是否允许进入实现
- 实现刚完成时，确认是否值得进入守护审查
- 守护刚完成时，确认是否可以归档

如果你更希望在 IDE（开发工具）里直接用“自定义命令 + 参数”的形式，推荐直接使用：

```text
/spec-start-review 创建订单列表 mock 页面
/spec-start-review --mode suggest 创建订单列表 mock 页面
/spec-start-review --mode manual --flow prd-to-delivery 创建订单列表 mock 页面
```

这条命令的默认行为等价于：

```bash
./node_modules/.bin/ai-spec-auto protocol-step --target . --user-input "<你的需求原文>" --mode auto --review-policy main-flow-blocking --json
```

也就是说：

- 你只是想保留人工审核时，优先用 `/spec-start-review`
- 你只是想先看建议计划时，在后面补 `--mode suggest`
- 你真的要手动锁定 `flow（流程模板）` 时，才补 `--mode manual --flow <flow-id>`

这里要再次强调一次：`manual（手动）` 只负责手动锁定 `flow（流程模板）`，不等于“每一步都审核”。如果你只是想显式保留三段主流程审核，默认的 `/spec-start-review` 就够了。

因此，如果你看到流程中途暂停，不要先怀疑“系统坏了”。先看是不是显式启用了这套审核策略，并且正在等待这三个门禁中的某一个被批准。

如果你只记一个实践原则，那就是：先用真实需求跑通一次最小闭环，不要第一次就从复杂案例开始。

最推荐的阅读方式，是把这一节和 [需求示例：从发起到归档](./需求示例-从发起到归档.md) 对照着看。

如果你的项目是在较早版本安装的，还没有这个命令入口，可执行：

```bash
npx @engineered/ai-spec-auto@latest update . --update-commands
```

## 做完之后看什么

完成一轮需求后，不要只看 AI 在对话里的回答，而要优先检查交付产物。

重点查看：

- `openspec/changes/<change-id>/`
- `openspec/specs/`

在进行中的变更目录下，通常会看到这些产物：

- `proposal.md`
- `design.md`
- `tasks.md`
- `checklist.md`
- `iterations.md`
- `specs/`

如果这些内容已经形成，说明这轮任务不仅完成了实现，还完成了可复用的沉淀。

## 一个推荐的第一次体验路径

下面是一条最稳的第一次使用路径：

1. 在一个真实项目中执行 `npx @engineered/ai-spec-auto@latest init .`
2. 在 IDE 中执行 `/project-init`
3. 用 `/spec-start` 提交一个 mock 页面或列表页改版需求
4. 用 `/spec-continue` 推进流程
5. 到交付结束后检查 `openspec/changes/<change-id>/`

如果第一次体验顺利，再继续扩展到真实接口、更多页面或更复杂的交付链。

## 常见问题

### 卡住了先看什么

先执行：

```text
/spec-status
```

优先确认当前阶段和下一步提示。仍然不清楚时，再去看当前变更目录下是否已经生成产物。

如果当前状态显示正在等待审批，优先判断是否停在下面三个门禁之一：

- `before-implementation（实现前门禁）`
- `before-guardian（守护前门禁）`
- `before-archive（归档前门禁）`

### 产物在哪里看

进行中的需求优先看 `openspec/changes/<change-id>/`；归档后的长期规范优先看 `openspec/specs/`。

### 什么时候找维护者

出现下面这些情况时，可以直接请维护者协助排查：

- 安装或命令入口不可用
- `/spec-status` 无法解释当前状态，且流程长期无法继续推进
- 产物反复缺失，或交付结果明显偏离项目约定
- 同类问题已经在多个项目里重复出现
