# 需求示例：从发起到归档

这份示例文档用于回答一个非常具体的问题：

> 如果我是第一次使用这套方法，发起一个真实需求、修改两轮、再归档，过程中到底会调用什么命令、生成什么文件、最后怎么收口？

下面用一个低风险的前端需求来演示完整过程。

这篇文档只讲一类场景：

> 完整需求主流程。

也就是说，本文只覆盖 `/spec-start`、`/spec-update`、`/spec-continue` 和“同意归档”这条主链，不展开小需求与补丁修正分支。

## 先分清两大类

当前使用方式可以先粗分为两大类：

### 第一类：完整需求主流程

适合下面这些场景：

- 新功能或新页面
- 跨模块改动，影响范围不止一个点
- 需要评审、需要沉淀完整交付产物
- 预计会有多轮迭代，或需求边界还需要逐步收敛
- 希望最终进入 `openspec/specs/` 和归档记录

这类场景的核心特征是：不仅要改代码，还要把需求、设计、任务、检查和归档完整沉淀下来。

本文下面的示例，讲的就是这一类。

### 第二类：小需求与补丁修正

适合下面这些场景：

- 文案调整、样式微调、局部交互修正
- 低风险、小范围、目标很明确的缺陷修复
- 当前运行中的小补充，或归档后针对既有内容补一个后续修正
- 不希望为一个很小的改动走完整需求主流程

这类场景的核心特征是：目标更小、影响面更窄、收口方式更轻。

如果你判断当前任务属于这一类，请直接看 [小需求与补丁修正指南](./小需求与补丁修正指南.md)。其中会展开 `quick-fix(快速修正流程)`、`followup-patch(后续补丁)`、`archive-fix(归档前修正)`、`bugfix-to-verification(缺陷修复到校验流程)` 等分支。本文不重复展开。

## 示例场景

本次需求设定为：

```text
创建一个订单列表 mock 页面，只做演示版，支持分页、状态筛选和空状态展示
```

后续会追加两次迭代：

1. 把顶部筛选改成状态 Tab
2. 增加空状态文案与重试按钮

这类需求适合第一次试点，因为：

- 风险低
- 页面效果容易验收
- 可以完整覆盖“发起 -> 迭代 -> 检查 -> 归档”这条链

## 先记住当前默认模式

当前内测阶段，完整主流程默认使用：

- `mode（运行模式） = auto（自动）`
- `review_policy（审核策略） = none（无阻塞审核）`

这两个字段是分开的：

- `mode（运行模式）` 决定主代理怎么起步，是自动推进、先给建议，还是手动指定流程模板。
- `review_policy（审核策略）` 决定主流程哪些节点必须停下来等人工审核。

本文为了把完整门禁链路讲清楚，下面示例会显式切换到 `main-flow-blocking（主流程阻塞审核）`。只有在这种显式开启的情况下，`prd-to-delivery（需求到交付流程）` 才会在三个节点停下来：

1. `requirement-analyst（需求解析专家）` 完成后，进入 `before-implementation（实现前门禁）`
2. `frontend-implementer（前端实现专家）` 完成后，进入 `before-guardian（守护前门禁）`
3. `code-guardian（规范守护专家）` 完成后，进入 `before-archive（归档前门禁）`

如果你直接走协议命令，当前推荐写法如下：

```bash
./node_modules/.bin/ai-spec-auto protocol-step --target . --user-input "创建一个订单列表 mock 页面，只做演示版，支持分页、状态筛选和空状态展示" --mode auto --review-policy main-flow-blocking --json
```

如果希望先看主代理建议、确认后再启动第一位专家，可以改成：

```bash
./node_modules/.bin/ai-spec-auto protocol-step --target . --user-input "创建一个订单列表 mock 页面，只做演示版，支持分页、状态筛选和空状态展示" --mode suggest --review-policy main-flow-blocking --json
```

如果要显式指定流程模板，则使用 `manual（手动）`，并且必须补 `--flow`：

```bash
./node_modules/.bin/ai-spec-auto protocol-step --target . --user-input "创建一个订单列表 mock 页面，只做演示版，支持分页、状态筛选和空状态展示" --mode manual --flow prd-to-delivery --review-policy main-flow-blocking --json
```

其中：

- `auto（自动）`：主代理自动选流程并推进
- `suggest（建议）`：先产出 `run-plan（运行计划）`，再停在 `start-review（启动确认门禁）`
- `manual（手动）`：只允许手动锁定 `flow（流程模板）`，不等于“每一步都人工审核”

如果你更习惯在 IDE（开发工具）里直接用自定义命令，也可以把上面三种写法对应成：

```text
/spec-start-review 创建一个订单列表 mock 页面，只做演示版，支持分页、状态筛选和空状态展示
/spec-start-review --mode suggest 创建一个订单列表 mock 页面，只做演示版，支持分页、状态筛选和空状态展示
/spec-start-review --mode manual --flow prd-to-delivery 创建一个订单列表 mock 页面，只做演示版，支持分页、状态筛选和空状态展示
```

这里推荐把 `/spec-start-review` 当成“我要显式保留人工审核”的固定入口来记，而不是把 `manual（手动）` 当作审核开关。`manual（手动）` 仍然只表示手动锁定 `flow（流程模板）`。

## 开始之前

先确认项目已经完成安装与初始化：

```bash
npx @engineered/ai-spec-auto@latest init .
```

然后在 IDE 中执行：

```text
/project-init
```

到这一步为止，项目已经具备统一的规则背景和命令入口。

## 第 1 步：发起第一个需求

在 IDE 中输入：

```text
/spec-start 创建一个订单列表 mock 页面，只做演示版，支持分页、状态筛选和空状态展示
```

### 这一步本质上做了什么

- IDE 入口读取 `/spec-start` 模板
- 主代理触发协议首轮
- 首轮需求分析开始生成交付产物

如果从协议视角理解，这一轮通常对应：

```text
protocol-step
```

### 这一步之后通常会看到什么

优先看：

- `openspec/changes/<change-id>/`
- `.ai-spec/current-run.json`

在 `openspec/changes/<change-id>/` 下，通常会开始出现：

- `proposal.md`
- `design.md`
- `tasks.md`
- `specs/`

这说明需求已经不只是停留在对话里，而是开始沉淀到文件。

## 第 2 步：等待需求解析完成并进入实现前审核

当 `requirement-analyst（需求解析专家）` 完成后，当前示例里显式启用的审核策略会把流程停在 `before-implementation（实现前门禁）`。这时不要直接继续推进实现，而是先看需求范围是否已经收敛。

在 IDE 中你通常会看到类似“当前停在实现前审核，等待批准”的提示。

如果确认当前范围没问题，可以输入：

```text
按当前范围进入实现
```

如果从协议视角理解，这一轮通常对应：

```text
protocol-update
```

### 这一步重点看什么

优先确认：

- `proposal.md` 是否已经把需求边界写清楚
- `design.md` 是否仍然保持“演示版 / mock 页面”的约束
- `tasks.md` 是否没有提前扩张到真实接口或额外模块

## 第 3 步：继续推进到实现

在完成实现前审核后，再输入：

```text
/spec-continue
```

### 这一步本质上做了什么

- 当前轮从需求分析推进到下一轮
- 实现专家接手当前任务
- 后续再由规范检查环节补交付检查结果

如果从协议视角理解，这一轮通常对应：

```text
protocol-advance
```

### 这一步之后通常会新增什么

除了业务代码改动以外，`openspec/changes/<change-id>/` 中会逐步补充：

- `checklist.md`
- `iterations.md`

同时，`.ai-spec/current-run.json` 会更新当前阶段、当前专家和门禁状态。

## 第 4 步：第一次迭代补充需求

假设你看完第一轮结果后，发现筛选形式需要调整，这时输入：

```text
/spec-update 把顶部筛选改成状态 Tab，先不要高级筛选
```

### 这一步本质上做了什么

- 当前运行收到增量补充输入
- 需求边界与设计约束被重新校正
- 后续实现不再继续沿用旧的筛选方式

如果从协议视角理解，这一轮通常对应：

```text
protocol-update
```

### 这一步之后重点看什么

优先重新查看：

- `proposal.md`
- `design.md`
- `tasks.md`

这些文件会体现本轮迭代后的约束变化。也就是说，增量修改不是只停留在一句补充说明里，而是要回写到交付产物。

## 第 5 步：第二次继续推进

继续输入：

```text
/spec-continue
```

### 这一步本质上做了什么

- 流程根据新的需求约束继续推进
- 当前实现围绕更新后的任务清单收口
- 检查结果继续补回交付产物

如果从协议视角理解，这一轮通常仍对应：

```text
protocol-advance
```

如果这一步正常完成，业务代码和 `checklist.md / iterations.md` 会继续更新。

## 第 6 步：第二次迭代补充需求

再补一次迭代，例如：

```text
/spec-update 增加空状态文案和重试按钮，保持演示版，不接真实接口
```

### 这一步应该重点关注什么

这类更新最容易导致范围扩散，因此建议重点看：

- `design.md` 是否仍然明确是演示版
- `tasks.md` 是否只增加本次小改动
- `specs/` 是否没有被误写成真实接口方案

这里虽然补的是一个较小改动，但它仍然属于当前完整需求主流程里的增量迭代，不等于“小需求与补丁修正”分支。

这类增量调整在协议视角下，通常仍对应：

```text
protocol-update
```

## 第 7 步：实现收口后进入守护前审核

继续输入：

```text
/spec-continue
```

如果前面的实现和检查已经基本收口，这一轮通常会先停在 `before-guardian（守护前门禁）`，而不是直接进入 `code-guardian（规范守护专家）`。

这一步的意义是：先让人确认“当前实现结果是否值得继续做守护审查”，避免把明显需要返工的结果直接推进到守护阶段。

此时你应重点确认：

- 业务代码是否符合本轮需求
- `proposal.md / design.md / tasks.md / checklist.md / iterations.md / specs/` 是否仍与当前实现一致
- 当前状态是否已经来到 `before-guardian（守护前门禁）`

如果确认可以进入守护阶段，可以输入：

```text
同意进入守护审查
```

这一步在协议视角下通常仍对应：

```text
protocol-update
```

## 第 8 步：继续推进到归档前审核

在守护前审核通过后，再输入：

```text
/spec-continue
```

当 `code-guardian（规范守护专家）` 完成后，流程会停在 `before-archive（归档前门禁）`。这时重点确认：

- `checklist.md` 中的检查结论是否清楚
- `iterations.md` 是否已经记录问题、修正动作和残留风险
- 当前实现是否可以作为本轮完整结果归档

## 第 9 步：确认归档

确认本轮可以收口后，输入：

```text
同意归档
```

### 这一步本质上做了什么

- 当前运行收到归档确认
- 流程进入归档收口
- 本轮需求从“进行中”走向“完成并沉淀”

如果从协议视角理解，这一步通常对应：

```text
protocol-update + archive-change
```

### 归档后通常会发生什么

归档后的结果重点看两处：

- `openspec/specs/`
- `openspec/changes/archive/`

通常可以这样理解：

- 可长期复用的规范内容会合并进 `openspec/specs/`
- 本次需求的变更目录会进入归档区，保留完整记录

## 命令与产物对照

| 动作 | 输入示例 | 通常对应的内部调用 | 重点结果 |
| --- | --- | --- | --- |
| 发起需求 | `/spec-start-review 创建一个订单列表 mock 页面...` | `protocol-step --mode auto --review-policy main-flow-blocking` | 生成首轮交付产物，如 `proposal.md`、`design.md`、`tasks.md`、`specs/` |
| 切到建议模式 | `/spec-start-review --mode suggest ...` | `protocol-step --mode suggest --review-policy main-flow-blocking` | 先生成 `run-plan（运行计划）`，停在 `start-review（启动确认门禁）` |
| 切到手动模式 | `/spec-start-review --mode manual --flow prd-to-delivery ...` | `protocol-step --mode manual --flow prd-to-delivery --review-policy main-flow-blocking` | 锁定 `flow（流程模板）`，不允许自动猜测 |
| 需求审核 | `按当前范围进入实现` | `protocol-update` | 清除 `before-implementation（实现前门禁）`，放行实现 |
| 继续推进 | `/spec-continue` | `protocol-advance` | 推进到下一专家，补充实现结果与检查结果 |
| 增量修改 | `/spec-update 把顶部筛选改成状态 Tab...` | `protocol-update` | 回写需求变化，更新 `proposal / design / tasks` |
| 实现审核 | `同意进入守护审查` | `protocol-update` | 清除 `before-guardian（守护前门禁）`，放行守护 |
| 再次推进 | `/spec-continue` | `protocol-advance` | 按最新约束继续实现和检查，或推进到守护与归档前审核 |
| 归档确认 | `同意归档` | `protocol-update + archive-change` | 合并长期规范并完成本轮归档 |

## 第一次使用时最应该记住什么

第一次使用这套方法，不需要先记住全部协议细节。只需要记住下面三件事：

1. `spec-start` 是启动需求的入口，默认等价于 `auto（自动） + none（无阻塞审核）`；如果你想演示完整审核门禁，再显式切到 `main-flow-blocking（主流程阻塞审核）`
   最直接的做法就是改用 `/spec-start-review`
2. `spec-update` 除了补需求，也承担审核意见回写，例如放行实现、放行守护、同意归档
3. `spec-continue` 是把流程继续往前推的入口，但前提是当前门禁已经被放行

前提是：你已经判断当前任务属于“完整需求主流程”。

最后，当你确认可以收口时，再输入归档确认意见即可。

如果你的项目还没有 `/spec-start-review`，可先执行：

```bash
npx @engineered/ai-spec-auto@latest update . --update-commands
```
