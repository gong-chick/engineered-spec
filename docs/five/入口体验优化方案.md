# 入口体验优化方案

## 1. 背景

当前项目已经逐步形成了安装、协议推进、专家协同、交付产物和运行状态的完整主链，但从第一次接触的用户视角看，入口体验仍然偏重。

问题不在于能力不够，而在于能力暴露顺序还不够友好。开发者在真正开始使用之前，容易先接触到过多入口、过多概念和过多内部实现细节，导致“知道系统很完整”，但“不确定应该如何开始”。

因此，这份方案不讨论底层机制是否成立，而重点解决一个问题：如何让用户更自然地进入这套体系，并逐步理解它。

## 2. 当前不足

### 2.1 入口分散

当前用户可能同时接触到：

- 根 `README`
- `docs/README`
- `quick-start`
- `install-guide`
- `docs/four`
- 历史阶段文档
- IDE 命令
- CLI 子命令

这会带来两个直接问题：

- 用户不知道“第一步应该看哪一个”。
- 不同入口的粒度不同，容易在尚未上手前就被信息量压住。

### 2.2 首屏过重

用户刚进入体系时，最需要的是一条最小上手路径，但当前材料里仍然容易先暴露这些概念：

- 协议命令
- 运行状态
- 专家协同
- 归档机制
- 平台扩展方向

这些内容对维护者很重要，但对首次试点开发者来说，过早出现会抬高心理门槛。

### 2.3 入口不统一

当前入口正在逐步扩展为多层结构：

- IDE
- CLI
- Hub
- 后续 `OpenClaw`

这些入口本身没有问题，问题在于用户不容易立刻理解：

- 谁是主入口
- 谁负责触发
- 谁负责安装
- 谁负责审批与回传

当入口边界表达不清时，用户会把“能力多”理解成“系统复杂”。

## 3. 优化目标

入口优化的目标不是减少能力，而是调整能力的暴露方式。

需要同时满足四个目标：

- 普通开发者第一次接触时，只需要理解一条主路径。
- 维护者和评审可以在不打扰普通开发者的前提下看到完整机制。
- 多入口体系对外表现为清晰分工，而不是多套并行逻辑。
- 深层机制继续保留，但默认下沉，不占据首屏。

## 4. 解决方案

### 4.1 文档入口分层

把文档入口固定成三层：

- 第一层：唯一推荐入口，只服务“第一次怎么开始”。
- 第二层：按受众分流，区分普通开发者与维护者 / 评审。
- 第三层：历史资料、底层实现、专题方案，作为按需阅读材料。

当前建议的落点是：

- `README` 与 `docs/README` 只负责引流。
- `docs/four/` 负责当前主入口和受众分层。
- `docs/five/` 负责专题优化方案与专项设计。
- `docs/paser_three/` 保留为历史阶段资料，不再承担默认入口职责。

### 4.2 用户动作收束

对普通开发者，只保留一条稳定动作链：

1. 安装
2. 初始化项目规范
3. 提需求
4. 继续推进
5. 看产物

对应的默认入口应尽量稳定在：

- 安装：`init`
- 初始化：`/project-init`
- 发起需求：`/spec-start`
- 继续推进：`/spec-continue`
- 查看状态：`/spec-status`

协议子命令、状态文件和内部编排细节不作为普通开发者的首屏信息。

### 4.3 入口职责统一表达

要把多入口体系讲清楚，关键不是减少入口，而是固定分工：

| 入口 | 主要职责 |
| --- | --- |
| Hub | 选择能力、组织场景方案、输出安装清单 |
| CLI | 安装能力、执行协议、承接结构化输入输出 |
| IDE | 提供本地交互入口，服务开发者日常使用 |
| `OpenClaw` | 承担远程触发、审批放行、状态查询和结果回传 |

只要这四层分工清楚，用户就不会把“入口变多”误解成“系统失控”。

### 4.4 首屏信息减负

首屏内容需要遵循一个原则：先给路径，再给原理。

普通开发者首屏只应该回答：

- 这套方法解决什么问题
- 我今天怎么开始
- 做完去哪里看结果

只有在进入第二层文档后，才逐步暴露：

- 协议命令
- 运行状态
- 专家协同
- 归档机制
- Hub / `OpenClaw` / 控制面关系

### 4.5 让状态可见，但不要求先懂底层

“流程会不会卡住”是开发者心理门槛的重要来源。更优雅的做法不是要求每个人都先理解 `.ai-spec/current-run.json`，而是优先提供：

- 可读的状态提示
- 明确的下一步动作
- 门禁阻塞原因说明
- 产物生成位置提示

也就是说，先让用户知道“现在到哪了，下一步做什么”，再按需引导维护者查看底层状态文件。

## 5. 分阶段实施建议

### 5.1 第一阶段：先收拢文档入口

- 固定当前主入口。
- 区分开发者文档与维护者文档。
- 将专题方案和历史资料剥离出主入口。

目标是先解决“第一次应该从哪里开始”的问题。

### 5.2 第二阶段：再收拢命令与交互

- 把普通开发者高频动作固定成少量命令。
- 把协议子命令继续留在维护者与底层实现视角。
- 让状态查看与下一步提示更可读。

目标是减少“命令很多、不知道该用哪个”的犹豫感。

### 5.3 第三阶段：统一远程与本地入口

- Hub 继续负责资产组合。
- IDE 继续负责本地开发交互。
- `OpenClaw` 接管远程触发、审批和回传。
- CLI 继续作为统一的协议执行与结构化输出底座。

目标是让本地入口和远程入口共用同一套底层能力，而不是形成两套平行体系。

## 6. 验收标准

入口体验优化是否生效，可以用下面几个标准判断：

- 普通开发者第一次阅读后，能在 5 分钟内说清楚“先做什么、再做什么、做完看什么”。
- 维护者能够快速找到协议、状态、排障和治理文档，而不会与开发者入口混在一起。
- 用户能够理解 Hub、CLI、IDE、`OpenClaw` 的边界，不再把它们看成重复入口。
- 常见卡点可以先通过提示和状态说明定位，而不是默认要求用户读底层状态文件。

## 7. 结论

当前项目的入口体验问题，本质上不是能力缺失，而是暴露顺序和入口分层还不够克制。

更优雅的解决方式，不是继续堆入口、堆概念、堆说明，而是让系统遵循一条清晰原则：

- 普通开发者先用起来
- 维护者再看全貌
- 平台入口各司其职
- 深层机制按需显露

只要这条原则建立起来，后续 Hub、CLI、IDE、`OpenClaw` 和自动化控制面的能力，就能在不抬高认知门槛的前提下继续扩展。
