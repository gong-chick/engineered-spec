# 入口最佳实践指南

> 适用对象：第一次接触这套体系的开发者、项目试点负责人、维护者、评审人员

这份指南不解释全部底层机制，只回答一个更实际的问题：

- 不同人应该从哪里进入
- 第一次使用时应该按什么顺序做
- 什么时候继续深入，什么时候先停在当前层

如果把这套体系讲成一句话，最重要的不是“入口很多”，而是“不同人只看自己当前需要的那一个入口”。

## 1. 总原则

入口体验要稳定，必须先遵守四条原则：

- 普通开发者只走一条主路径，不同时暴露多套入口。
- 维护者和评审进入更深一层，但不把深层机制前置给所有人。
- Hub、CLI、IDE、`OpenClaw` 各自承担清晰职责，不互相抢入口。
- 先让人用起来，再让人理解全貌。

如果这四条原则没有被坚持，再多文档和能力也只会让人觉得系统越来越重。

## 2. 如果你是开发者，直接这样用

### 2.1 只记住一条主路径

开发者第一次接触时，只需要按下面这条顺序使用：

1. 安装
2. 初始化项目规范
3. 提需求
4. 继续推进
5. 看产物

对应的默认入口固定为：

- 安装：`npx @engineered/ai-spec-auto@latest init .`
- 初始化：`/project-init`
- 发起需求：`/spec-start`
- 继续推进：`/spec-continue`
- 查看状态：`/spec-status`

这条路径之外的协议细节、状态文件和内部编排机制，不是第一次使用时必须先理解的内容。

### 2.2 第一步：先判断任务适不适合走这套方法

优先从下面这类任务开始：

- Mock 页面
- 列表页改版
- 设计稿还原
- 低到中风险的前端增量需求

第一次不建议直接选择：

- 高风险权限、合规、安全相关需求
- 时间极短的线上救火
- 基础工程尚未稳定的项目

最好的试点任务，是半天到一天内可以完成验收、能看见产物沉淀价值的任务。

### 2.3 第二步：只通过默认入口完成项目接入

在项目根目录执行：

```bash
npx @engineered/ai-spec-auto@latest init .
```

接入完成后，在 IDE 中执行：

```text
/project-init
```

到这一步为止，开发者只需要知道两件事：

- 项目已经具备统一规范背景
- 后续需求会沿着同一条交付链推进

不需要先去理解 registry、manifest、状态机或底层脚本。

### 2.4 第三步：用真实需求进入流程

建议直接从真实但低风险的任务开始，例如：

```text
/spec-start 创建一个商品详情 mock 页面，只做演示版，数据本地 mock
```

中途如果需要补充需求，用：

```text
/spec-update 把顶部筛选改成状态 Tab，先不要高级筛选
```

需要继续推进时，用：

```text
/spec-continue
```

### 2.5 第四步：做完不要只看对话，要看产物

开发者验收时，不要只看“AI 回答说完成了没有”，而是优先看：

- `openspec/changes/<change-id>/`
- `openspec/specs/`

最关键的是确认下面这些内容是否已经形成：

- `proposal.md`
- `design.md`
- `tasks.md`
- `checklist.md`
- `iterations.md`
- `specs/`

如果产物没有形成，这轮任务就还不能算真正稳定交付。

### 2.6 第五步：卡住时先这样处理

如果流程卡住，不要第一时间去读底层文件，先按这个顺序处理：

1. 执行 `/spec-status`
2. 看当前提示的阶段和下一步动作
3. 看 `openspec/changes/<change-id>/` 是否已有产物
4. 仍然无法判断时，再找维护者

这一步的关键，不是让开发者学会排查所有底层机制，而是让开发者先知道“现在该做什么”。

## 3. 如果你是维护者或评审，应该这样进入

维护者和评审不应该复用开发者入口，而应直接进入更深一层的资料。

推荐路径如下：

1. [架构设计与治理说明](../four/架构设计与治理说明.md)
2. [OpenSpec / 协议流说明](../openspec-guide.md)
3. [入口体验优化方案](./入口体验优化方案.md)

你们更需要关心的是：

- 协议如何推进
- 门禁如何阻断或放行
- 运行状态如何判断
- 出问题时先查哪里
- 文档入口是否对普通开发者足够友好

维护者与评审的最佳实践不是“把所有细节解释给开发者”，而是“替开发者吸收复杂度，让入口始终保持克制”。

## 4. 如果你是项目负责人，应该怎么组织试点

项目负责人不需要自己替代开发者或维护者，而是要把试点节奏安排对。

建议按下面顺序组织：

1. 先选择一个低风险、可快速验收的真实需求
2. 明确本轮只验证一条最小闭环
3. 要求开发者只走默认入口，不扩展更多命令
4. 要求维护者只在卡点出现时介入，不抢开发者路径
5. 做完后统一看产物和复盘结论

项目负责人最重要的动作，不是让所有人一开始就学会系统全貌，而是先让团队获得一次“这套方法确实能跑通”的正反馈。

## 5. 入口分工的最佳实践

多入口本身不是问题，入口职责不清才是问题。

推荐的长期分工如下：

| 入口 | 最佳实践定位 |
| --- | --- |
| `README` / `docs/README` | 只做引流，不承载全部细节 |
| `docs/four` | 当前主入口，负责按受众分层 |
| `docs/five` | 专题优化方案与最佳实践沉淀 |
| CLI | 安装、协议执行、结构化输入输出 |
| IDE | 开发者日常本地交互入口 |
| Hub | 选择能力、组织场景方案、输出安装清单 |
| `OpenClaw` | 远程触发、审批放行、状态回传与团队协同控制面 |

只要这一层分工稳定，用户就不会把“入口变多”理解为“系统越来越难用”。

## 6. 推荐的阅读与使用顺序

### 路径一：普通开发者

1. 读这篇《入口最佳实践指南》
2. 看 [开发最佳实践指南](../four/开发最佳实践指南.md)
3. 按 [5 分钟快速上手](../quick-start.md) 操作
4. 需要更多安装细节时，再看 [安装指南](../install-guide.md)

### 路径二：维护者 / 评审

1. 读这篇《入口最佳实践指南》
2. 看 [架构设计与治理说明](../four/架构设计与治理说明.md)
3. 看 [入口体验优化方案](./入口体验优化方案.md)
4. 再进入 OpenSpec 与运行机制文档

### 路径三：团队协同与远程入口

1. 先明确本地入口已经稳定
2. 再让 Hub 承接能力组合
3. 再让 `OpenClaw` 承接远程触发、审批和回传
4. 始终复用同一套 CLI 协议底座

## 7. 常见反模式

下面这些做法会明显拉高入口门槛，应尽量避免：

- 第一次接触就把历史阶段文档、架构文档、命令清单全部抛给开发者
- 让开发者先学习 `protocol-step`、状态文件和内部编排机制，再开始做需求
- 同时把 README、文档索引、CLI 子命令、IDE 命令都当成“默认入口”
- 用维护者视角写普通开发者文档
- 在本地入口还未收稳时，就急于扩展成远程自动化入口

## 8. 验收标准

如果这份最佳实践真的生效，应该能看到下面这些结果：

- 普通开发者第一次接触时，知道自己只需要走一条主路径
- 开发者能在 5 分钟内说清楚“先做什么、再做什么、做完看什么”
- 维护者和评审有自己的深层入口，不再把复杂度前置给开发者
- Hub、CLI、IDE、`OpenClaw` 的职责边界逐步稳定
- 系统能力继续增长，但入口认知负担没有同步增长

## 9. 结语

入口体验的最佳实践，不是把所有能力一次性讲全，而是让每类用户都能在正确的时间看到正确的入口。

对开发者来说，最重要的是先把一条主路径跑通；对维护者来说，最重要的是把复杂度挡在开发者入口之后；对平台来说，最重要的是让多入口最终表现为一套统一体验，而不是多套分裂逻辑。
