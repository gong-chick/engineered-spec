# 全局 Launcher 与用户侧 Runtime 缓存机制说明

> 文档定位：用于说明 `ai-spec-auto` 为什么要把协议入口切到全局 `launcher(启动器)`，以及用户侧 `runtime cache(运行时缓存)` 如何工作、为什么这样设计、失败时怎么兜底。

## 1. 先回答一个最核心的问题

这套机制的目标只有三个：

- 让用户执行协议命令时，尽量总是使用最新可用版本
- 让更新过程尽量无感，不要求用户频繁手动更新
- 不再把自动更新副作用压到目标项目目录里

如果只保留一句话，可以先理解成：

> 项目里的协议入口只是触发点，真正执行命令的是用户侧全局 `launcher(启动器)` 和 `runtime cache(运行时缓存)`。

## 2. 为什么要这样做

在持续迭代阶段，`ai-spec-auto` 更新频率很高。如果仍然依赖每个项目自己的 `node_modules/@engineered/ai-spec-auto`，会带来几个明显问题：

- 用户需要在多个项目里重复更新依赖
- 更新动作容易改动 `package.json`、lockfile 或项目内命令模板
- 新旧项目可能同时跑着不同版本，排查问题成本高
- 每次协议执行前都要求用户先手动更新，体验会明显变差

因此，当前选择把“版本更新”和“项目交付”拆开：

- 项目目录只保留需求产物、规则、命令模板和业务代码
- 自动更新只发生在用户目录 `~/.ai-spec-auto/`

## 3. 现在的结构长什么样

当前推荐这样理解目录分层：

```text
目标项目/
├── .agents/
├── openspec/
├── .cursor/commands/ 或 .claude/commands/
└── 业务代码

用户目录 ~/.ai-spec-auto/
├── bin/
│   └── ai-spec-auto                # 全局 launcher（启动器）
├── runtime/
│   ├── releases/                   # 已下载 runtime 版本
│   └── embedded/                   # 内置保底 runtime
├── lib/
│   ├── runtime-bootstrap.js
│   ├── runtime-fallback.js
│   └── runtime-embedded.js
└── state/
    ├── runtime-refresh.json
    ├── runtime-refresh.lock
    └── runtime-bootstrap-config.json
```

这里最关键的边界是：

- 目标项目不负责保存“最新执行版本”
- 用户目录负责保存“当前可用执行版本”

## 4. 执行链路是怎么走的

### 4.1 新项目

新安装或新同步出来的命令模板，会直接调用：

```bash
"$HOME/.ai-spec-auto/bin/ai-spec-auto"
```

也就是说，新的 `spec-start(发起需求)`、`spec-update(补充需求)`、`spec-continue(继续推进)`、`spec-status(查看状态)` 等命令，不再优先走项目内 `node_modules/.bin/ai-spec-auto`。

### 4.2 老项目

老项目即使暂时还保留本地 CLI(命令行入口)，也会先进入项目内的 `bin/cli.js`。这个入口会在真正处理协议命令前，先尝试把执行切换到用户侧最新可用 `runtime(运行时)`。

也就是说：

- 新项目默认直接走全局入口
- 老项目通过本地入口转发到全局运行时

两条路径最终都会汇合到同一套用户侧缓存版本。

## 5. 什么叫 runtime cache

`runtime cache(运行时缓存)` 指的是：

- 当前机器上已经下载好的 `ai-spec-auto` 可执行版本集合
- 存在 `~/.ai-spec-auto/runtime/releases/` 下
- 每次协议命令执行时，从这里选择“最新可用版本”来运行

它解决的不是“安装一次依赖”，而是“把运行时版本和项目本身解耦”。

这样做的直接好处是：

- 多个项目共享同一套最新执行版本
- 用户不用在每个项目里重复安装和升级
- 目标项目目录不会因为升级工具而不断被修改

## 6. 为什么需要全局 launcher

`launcher(启动器)` 的职责很小，但很关键：

1. 接住所有协议命令
2. 判断当前是否需要刷新用户侧 runtime
3. 选择可用版本并转发执行
4. 在刷新失败时做兜底回退

它本身不负责协议编排，不负责需求解析，也不直接修改项目代码。  
可以把它理解成“执行入口调度层”，而不是“协议层”。

## 7. 刷新策略

当前默认策略如下：

- `TTL(刷新时间窗)`：`180` 分钟
- 在刷新时间窗内，默认不重复检查远端更新
- 超过 TTL 后，再尝试静默刷新
- 如果刷新成功，切到新的 `release(运行版本)`
- 如果刷新失败，优先回退到上一个可用版本

这样设计的目的是平衡两件事：

- 尽量让用户用到新版本
- 避免每次执行命令都联网刷新，拖慢体验或放大失败概率

## 8. 当前的兜底策略

为了避免“第一次使用就因为更新失败而完全不可用”，当前已经补了三层兜底：

### 8.1 last-known-good(最后一次可用版本)

状态文件里会记录最近一次成功可用的 `release(运行版本)`。  
当最新刷新失败时，优先回退到这个版本，而不是直接报错。

### 8.2 disable refresh(禁用刷新开关)

支持通过环境变量关闭自动刷新：

```bash
AI_SPEC_DISABLE_RUNTIME_REFRESH=1
```

这个开关主要用于两类场景：

- 临时怀疑远端刷新链路异常
- 需要快速定位“是刷新问题还是协议问题”

### 8.3 embedded fallback(内置保底运行时)

用户侧目录里还会同步一份 `embedded runtime(内置保底运行时)`：

- 它不是最新版本缓存
- 但可以在“没有任何 release 可用、同时刷新失败”时充当最后兜底

这意味着：

- 新项目第一次执行协议命令
- 即使当前机器上还没有成功刷新过任何 release
- 只要 launcher 和 embedded runtime 已同步成功
- 也不会立刻因为“没有可用执行版本”而完全卡死

## 9. 这套机制不会改哪些东西

当前明确不应该被自动更新逻辑修改的内容包括：

- 目标项目里的 `package.json`
- 目标项目里的 lockfile
- 目标项目里的 `node_modules`
- 目标项目里的 `.agents`
- 目标项目里的 `openspec`
- 目标项目里的业务代码文件

如果后续发现某次“自动更新”引起了上面这些目录或文件的变化，应优先视为异常，而不是正常行为。

## 10. 这套机制会改哪些东西

自动更新只应修改用户目录里的内容，例如：

- `~/.ai-spec-auto/bin/`
- `~/.ai-spec-auto/runtime/`
- `~/.ai-spec-auto/lib/`
- `~/.ai-spec-auto/state/`

也就是说，版本更新的副作用应该被限制在用户侧，而不应泄漏到具体项目里。

## 11. 开发者应该怎么理解这套变化

如果你是普通开发者，最简单的理解方式是：

- 继续按 `spec-start / spec-update / spec-continue / spec-status` 使用
- 不需要手动去每个项目里升级 `ai-spec-auto`
- 协议命令会尽量自动切到当前机器上最新可用的执行版本

如果你是维护者，则需要额外知道：

- 更新链路优先看 `~/.ai-spec-auto/state/runtime-refresh.json`
- 冷启动兜底优先看 `last-known-good(最后可用版本)` 和 `embedded runtime(内置保底运行时)`
- 真正出问题时，先判断是“刷新失败”还是“协议执行失败”，不要混为一类

## 12. 推荐的排查顺序

当你怀疑“用户没用上最新版本”或“自动更新链路异常”时，建议按下面顺序检查：

1. 看 `~/.ai-spec-auto/state/runtime-refresh.json`
2. 看当前 `active_release(当前版本)` 和 `last_known_good_release(最后可用版本)`
3. 看 `~/.ai-spec-auto/runtime/releases/` 下是否已有可用版本
4. 看 `~/.ai-spec-auto/runtime/embedded/` 是否存在
5. 再判断问题是在远端刷新、用户侧缓存、还是协议执行本身

## 13. 一句话总结

这次机制调整的核心不是“多加了一层脚本”，而是把“项目交付”和“版本更新”彻底拆开：

- 项目目录只承载交付事实
- 用户目录承载执行版本
- `launcher(启动器)` 负责把协议命令切到最新可用 runtime
- 多层 fallback(回退兜底) 负责避免冷启动或刷新失败时直接中断

如果这条边界一直守住，后续即使继续高频迭代，也不需要再把升级压力转嫁给每个目标项目。
