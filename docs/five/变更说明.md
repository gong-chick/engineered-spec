# Visual 集成变更说明

> 本文档说明 ai-spec-auto 在集成 visual 监控功能后的变更内容

## 变更概述

### 新增功能

1. **Visual Hooks 机制** - 在关键协议节点自动推送运行状态到 visual 服务
2. **配置驱动集成** - 通过 `.ai-spec/visual-config.json` 控制集成行为
3. **优雅降级** - visual 服务不可用时，协议正常推进不受影响
4. **默认启用 Visual Bridge** - init 命令默认启用 visual bridge，无需手动确认

### 变更文件

#### 核心集成代码

```
internal/visual-hooks/
├── index.js                    # Visual Hooks 入口
├── config-loader.js            # 配置加载器
├── push-client.js              # HTTP/WebSocket 推送客户端
└── README.md                   # 使用文档
```

#### 协议集成点

- `internal/ai-protocol-workflow.js` - 在 `advanceProtocolStep` 中调用 `onRunStateChange`
- `bin/archive-change.js` - 在 `archiveChange` 完成后调用 `onArchiveComplete`

#### 配置示例

- `.ai-spec/visual-config.example.json` - 配置模板

#### Install 流程变更

- `bin/install-workflow.js` (Line 170) - `visualBridge` 默认值从 `'ask'` 改为 `'yes'`
- `bin/install-workflow.js` (Line 1157-1160) - 移除交互式提示

#### 文档

## 行为变更

### 1. Init 命令行为变更

#### 变更前

```bash
npx ai-spec-auto init .

# 会提示：
# ℹ 是否启用 Visual 平台桥接？
#   启用后会在 .ai-spec/visual-bridge.json 落可视化桥接配置，但默认不影响现有协议执行链。
# 启用 visual bridge? (y/N) [默认 N]:
```

#### 变更后

```bash
npx ai-spec-auto init .

# 不再提示，自动启用 visual bridge
# 自动创建 .ai-spec/visual-bridge.json
```

**影响**：
- ✅ 用户体验更流畅，减少交互步骤
- ✅ Visual bridge 默认启用，便于后续集成
- ⚠️ 如需禁用，可以手动修改 `.ai-spec/visual-bridge.json` 中的 `enabled: false`

### 2. 协议推进行为变更

#### 变更前

```bash
npx ai-spec-auto protocol-step --user-input "需求"

# 纯本地执行，不推送数据
```

#### 变更后

```bash
npx ai-spec-auto protocol-step --user-input "需求"

# 如果 .ai-spec/visual-config.json 启用，会尝试推送到 visual 服务
# 如果 visual 服务不可用，降级为本地执行，不影响协议推进
```

**影响**：
- ✅ 实时监控运行状态
- ✅ 优雅降级，不影响现有功能
- ⚠️ 如不需要监控，设置 `visual-config.json` 中 `enabled: false`

### 3. Archive 归档行为变更

#### 变更前

```bash
npx ai-spec-auto archive-change

# 纯本地归档
```

#### 变更后

```bash
npx ai-spec-auto archive-change

# 如果 visual 配置启用，会推送归档结果
# 降级机制同上
```

**影响**：
- ✅ 归档结果可在 visual 控制台查看
- ✅ 不影响本地归档逻辑

## 配置文件说明

### .ai-spec/visual-config.json

**位置**：项目根目录 `.ai-spec/visual-config.json`

**示例**：

```json
{
  "enabled": true,
  "visual_url": "http://localhost:3000",
  "workspace_id": "my-project",
  "workspace_name": "我的项目",
  "push_mode": "hook",
  "push_timeout_ms": 3000,
  "retry_times": 1
}
```

**字段说明**：

| 字段 | 类型 | 说明 | 默认值 |
|------|------|------|--------|
| `enabled` | boolean | 是否启用 visual 集成 | `false` |
| `visual_url` | string | Visual 服务地址 | - |
| `workspace_id` | string | 工作区唯一标识 | - |
| `workspace_name` | string | 工作区显示名称 | - |
| `push_mode` | string | 推送模式（`hook`/`websocket`） | `"hook"` |
| `push_timeout_ms` | number | 推送超时（毫秒） | `3000` |
| `retry_times` | number | 失败重试次数 | `1` |

**禁用 visual 集成**：

```json
{
  "enabled": false
}
```

### .ai-spec/visual-bridge.json

**位置**：项目根目录 `.ai-spec/visual-bridge.json`

**自动生成**：由 `init` 命令自动创建

**示例**：

```json
{
  "enabled": true,
  "agent_id": "ai-spec-auto",
  "push_on_runtime_state": true
}
```

**字段说明**：

| 字段 | 类型 | 说明 |
|------|------|------|
| `enabled` | boolean | 是否启用 visual bridge |
| `agent_id` | string | Agent 标识符 |
| `push_on_runtime_state` | boolean | 是否在 runtime-state 变更时推送 |

## 兼容性

### 向后兼容

✅ **100% 向后兼容**

- 不修改现有协议逻辑
- 不影响现有项目运行
- Visual 集成通过配置开关控制

### 降级策略

✅ **优雅降级**

1. **配置未启用**：不尝试推送，零开销
2. **Visual 服务不可用**：协议正常推进，仅记录日志
3. **推送超时**：3 秒超时，不阻塞协议推进
4. **推送失败**：重试 1 次后放弃，协议继续

### 环境要求

- **Node.js**：>=18.0.0（与 ai-spec-auto 一致）
- **依赖**：无新增外部依赖
- **Visual 服务**：可选，不是必需

## 使用场景

### 场景 1：本地开发（默认）

```bash
# 1. 安装 ai-spec-auto
npm install ai-spec-auto

# 2. 初始化（visual bridge 自动启用）
npx ai-spec-auto init .

# 3. 不配置 visual-config.json，visual 集成不生效
# 协议正常运行，与之前一致
```

**行为**：与之前完全一致，无感知

### 场景 2：本地开发 + 本地监控

```bash
# 1. 启动 visual 服务
cd /path/to/engineered-spec-visual
docker-compose up -d

# 2. 配置 visual 集成
cat > .ai-spec/visual-config.json <<EOF
{
  "enabled": true,
  "visual_url": "http://localhost:3000",
  "workspace_id": "my-project-local"
}
EOF

# 3. 正常使用 ai-spec-auto
npx ai-spec-auto protocol-step --user-input "需求"

# 4. 在浏览器查看实时状态
open http://localhost:3000/workspaces
```

**行为**：协议推进 + 实时监控

### 场景 3：团队协作 + 内网监控

```bash
# 1. 团队在内网服务器部署 visual（一次性）
# 服务器: http://visual.internal:3000

# 2. 每个项目配置 visual 集成
cat > .ai-spec/visual-config.json <<EOF
{
  "enabled": true,
  "visual_url": "http://visual.internal:3000",
  "workspace_id": "team-project-a"
}
EOF

# 3. 团队成员正常开发
npx ai-spec-auto protocol-step --user-input "需求"

# 4. 在内网控制台查看所有项目状态
open http://visual.internal:3000/workspaces
```

**行为**：多项目统一监控

**原因**：代码未更新

**解决**：

```bash
# 更新 ai-spec-auto
cd /path/to/project
npm install ai-spec-auto@latest

# 或使用本地开发版本
npm install /path/to/br-ai-spec
```

### 问题 2：协议推进时报错 "visual-hooks not found"

**原因**：visual-hooks 目录缺失

**解决**：

```bash
# 验证 visual-hooks 存在
ls node_modules/ai-spec-auto/internal/visual-hooks/

# 如果不存在，重新安装
npm install ai-spec-auto@latest
```

### 问题 3：visual 服务运行，但数据未推送

**原因 1**：`visual-config.json` 未启用

```bash

# 确保 enabled: true
```

**原因 2**：`visual_url` 配置错误

```bash
# 验证服务可达
curl http://localhost:3000/api/health

# 如果失败，检查 docker-compose 状态
cd /path/to/engineered-spec-visual
docker-compose ps
```

**原因 3**：`workspace_id` 配置错误

```bash
# 检查 workspace_id 是否与 visual 服务一致
# 建议使用项目名作为 workspace_id
```

### 问题 4：init 命令仍提示 visual bridge

**原因**：代码未更新到最新版本

**解决**：

## FAQ

### Q1：visual 集成会影响性能吗？

**A**：几乎无影响

- **配置未启用**：零开销，不执行任何 visual 相关代码
- **配置启用**：推送是异步的，3 秒超时，不阻塞协议推进
- **网络延迟**：推送失败时自动降级，不重试超过 1 次

### Q2：visual 服务必须部署吗？

**A**：不是必需的

- Visual 是可选组件，不影响 ai-spec-auto 核心功能
- 不部署 visual，协议仍正常运行

### Q3：多个项目可以共享一个 visual 服务吗？

**A**：可以

- 每个项目使用不同的 `workspace_id`
- Visual 服务通过 `workspace_id` 区分不同项目

### Q4：visual 集成会记录敏感信息吗？

**A**：不会

- 仅推送协议状态（run_id、status、current_role 等）
- 不推送用户输入内容、API 密钥等敏感信息
- 详见 `internal/visual-hooks/README.md` 中的数据结构

### Q5：如何在 CI/CD 中禁用 visual 集成？

**A**：环境变量控制

```bash
# 方案 1：删除配置文件
rm .ai-spec/visual-config.json

# 方案 2：设置环境变量（未来版本支持）
export AI_SPEC_VISUAL_ENABLED=false

# 方案 3：修改配置
echo '{"enabled": false}' > .ai-spec/visual-config.json
```
