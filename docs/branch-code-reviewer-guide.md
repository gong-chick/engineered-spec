# 分支代码评审使用指南

## 概述

`branch-code-reviewer` 技能提供强大的分支间代码差异分析能力,不仅检查技术类风险,还能结合需求文档分析业务类风险,最终生成精美的可视化 HTML 评审报告。

## 快速开始

### 1. 安装

```bash
npx @engineered/ai-spec-auto@latest init . --ide qoder
```

### 2. 使用

在 Qoder/Cursor/Claude 中执行:

```text
/branch-review
```

或在对话中说明:

```text
请评审当前分支,需求文档在 docs/prd-order-module.md
```

## 使用场景

### 场景 1: 合并前代码检查

**时机**: 功能开发完成,准备合并到主分支前

**目的**: 
- 发现潜在技术问题
- 确保代码质量
- 避免将问题带入主分支

**操作**:
```text
/branch-review
```

**输出**: 
- 技术风险报告
- 代码质量评分
- 改进建议

### 场景 2: 需求实现验证

**时机**: 功能开发完成后,验证是否完整实现需求

**目的**:
- 检查功能完整性
- 验证业务流程
- 确保需求覆盖度

**操作**:
```text
/branch-review docs/prd-order-module.md
```

**输出**:
- 技术风险报告
- 业务风险报告
- 需求覆盖度评分
- 缺失功能清单

### 场景 3: 代码质量审计

**时机**: 定期代码质量检查、技术债务评估

**目的**:
- 识别技术债务
- 评估重构需求
- 制定改进计划

**操作**:
```text
请分析最近一周的所有功能分支变更
```

### 场景 4: 安全审查

**时机**: 涉及敏感数据操作、权限变更的功能

**目的**:
- 发现安全漏洞
- 验证权限控制
- 检查数据保护

**操作**:
```text
/branch-review docs/security-requirements.md
```

## 工作流程详解

### 第一步: 确认评审上下文

AI 会:
1. 检测当前 Git 分支
2. 自动识别主分支(master/main)
3. 询问是否需要需求文档

**示例对话**:
```
AI: 请提供本次开发需求的文档路径(可选),用于业务风险分析:
    - 如果不提供,将仅进行技术类风险分析。

用户: docs/prd-order-module.md

AI: ✅ 评审上下文已确认:
    - 功能分支: feature/order-module
    - 主分支: master
    - 需求文档: 已读取
    - 评审范围: 技术风险 + 业务风险
```

### 第二步: 收集代码差异

AI 会执行:
```bash
git diff master..feature/order-module --stat
git diff master..feature/order-module --numstat
git log master..feature/order-module --oneline
```

**输出示例**:
```
📊 差异数据已收集:
- 变更文件数: 15
- 新增行数: +523
- 删除行数: -87
- 提交次数: 8
- 变更文件列表: 
  - src/components/OrderForm.vue
  - src/api/order.js
  - src/utils/validator.js
  - ...
```

### 第三步: 技术风险分析

AI 会检查每个变更文件,识别:

#### 性能问题
- ❌ 不必要的重渲染
- ❌ 内存泄漏
- ❌ 大循环性能瓶颈
- ❌ 未使用缓存

#### 安全漏洞
- ❌ XSS 攻击风险
- ❌ SQL/命令注入
- ❌ 敏感信息暴露
- ❌ CSRF 防护缺失

#### 错误处理
- ❌ Promise 未 catch
- ❌ try-catch 缺失
- ❌ 边界条件未处理
- ❌ 空值未检查

#### 代码质量
- ❌ 命名不规范
- ❌ 代码重复
- ❌ 函数过长
- ❌ 注释不足

### 第四步: 业务风险分析

**仅当提供需求文档时执行**

AI 会对照需求文档检查:

#### 需求覆盖度
- ✅ 功能完整性: 所有功能点是否已实现
- ✅ 验收标准: 是否满足验收条件
- ✅ 业务规则: 代码是否符合业务规则
- ✅ 边界场景: 边界情况是否都处理

#### 业务流程
- ✅ 流程完整性: 关键流程是否遗漏
- ✅ 状态流转: 状态变更是否正确
- ✅ 数据一致性: 数据操作是否符合约束
- ✅ 异常处理: 异常场景是否有处理

#### 示例分析
```
需求文档要求:
> 用户下单时必须校验库存,库存不足时显示"暂时缺货"并禁止提交订单。

代码检查:
✅ 发现库存校验逻辑
❌ 但缺少库存不足时的错误提示
🔴 业务风险: 用户看到库存为0时仍可提交订单
```

### 第五步: 生成 HTML 报告

AI 生成独立的 HTML 文件,包含:

#### 1. 概览面板
```
┌─────────────────────────────────────────────┐
│  📊 代码评审报告                              │
│  feature/order-module vs master               │
├─────────────────────────────────────────────┤
│  变更文件: 15 | 新增: +523 | 删除: -87        │
│  提交次数: 8                                  │
│  技术风险: 12 (🔴2 🟡5 🔵4 ⚪1)              │
│  业务风险: 5 (🔴2 🟡3)                       │
│  需求覆盖度: 85%                              │
│  整体通过率: 78%                              │
└─────────────────────────────────────────────┘
```

#### 2. 文件导航树
```
src/
├── components/
│   ├── OrderForm.vue (+120/-15) 🟡 2个风险
│   └── PaymentButton.vue (+80/-0) 🔴 1个风险
├── api/
│   └── order.js (+200/-30) 🔵 3个建议
└── utils/
    └── validator.js (+50/-0) ⚪ 1个提示
```

#### 3. 代码差异展示
- 支持并排/统一模式
- 语法高亮
- 变更行颜色标记
- 可折叠未变更区域

#### 4. 风险详情
```json
{
  "file": "src/components/OrderForm.vue",
  "line": 45,
  "severity": "critical",
  "category": "business",
  "title": "缺少库存不足处理逻辑",
  "description": "需求文档要求库存不足时禁止提交订单",
  "requirement_ref": "docs/prd-order-module.md §3.2",
  "suggestion": "添加库存校验,不足时禁用提交按钮"
}
```

#### 5. 交互功能<think>

- 🔍 搜索文件名和代码
- 🎛️ 按类型/等级过滤
- ⏮️⏭️ 快速跳转
- 💬 添加评论
- 🌓 主题切换
- 📤 导出报告

## 风险等级说明

### 技术风险

| 等级 | 标识 | 说明 | 处理建议 |
|------|------|------|----------|
| 严重 | 🔴 | 崩溃、安全漏洞、数据丢失 | 必须修复后合并 |
| 警告 | 🟡 | 性能问题、可维护性差 | 建议修复后合并 |
| 建议 | 🔵 | 代码优化、重构建议 | 后续迭代处理 |
| 提示 | ⚪ | 信息说明、最佳实践 | 可选处理 |

### 业务风险

| 等级 | 标识 | 说明 | 处理建议 |
|------|------|------|----------|
| 严重 | 🔴 | 缺失核心功能、流程错误 | 必须修复后上线 |
| 警告 | 🟡 | 缺少异常处理、边界遗漏 | 建议补充后上线 |
| 建议 | 🔵 | 体验优化、流程改进 | 后续迭代优化 |

## 最佳实践

### ✅ 推荐做法

1. **每次合并前评审**: 养成合并前运行 `/branch-review` 的习惯
2. **提供需求文档**: 尽量提供需求文档,获得完整的业务风险分析
3. **定期审计**: 每周运行一次,识别技术债务
4. **团队协作**: 分享 HTML 报告,团队共同 review
5. **持续改进**: 基于评审结果改进开发规范

### ❌ 避免做法

1. **不要跳过评审**: 即使是小改动也建议评审
2. **不要忽略严重风险**: 🔴 严重风险必须修复
3. **不要仅依赖工具**: 人工 review 仍不可少
4. **不要事后补文档**: 需求文档应在开发前就有

## 常见问题

### Q: 可以不指定需求文档吗?
A: 可以,将仅进行技术风险分析,不包含业务风险。

### Q: 支持哪些需求文档格式?
A: Markdown、PRD、用户故事等文本格式均可。

### Q: 报告能否分享给团队?
A: 可以,HTML 文件完全独立,发给任何人打开都能正常查看。

### Q: 会修改我的代码吗?
A: 不会,仅进行只读分析,不修改任何源代码。

### Q: 支持哪些分支名称?
A: 支持任意分支名称,自动检测 master 或 main 作为主分支。

### Q: 如何自定义主分支?
A: 在对话中说明即可:
```text
请对比 feature/new-payment 和 develop 分支
```
