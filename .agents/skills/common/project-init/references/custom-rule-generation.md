# 自定义规则生成要求

仅对 `待生成列表` 中的规则执行本文件中的要求。每个规则都要基于仓库事实生成，不得照搬通用模板。

## 04-组件规范.md

必须覆盖：

- 当前组件结构写法
- 组件目录组织方式
- 页面专属组件与通用组件的放置规则
- Props / Emits / 回调 / 事件的现状约定
- 拆分边界和不可违背的规则

若项目当前没有独立组件目录，也要明确写清：

- 当前不存在
- 推荐落点
- 后续新增约束

## 05-API规范.md

必须覆盖：

- 接口目录和请求封装入口
- API 与类型文件的推荐落点
- 函数命名模式
- 错误处理模式
- 页面/组件禁止直接请求的约束（若项目已有这类惯例）

若项目还没有统一 API 层，要写清：

- 当前现状
- 推荐落点
- 后续统一约束

## 06-路由规范.md

必须覆盖：

- 路由入口
- 路由模块或文件路由组织方式
- 页面目录与路由目录的对应关系
- 懒加载要求
- `meta` / 守卫现状与约束

若项目当前没有路由骨架，要明确写清：

- 当前不存在
- 推荐骨架
- 新增页面时的落点约束

## 07-状态管理.md

必须覆盖：

- 当前状态库
- store 目录与命名约定
- 模块拆分方式
- 在组件中使用 store 的方式
- 持久化现状

若项目没有全局状态目录，要明确写清：

- 当前无全局状态层
- 何时才需要新增
- 推荐落点

## 09-样式规范.md

必须覆盖：

- 当前样式方案
- 主题变量或主题 token 的现状
- 全局样式入口
- scoped / CSS Modules / Tailwind / Less / Sass 的实际使用规则
- 禁止硬编码颜色值或主题偏离的约束（若项目有主题体系）

若项目暂无主题变量，也要写清：

- 当前现状
- 推荐入口
- 后续新增样式时的基本约束

## 通用约束

- 每条规则都必须基于项目实际代码、目录和依赖归纳
- 严禁照搬通用模板
- “当前不存在” 也属于需要写清的现状
