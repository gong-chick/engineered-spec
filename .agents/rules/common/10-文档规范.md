# 文档规范

## 注释分类

项目中使用两种注释，用途不同：

| 类型 | 语法 | 面向 | 用途 |
|------|------|------|------|
| 文档注释 | `/** */` | 调用方 | 描述公开 API 的契约（导出函数、组件、接口等） |
| 实现注释 | `//` | 维护者 | 解释内部逻辑的"为什么"，标记 TODO/FIXME |

## 文档注释规范（`/** */`）

### 哪些代码需要写

- 导出的函数、类、接口、类型、枚举
- 组件的 Props 接口（说明各属性含义）
- 复杂的工具函数（说明功能、边界条件、副作用）

### 怎么写（TypeScript 项目）

TypeScript 已通过类型系统表达了参数类型和返回类型，**JSDoc 中不要重复类型信息**，聚焦于语义描述：

```ts
// 好 — 描述语义，不重复类型
/** 根据资产 ID 获取资产详情，未找到时返回 null */
export function getAssetDetailApi(id: number): Promise<AssetDetail | null> { ... }

// 差 — 重复 TS 已表达的类型信息
/**
 * @param {number} id - 资产ID
 * @returns {Promise<AssetDetail | null>} 资产详情
 */
export function getAssetDetailApi(id: number): Promise<AssetDetail | null> { ... }
```

### 常用 JSDoc 标签

| 标签 | 用途 | 示例 |
|------|------|------|
| `@param` | 补充参数语义（不写类型） | `@param id 资产唯一标识` |
| `@returns` | 补充返回值语义（不写类型） | `@returns 包含分页信息的列表` |
| `@example` | 提供使用示例 | 复杂工具函数推荐添加 |
| `@deprecated` | 标记即将废弃的 API | `@deprecated 请使用 getAssetListApi 替代` |
| `@throws` | 说明可能抛出的异常 | `@throws 当 token 过期时抛出 AuthError` |
| `@see` | 引用相关代码或文档 | `@see https://apifox.com/xxx` |

### 怎么写（纯 JavaScript 项目）

无静态类型系统时，JSDoc 承担更多契约表达职责：可在 `@param`、`@returns`、`@typedef` 中**写出类型信息**（如 `{number}`、`{Promise<AssetDetail|null>}`），与 02-编码规范中的 JavaScript 小节一致。

```js
/**
 * 根据资产 ID 获取资产详情，未找到时返回 null
 * @param {number} id
 * @returns {Promise<import('./types').AssetDetail | null>}
 */
export async function getAssetDetailApi(id) { ... }
```

- 导出函数、公共模块边界应优先写全 JSDoc，便于 IDE 提示与后续迁移 TypeScript。
- 团队若使用 `// @ts-check` 或 `checkJs`，JSDoc 类型应与 ESLint/TS 校验一致。

## 实现注释规范（`//`）

- 注释应该解释"为什么"而不是"是什么"，避免冗余注释
- 放在被注释代码的上方，而非行尾（行尾仅用于极短说明）

```ts
// 好 — 解释不明显的业务原因
// 延迟 300ms 等待动画完成后再请求，避免页面闪烁
await sleep(300);

// 差 — 代码本身已经表达清楚
// 设置 loading 为 true
setLoading(true);
```

## TODO / FIXME 约定

| 标记 | 含义 | 格式 |
|------|------|------|
| `TODO` | 待完成的功能或改进 | `// TODO: 描述内容` |
| `FIXME` | 已知问题待修复 | `// FIXME: 描述问题` |

- 使用统一格式 `// TODO: xxx`（冒号后加空格），便于全局检索
- 标记应包含足够的上下文，让其他人能理解意图

## 禁止模式（NON-NEGOTIABLE）

| 禁止项 | 原因 | 正确做法 |
|--------|------|----------|
| 注释掉的代码 | 造成噪音，增加维护负担 | 直接删除，靠 Git 历史追溯 |
| 冗余注释 | 重复代码已表达的信息 | 只注释代码无法表达的意图 |
| 过时注释 | 错误的注释比没有注释更有害 | 修改代码时同步更新注释 |
| JSDoc 中重复 TS 已写的类型 | 与类型系统重复 | TS 项目只写语义；JS 项目用 JSDoc 补类型 |
